## 1. Purpose

The **first Qt-touching module in the codebase.** `controller/model_bridge.py` is the single seam where the Qt-free model layer meets PySide6 (`DESIGN.md` §9.1 — *"the single seam where both worlds meet"*). The model emits four synchronous **blinker** signals on the `_stmrr_events` bus (`model/events.py`); the bridge subscribes to all four and **re-emits each as a Qt `Signal`** that the view layer connects to. This keeps ADR-0003 (`model/` imports no Qt) satisfiable: every PySide6 import lives on the controller side of this seam, and the `view-does-not-import-model-events` `import-linter` contract mechanically forbids the view from bypassing the bridge.

`ModelBridge` ships as a single `QObject` subclass exposing four `Signal(object)` attributes whose names mirror the blinker signals exactly (`ship_moved`, `docked`, `turn_advanced`, `state_changed`), so the translation is mechanical (umbrella §5.2). It also holds the `GameStateManager` (umbrella §8.1 lock — *"constructed once in `app.py` and passed into the bridge"*) and exposes a read-only `current_state` property so the view can render the **initial** mode on startup, before any `state_changed` event fires.

This is a **bridge-only** step: the module, its tests, and a new `tests/integration/conftest.py` fixture. The `app.py` wiring that owns the bridge's lifetime (GSM construction, `destroyed → teardown` connection) is specified here as the contract for a future `app.py` step but is **not** built now — building it would pull in `MainWindow`, `QGraphicsScene`, and view modules that have no spec yet.

## 3. Layering and constraints

- **ADR-0003 (model-is-Qt-free)** is *not* weakened: the bridge lives in `stmrr.controller`, outside the `model-is-qt-free` contract's `source_modules = stmrr.model`. The bridge importing `PySide6` is correct and intended.
- **The bridge is the only module permitted to import both `stmrr.model.events` and `PySide6`** (`DESIGN.md` §9.1). The existing `view-does-not-import-model-events` contract forbids the *view* from importing `model.events`; nothing forbids the *controller* from doing so. **No new `.importlinter` contract** is added — the existing four (`model-is-qt-free`, `view-does-not-import-model-events`, `projection-is-qt-free`, `blinker-only-in-events`) all continue to pass unchanged. (A "only `model_bridge` may import both" contract is not expressible as a forbidden/independence rule in import-linter and is unnecessary for v0.1; it is left as a convention.)
- **`blinker-only-in-events`** is unaffected: that contract forbids direct `blinker` imports *outside `model.events`* within the model tree (`allow_indirect_imports = True`). The bridge imports the named **signals** (`from stmrr.model.events import ship_moved, ...`), not `blinker` itself, and is outside the model tree regardless.
- **`test_no_qt_imports.py` (umbrella invariant 1b)** is unaffected: it walks only `stmrr.model.*` in a **subprocess**, so PySide6 being loaded in the main pytest process by the integration tests does not pollute it. No change to that test.
- `from __future__ import annotations` at the top per `DESIGN.md` §10.6. **mypy `--strict`** applies — `[[tool.mypy.overrides]]` sets `strict = true` for `stmrr.controller.*`.
- **Imports** (runtime vs. `TYPE_CHECKING`):
  - **Runtime:** `from PySide6.QtCore import QObject, Signal`; `from stmrr.model.events import docked, ship_moved, state_changed, turn_advanced` (the signal objects are needed for `.connect()` / `.disconnect()`).
  - **`TYPE_CHECKING` only:** `from stmrr.model.events import DockedPayload, ShipMovedPayload, StateChangedPayload, TurnAdvancedPayload` (used only as `_on_*` parameter annotations — strings under deferred evaluation); `from stmrr.model.state.game_state_manager import GameStateManager` and `from stmrr.model.state.states import GameState` (used only as annotations on `__init__`, the `_manager` attribute, and the `current_state` property). The bridge calls `self._manager.current_state` by duck-typed attribute access, never `isinstance`, so no runtime import of `GameStateManager`/`GameState` is required.
- No new dependencies (`PySide6>=6.5`, `blinker>=1.8`, `pytest-qt>=4.4` already declared in `pyproject.toml`).
- Module + class docstrings mandatory. Method docstring on `teardown` (the two-obligation teardown is non-obvious, §5.7) and on `current_state` (initial-render rationale). `_on_*` handlers carry a one-line comment noting they are blinker receivers, not Qt slots.

## 4. `controller/model_bridge.py` specification

### 4.1 Class declaration

```python
from __future__ import annotations

from typing import TYPE_CHECKING

from PySide6.QtCore import QObject, Signal

from stmrr.model.events import docked, ship_moved, state_changed, turn_advanced

if TYPE_CHECKING:
    from stmrr.model.events import (
        DockedPayload,
        ShipMovedPayload,
        StateChangedPayload,
        TurnAdvancedPayload,
    )
    from stmrr.model.state.game_state_manager import GameStateManager
    from stmrr.model.state.states import GameState


class ModelBridge(QObject):
    """The single MVC seam: re-emits model blinker events as Qt signals.

    Subscribes to the four `model.events` blinker signals and re-emits each
    as an identically named Qt `Signal(object)` carrying the same frozen
    payload dataclass instance. This is the ONLY module that imports both
    `PySide6` and `stmrr.model.events` (DESIGN.md §9.1); it is what keeps
    ADR-0003's model-is-Qt-free rule satisfiable.

    Holds the GameStateManager (umbrella §8.1) so the view can read the
    initial `current_state` on startup — before the first `state_changed`
    transition fires.

    NOT an ABC: subclassing `QObject` alongside `abc.ABC` raises a Shiboken
    metaclass TypeError (§5.2). Plain QObject subclass only.
    """

    ship_moved = Signal(object)
    docked = Signal(object)
    turn_advanced = Signal(object)
    state_changed = Signal(object)

    def __init__(self, manager: GameStateManager, parent: QObject | None = None) -> None: ...
    @property
    def current_state(self) -> GameState: ...
    def teardown(self) -> None: ...
    # private blinker receivers: _on_ship_moved, _on_docked, _on_turn_advanced, _on_state_changed
```

The four `Signal(object)` declarations **must be class-level attributes** — declaring a `Signal` inside `__init__` raises `AttributeError` on `.connect()`/`.emit()` (§5.2). `Signal(object)` is the PySide6 idiom for a signal carrying an arbitrary Python value, including a frozen dataclass (§5.4).

### 4.2 `__init__(manager, parent=None)` semantics

1. `super().__init__(parent)` — pass `parent` through to `QObject`; `None` for the test fixture, the `MainWindow` in the future `app.py` (§5.7).
2. Store the manager: `self._manager: GameStateManager = manager`.
3. Connect each blinker signal to its private receiver with **`weak=False`**:
   ```python
   ship_moved.connect(self._on_ship_moved, weak=False)
   docked.connect(self._on_docked, weak=False)
   turn_advanced.connect(self._on_turn_advanced, weak=False)
   state_changed.connect(self._on_state_changed, weak=False)
   ```
   `weak=False` is **load-bearing** (§5.3): blinker holds receivers weakly by default, and the receivers are bound methods of `self`. With the default `weak=True`, once the only external reference to the bridge is dropped, blinker silently stops invoking the handlers — no exception, no log. `weak=False` makes blinker hold the bridge strongly, so subscriptions survive for the bridge's lifetime; `teardown()` (§4.5) is the explicit counterpart that releases them.

No defensive `isinstance(manager, GameStateManager)` check — the annotation is the contract (consistent with the model layer's §9.1-Q4 posture in step 8).

### 4.3 `current_state` property

```python
@property
def current_state(self) -> GameState:
    """The model's current state, for initial view render on startup.

    The view reads this once at wire-up to render the starting mode
    (e.g. MainMenu) before any `state_changed` transition fires; thereafter
    it tracks transitions via the `state_changed` Qt signal.
    """
    return self._manager.current_state
```

Read-only — no setter. This is the sole reason the bridge holds the manager; without it `self._manager` would be a stored-but-unread attribute.

### 4.4 Blinker receivers `_on_*`

Blinker invokes a receiver as `receiver(sender, **kwargs)`; the model sends `signal.send(self, payload=Payload(...))` (umbrella §5.2). Each receiver therefore takes the sender positionally and `payload` keyword-only, and re-emits on the matching Qt signal:

```python
def _on_ship_moved(self, _sender: object, *, payload: ShipMovedPayload) -> None:
    # blinker receiver — not a Qt slot; @Slot not applicable (see §5.6).
    self.ship_moved.emit(payload)

def _on_docked(self, _sender: object, *, payload: DockedPayload) -> None:
    self.docked.emit(payload)

def _on_turn_advanced(self, _sender: object, *, payload: TurnAdvancedPayload) -> None:
    self.turn_advanced.emit(payload)

def _on_state_changed(self, _sender: object, *, payload: StateChangedPayload) -> None:
    self.state_changed.emit(payload)
```

- The `_sender` parameter is unused (named `_sender` to signal that). The bridge re-emits for any sender; v0.1 has exactly one sender per signal (umbrella §5.2).
- The **same payload instance** is forwarded — no copy, no field unpacking. The payload is a frozen dataclass; the Qt slot narrows it with `assert isinstance(payload, ShipMovedPayload)` (§5.4).
- The typed `payload` parameter is what gives mypy `--strict` the concrete type at the `.emit(payload)` call site, which is why no separate typed `emit_*()` wrapper is needed in v0.1 (§5.5).

### 4.5 `teardown()` — blinker disconnect

```python
def teardown(self) -> None:
    """Disconnect all four blinker subscriptions.

    Independent of Qt-side teardown: Qt object-tree parenting disconnects
    the *Qt signal* connections (view slots ← bridge signals) when the
    parent is destroyed, but does NOT touch the *blinker* subscriptions
    (bridge ← model events). Both are required for a clean shutdown; this
    method owns the blinker half. The future app.py wires
    `bridge.destroyed.connect(bridge.teardown)` (§5.7).
    """
    ship_moved.disconnect(self._on_ship_moved)
    docked.disconnect(self._on_docked)
    turn_advanced.disconnect(self._on_turn_advanced)
    state_changed.disconnect(self._on_state_changed)
```

`blinker.Signal.disconnect(receiver)` of a receiver that is not currently connected is a no-op (does not raise), so `teardown()` is safe to call more than once; the test fixture (§6.1) always calls it once in teardown regardless of what a test did.

### 4.6 Error policy: propagate, no rollback

The bridge wraps **nothing** in try/except. v0.1 is single-threaded, so the blinker → bridge → Qt path runs on one thread (`DirectConnection` semantics). If a connected Qt slot raises during `self.<signal>.emit(payload)`, the exception propagates synchronously: slot → `emit()` → `_on_*` → blinker `send()` → the model method that fired the event. The model mutation that justified the event has **already committed** before `send()` was called (umbrella invariant 3 + 9). This matches the model's own lifecycle-hook policy (step-8 §6.5: propagate without rollback) and the umbrella §9 synchronous-emit risk row. v0.1 has exactly one subscriber per signal (this bridge), so the failure mode is bounded; the `_send_robust` mitigation is the documented v0.2 escalation.

### 4.7 Worked example

```python
from PySide6.QtCore import QObject
from stmrr.controller.model_bridge import ModelBridge
from stmrr.model.events import ShipMovedPayload, ship_moved
from stmrr.model.state.game_state_manager import GameStateManager
from stmrr.model.state.states import MainMenuState

bridge = ModelBridge(GameStateManager(MainMenuState()))
assert isinstance(bridge.current_state, MainMenuState)  # initial render hook

received: list[object] = []
bridge.ship_moved.connect(received.append)              # a view slot would do this

payload = ShipMovedPayload(ship_id=..., from_position=..., to_position=...)
ship_moved.send(object(), payload=payload)              # the model fires the blinker event
assert received == [payload]                            # same instance, re-emitted on Qt side

bridge.teardown()                                       # releases the blinker subscriptions
ship_moved.send(object(), payload=payload)              # no longer re-emitted
assert received == [payload]                            # unchanged
```

## 5. Cross-cutting design decisions

### 5.1 Single `ModelBridge` QObject, not one QObject per signal

One `QObject` owning all four `Signal`s — the `WorkerSignals(QObject)` idiom from Qt's own threading documentation. One object means one ownership point in `app.py`, one uniform thread affinity, and one teardown call. Per-signal QObjects would multiply ownership, parenting, and teardown surface for zero benefit at four signals. (OQ best-practice, research `…-open-questions.md`.)

### 5.2 Plain `QObject`, never `QObject + ABC`; `Signal`s are class-level

`class ModelBridge(QObject, ABC)` raises `TypeError: Shiboken.ObjectType.__new__(...) is not safe` — Shiboken's metaclass conflicts with `ABCMeta`. The bridge is a plain `QObject` subclass. This is a real trap because the model layer leans heavily on ABCs (`GameState`), so the instinct to reach for `ABC` must be explicitly resisted. Likewise, a `Signal` declared inside `__init__` rather than at class scope raises `AttributeError` on first `.connect()` — the descriptor protocol requires class-level declaration. (OQ footguns, both corroborated.)

### 5.3 `weak=False` on every blinker connection

blinker's `connect(receiver, weak=True)` default holds receivers via weak reference. The bridge's receivers are **bound methods** (`self._on_*`), whose weakref tracks the bridge instance; when the last external reference to the bridge is dropped, blinker silently stops calling them — no error, no log (`has_receivers_for()` even keeps returning `True` for the dead receiver). For a long-lived singleton bridge this is a latent "events silently stop arriving" bug. `weak=False` makes blinker hold the bridge strongly; `teardown()` is the explicit release. The GC-survival test (§6.2) pins this. (OQ headline footgun.)

### 5.4 `Signal(object)` carrying the intact frozen payload

The four payloads carry non-primitive fields (`EntityId`/`NewType`, `GridPosition`, `type[GameState]`) that cannot be expressed as Qt primitive signal types. `Signal(object)` carries the whole frozen dataclass; the receiving slot narrows it with `assert isinstance(payload, ShipMovedPayload)`. No field unpacking — that would couple the bridge to each payload's shape and lose the single-`object` uniformity.

### 5.5 No typed `emit_*()` wrappers in v0.1 (OQ-1)

There is no parametric `Signal[T]` in PySide6 (official or `PySide6-stubs` 6.7.3.0); the bundled stubs type `Signal(object).emit` as accepting `Any`. The research-recommended workaround is a typed `emit_ship_moved(self, payload: ShipMovedPayload)` wrapper that the `_on_*` handler calls instead of `.emit()` directly. **Deferred:** the `_on_*` handler's `payload` parameter is *already* typed with the concrete payload, so mypy `--strict` already sees the real type at the single emit call site. Adding four wrapper methods whose only caller is the already-typed handler is indirection without additional static safety (YAGNI). The wrapper is the documented v0.2 option **if** a second, non-blinker caller ever needs to emit a bridge signal type-safely. View slots unavoidably annotate `(self, payload: object)` and narrow internally — this is a stub limitation, not a design choice.

### 5.6 No `@Slot` on `_on_*` in v0.1 (OQ-2)

`@Slot` is cosmetic on the `_on_*` handlers here, for two independent reasons: (a) they sit in blinker's receiver table and are invoked as plain Python callables — Qt's meta-object slot dispatch never touches them; (b) v0.1 is single-threaded (`DirectConnection`), which needs no `@Slot` regardless. `@Slot` becomes load-bearing on the **view-slot** side (not the bridge) only under a cross-thread `QueuedConnection`, where an undecorated slot's proxy can segfault (confirmed PySide6 6.6–6.8). **v0.2 trigger to document:** *when a background model thread is introduced, add `@Slot(object)` to view slots invoked via `QueuedConnection` — not to the bridge's blinker handlers.*

### 5.7 Two orthogonal teardown obligations + the future `app.py` wiring (OQ-4)

Shutdown has **two** independent cleanup paths that must not be conflated:

| Obligation | What it disconnects | Mechanism |
|---|---|---|
| Qt side | view slots ← bridge `Signal`s | Qt object-tree parenting: `~QObject` auto-disconnects when the parent (`MainWindow`) is destroyed |
| blinker side | bridge `_on_*` ← model events | **explicit** `bridge.teardown()` (§4.5) — Qt parenting does NOT touch blinker |

The future `app.py` owns both: it constructs the GSM, constructs `ModelBridge(manager, parent=main_window)` (Qt side handled by parenting), and wires the blinker side with `bridge.destroyed.connect(bridge.teardown)`. `QObject.destroyed` fires before the C++ object is finalized, and `teardown()` touches only Python/blinker structures (no Shiboken calls), so there is no use-after-free. This wiring is **specified, not built** in step 9 — but §6.2 includes a test that exercises the `destroyed → teardown` connection on a parentless bridge to validate the contract ahead of `app.py` (residual OQ-2 in the research).

### 5.8 Manager injection over no-arg construction

Umbrella §8.1 locks *"`GameStateManager` … constructed once in `app.py` and passed into the bridge."* The bridge therefore takes `manager` as a required constructor argument (not the no-arg `ModelBridge()` the research sketch assumed). The `current_state` property (§4.3) is the concrete consumer of that reference. No singleton, no module-level instance, no DI container — the same rationale the umbrella gives for the GSM itself.

### 6.1 `tests/integration/conftest.py` — the `bridge` fixture (new; first conftest in the repo)

```python
from collections.abc import Iterator

import pytest
from pytestqt.qtbot import QtBot

from stmrr.controller.model_bridge import ModelBridge
from stmrr.model.state.game_state_manager import GameStateManager
from stmrr.model.state.states import MainMenuState


@pytest.fixture
def bridge(qtbot: QtBot) -> Iterator[ModelBridge]:
    """Fresh ModelBridge per test, with blinker teardown on exit.

    Function-scoped (NOT session): with weak=False, blinker holds each
    constructed bridge strongly, so a session-scoped or un-torn-down bridge
    would accumulate subscribers and fire the same event on multiple bridges
    across tests. `qtbot` provides the session-scoped QApplication singleton.
    A yield fixture is a generator — annotate `Iterator[ModelBridge]`, not
    `ModelBridge` (mypy checks generator return types).
    """
    b = ModelBridge(GameStateManager(MainMenuState()))
    yield b
    b.teardown()
```

`qapp` is session-scoped by hard constraint (a `QApplication` cannot be destroyed and recreated within one process); `qtbot` is function-scoped and pulls `qapp` in implicitly. The `bridge` fixture is function-scoped with explicit `teardown()` to prevent cross-test subscriber accumulation (§5.3).

## 7. Invariants

Properties step 9 contributes; a reviewer should probe these.

1. **The bridge is the sole model→view event path.** It is the only module importing both `stmrr.model.events` and `PySide6`; the `view-does-not-import-model-events` contract guarantees the view cannot subscribe to blinker directly. (DESIGN §9.1.)
2. **Qt signal names equal blinker signal names** (`ship_moved`, `docked`, `turn_advanced`, `state_changed`). A rename on either side without the other breaks the mechanical mapping; tests assert each pairing.
3. **Payloads cross the seam by identity, unmodified.** `_on_*` forwards the exact frozen dataclass instance; no copy, no field unpacking. Tested by an `is`-identity assertion.
4. **All four blinker connections use `weak=False`.** The long-lived bridge must not be silently GC-disconnected. Pinned by the GC-survival test.
5. **`teardown()` releases exactly the four blinker subscriptions and is idempotent.** It does not touch Qt-side connections (those are Qt-parenting's job, §5.7).
6. **No try/except around emission.** A raising Qt slot propagates synchronously through the bridge to the model call site; the model mutation is already committed (umbrella invariants 3 + 9). Matches step-8 §6.5.
7. **`ModelBridge` is a plain `QObject`, not an ABC.** No `abc.ABC` in its MRO (Shiboken metaclass constraint, §5.2).
8. **`Signal`s are class-level attributes.** Not declared in `__init__` (§5.2).
9. **`current_state` is read-only and delegates to the injected manager.** It is the only consumer of `self._manager`; the bridge performs no `isinstance`/`type` dispatch on the manager.
10. **No new `.importlinter` contract; no umbrella amendment.** Step 9 is the first controller module; the model-layer umbrella spec's module table (§4) and DAG (§6) describe `stmrr.model.*` only and are not extended by a controller module. The bridge is described by `DESIGN.md` §9.1 and umbrella §3/§5.2.

## 8. Open questions

### 8.1 Locked decisions

- **Single `ModelBridge(QObject)` holding four `Signal(object)`** — not per-signal QObjects (§5.1).
- **Plain `QObject` subclass, no `ABC`; `Signal`s declared at class scope** (§5.2).
- **`weak=False` on all four blinker connections; explicit `teardown()` for release** (§5.3).
- **Payload carried intact as `object`; view narrows with `isinstance`** (§5.4).
- **No typed `emit_*()` wrappers in v0.1** — the typed `_on_*` parameter already types the emit call site; wrapper is a v0.2 option if a non-blinker emitter appears (§5.5).
- **No `@Slot` on `_on_*`** — blinker receivers, single-thread; v0.2 trigger (`QueuedConnection` across threads) documented (§5.6).
- **Error policy: propagate, no rollback, no try/except** (§4.6) — matches step-8 §6.5 and umbrella §9.
- **`__init__(manager, parent=None)`; manager required** per umbrella §8.1; `current_state` is its consumer (§5.8).
- **`current_state` read-only, delegates to manager** — initial-render hook for the view (§4.3).
- **Bridge-only scope; `app.py` wiring (`destroyed → teardown`, GSM construction) specified but not built** (§2, §5.7).
- **No new `.importlinter` contract; no umbrella amendment** (§3, invariant 10).
- **Tests under `tests/integration/`; function-scoped `bridge` fixture in a new `tests/integration/conftest.py`** (§6.1).

### 8.2 Still open (v0.2 concerns, NOT v0.1 blockers)

1. **Parametric `Signal[T]` typing.** No path in official or `PySide6-stubs` stubs as of May 2026 (needs a mypy plugin or PEP 646 stubs). View slots eat `object` + `isinstance` for now. Track `python-qt-tools/PySide6-stubs`. (Research OQ-1.)
2. **`@Slot` + `QueuedConnection` when a model worker thread lands.** v0.2 must add `@Slot(object)` to *view* slots crossing threads (not the bridge handlers), or a non-decorated proxy may segfault on PySide6 6.6–6.8. (Research OQ-2.)
3. **`_send_robust` fault isolation.** When a second non-bridge subscriber connects to any signal, a raising receiver could skip the bridge (or vice versa). Implement the per-receiver try/except utility then (umbrella §9 / step-7 §9.2).
4. **Python 3.14 free-threading.** blinker's `receivers` dict is unlocked; safe under v0.1's single-thread assumption, a data race under a free-threaded multi-thread model. Confine model mutation/sends to one thread or lock `send()` in v0.2. (Research, compatibility.)
5. **A bridge registry / multi-bridge ownership.** If a second `ModelBridge` instance or multiple independent view subscribers appear, revisit single-instance ownership and the context-manager-vs-`teardown()` choice. (Research OQ-4.)
