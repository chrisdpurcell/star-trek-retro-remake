## 1. Purpose

The **first Qt-touching module in the codebase.** `controller/model_bridge.py` is the single seam where the Qt-free model layer meets PySide6 (`DESIGN.md` ¬ß9.1 ‚Äî *"the single seam where both worlds meet"*). The model emits four synchronous **blinker** signals on the `_stmrr_events` bus (`model/events.py`); the bridge subscribes to all four and **re-emits each as a Qt `Signal`** that the view layer connects to. This keeps ADR-0003 (`model/` imports no Qt) satisfiable: every PySide6 import lives on the controller side of this seam, and the `view-does-not-import-model-events` `import-linter` contract mechanically forbids the view from bypassing the bridge.

`ModelBridge` ships as a single `QObject` subclass exposing four `Signal(object)` attributes whose names mirror the blinker signals exactly (`ship_moved`, `docked`, `turn_advanced`, `state_changed`), so the translation is mechanical (umbrella ¬ß5.2). It also holds the `GameStateManager` (umbrella ¬ß8.1 lock ‚Äî *"constructed once in `app.py` and passed into the bridge"*) and exposes a read-only `current_state` property so the view can render the **initial** mode on startup, before any `state_changed` event fires.

This is a **bridge-only** step: the module, its tests, and a new `tests/integration/conftest.py` fixture. The `app.py` wiring that owns the bridge's lifetime (GSM construction, owner-side `MainWindow.closeEvent` ‚Üí `bridge.teardown()`) is specified here as the contract for a future `app.py` step but is **not** built now ‚Äî building it would pull in `MainWindow`, `QGraphicsScene`, and view modules that have no spec yet.

## 3. Layering and constraints

- **ADR-0003 (model-is-Qt-free)** is *not* weakened: the bridge lives in `stmrr.controller`, outside the `model-is-qt-free` contract's `source_modules = stmrr.model`. The bridge importing `PySide6` is correct and intended.
- **The bridge is the only module permitted to import both `stmrr.model.events` and `PySide6`.** ADR-0003 and `docs/architecture.md` both state this seam is *"enforced mechanically by import-linter"* ‚Äî so step 9 honors that accepted decision by adding a **fifth `.importlinter` contract** for the controller side, exactly mirroring the in-repo `blinker-only-in-events` idiom (a `forbidden` contract whose single allowed edge is whitelisted via `ignore_imports`):

  ```ini
  [importlinter:contract:controller-events-via-bridge-only]
  name = Only model_bridge imports model.events within the controller
  type = forbidden
  source_modules =
      stmrr.controller
  forbidden_modules =
      stmrr.model.events
  ignore_imports =
      stmrr.controller.model_bridge -> stmrr.model.events
  allow_indirect_imports = True
  ```

  The `ignore_imports` line whitelists the one direct edge the bridge needs (covering both its runtime signal import and its `TYPE_CHECKING` payload imports ‚Äî import-linter sees both); `allow_indirect_imports = True` permits legitimate transitive paths such as a future `action_handlers ‚Üí model_bridge ‚Üí model.events`. The contract **lands in the same step/commit as `model_bridge.py`** so `lint-imports` stays green (adding it earlier would fail on the not-yet-existing `model_bridge` edge). The **view** side stays enforced by the existing `view-does-not-import-model-events`; the other four contracts (`model-is-qt-free`, `view-does-not-import-model-events`, `projection-is-qt-free`, `blinker-only-in-events`) are unchanged. (Round-1 review SA-003 surfaced the gap; an interim regex-scan test was considered but rejected at round 2 ‚Äî an import-linter contract matches ADR-0003's wording and the `blinker-only-in-events` precedent, so **no ADR/architecture amendment is needed**.)
- **`blinker-only-in-events`** is unaffected: that contract forbids direct `blinker` imports *outside `model.events`* within the model tree (`allow_indirect_imports = True`). The bridge imports the named **signals** (`from stmrr.model.events import ship_moved, ...`), not `blinker` itself, and is outside the model tree regardless.
- **`test_no_qt_imports.py` (umbrella invariant 1b)** is unaffected: it walks only `stmrr.model.*` in a **subprocess**, so PySide6 being loaded in the main pytest process by the integration tests does not pollute it. No change to that test.
- `from __future__ import annotations` at the top per `DESIGN.md` ¬ß10.6. **mypy `--strict`** applies ‚Äî `[[tool.mypy.overrides]]` sets `strict = true` for `stmrr.controller.*`.
- **Imports** (runtime vs. `TYPE_CHECKING`):
  - **Runtime:** `from PySide6.QtCore import QObject, Signal`; `from stmrr.model.events import docked, ship_moved, state_changed, turn_advanced` (the signal objects are needed for `.connect()` / `.disconnect()`).
  - **`TYPE_CHECKING` only:** `from stmrr.model.events import DockedPayload, ShipMovedPayload, StateChangedPayload, TurnAdvancedPayload` (used only as `_on_*` parameter annotations ‚Äî strings under deferred evaluation); `from stmrr.model.state.game_state_manager import GameStateManager` and `from stmrr.model.state.states import GameState` (used only as annotations on `__init__`, the `_manager` attribute, and the `current_state` property). The bridge calls `self._manager.current_state` by duck-typed attribute access, never `isinstance`, so no runtime import of `GameStateManager`/`GameState` is required.
- No new dependencies (`PySide6>=6.5`, `blinker>=1.8`, `pytest-qt>=4.4` already declared in `pyproject.toml`).
- Module + class docstrings mandatory. Method docstring on `teardown` (the two-obligation teardown is non-obvious, ¬ß5.7) and on `current_state` (initial-render rationale). `_on_*` handlers carry a one-line comment noting they are blinker receivers, not Qt slots.

## 4. `controller/model_bridge.py` specification

### 4.1 Class declaration

```python
from __future__ import annotations

from typing import TYPE_CHECKING

from PySide6.QtCore import QObject, Signal

from stmrr.model.events import docked, ship_moved, state_changed, turn_advanced

if TYPE_CHECKING:
    from stmrr.model.events import (
        DockedPayload,
        ShipMovedPayload,
        StateChangedPayload,
        TurnAdvancedPayload,
    )
    from stmrr.model.state.game_state_manager import GameStateManager
    from stmrr.model.state.states import GameState


class ModelBridge(QObject):
    """The single MVC seam: re-emits model blinker events as Qt signals.

    Subscribes to the four `model.events` blinker signals and re-emits each
    as an identically named Qt `Signal(object)` carrying the same frozen
    payload dataclass instance. This is the ONLY module that imports both
    `PySide6` and `stmrr.model.events` (DESIGN.md ¬ß9.1); it is what keeps
    ADR-0003's model-is-Qt-free rule satisfiable.

    Holds the GameStateManager (umbrella ¬ß8.1) so the view can read the
    initial `current_state` on startup ‚Äî before the first `state_changed`
    transition fires.

    NOT an ABC: subclassing `QObject` alongside `abc.ABC` raises a Shiboken
    metaclass TypeError (¬ß5.2). Plain QObject subclass only.
    """

    ship_moved = Signal(object)
    docked = Signal(object)
    turn_advanced = Signal(object)
    state_changed = Signal(object)

    def __init__(self, manager: GameStateManager, parent: QObject | None = None) -> None: ...
    @property
    def current_state(self) -> GameState: ...
    def teardown(self) -> None: ...
    # private blinker receivers: _on_ship_moved, _on_docked, _on_turn_advanced, _on_state_changed
```

The four `Signal(object)` declarations **must be class-level attributes** ‚Äî declaring a `Signal` inside `__init__` raises `AttributeError` on `.connect()`/`.emit()` (¬ß5.2). `Signal(object)` is the PySide6 idiom for a signal carrying an arbitrary Python value, including a frozen dataclass (¬ß5.4).

### 4.2 `__init__(manager, parent=None)` semantics

1. `super().__init__(parent)` ‚Äî pass `parent` through to `QObject`; `None` for the test fixture, the `MainWindow` in the future `app.py` (¬ß5.7).
2. Store the manager: `self._manager: GameStateManager = manager`.
3. Connect each blinker signal to its private receiver with **`weak=False`**:
   ```python
   ship_moved.connect(self._on_ship_moved, weak=False)
   docked.connect(self._on_docked, weak=False)
   turn_advanced.connect(self._on_turn_advanced, weak=False)
   state_changed.connect(self._on_state_changed, weak=False)
   ```
   `weak=False` is **load-bearing** (¬ß5.3): blinker holds receivers weakly by default, and the receivers are bound methods of `self`. With the default `weak=True`, once the only external reference to the bridge is dropped, blinker silently stops invoking the handlers ‚Äî no exception, no log. `weak=False` makes blinker hold the bridge strongly, so subscriptions survive for the bridge's lifetime; `teardown()` (¬ß4.5) is the explicit counterpart that releases them.

No defensive `isinstance(manager, GameStateManager)` check ‚Äî the annotation is the contract (consistent with the model layer's ¬ß9.1-Q4 posture in step 8).

### 4.3 `current_state` property

```python
@property
def current_state(self) -> GameState:
    """The model's current state, for initial view render on startup.

    The view reads this once at wire-up to render the starting mode
    (e.g. MainMenu) before any `state_changed` transition fires; thereafter
    it tracks transitions via the `state_changed` Qt signal.
    """
    return self._manager.current_state
```

Read-only ‚Äî no setter. This is the sole reason the bridge holds the manager; without it `self._manager` would be a stored-but-unread attribute.

### 4.4 Blinker receivers `_on_*`

Blinker invokes a receiver as `receiver(sender, **kwargs)`; the model sends `signal.send(self, payload=Payload(...))` (umbrella ¬ß5.2). Each receiver therefore takes the sender positionally and `payload` keyword-only, and re-emits on the matching Qt signal:

```python
def _on_ship_moved(self, _sender: object, *, payload: ShipMovedPayload) -> None:
    # blinker receiver ‚Äî not a Qt slot; @Slot not applicable (see ¬ß5.6).
    self.ship_moved.emit(payload)

def _on_docked(self, _sender: object, *, payload: DockedPayload) -> None:
    self.docked.emit(payload)

def _on_turn_advanced(self, _sender: object, *, payload: TurnAdvancedPayload) -> None:
    self.turn_advanced.emit(payload)

def _on_state_changed(self, _sender: object, *, payload: StateChangedPayload) -> None:
    self.state_changed.emit(payload)
```

- The `_sender` parameter is unused (named `_sender` to signal that). The bridge re-emits for any sender; v0.1 has exactly one sender per signal (umbrella ¬ß5.2).
- The **same payload instance** is forwarded ‚Äî no copy, no field unpacking. The payload is a frozen dataclass; the Qt slot narrows it with `assert isinstance(payload, ShipMovedPayload)` (¬ß5.4).
- The typed `payload` parameter narrows the payload type **within the handler body**, but the `self.<signal>.emit(payload)` call itself is unchecked ‚Äî `SignalInstance.emit` is typed `*args: Any` (¬ß5.5). bridge‚ÜíQt payload correctness is pinned by the per-signal payload-identity tests (¬ß6.2), not by mypy.

### 4.5 `teardown()` ‚Äî blinker disconnect

```python
def teardown(self) -> None:
    """Disconnect all four blinker subscriptions.

    Independent of Qt-side teardown: Qt object-tree parenting disconnects
    the *Qt signal* connections (view slots ‚Üê bridge signals) when the
    parent is destroyed, but does NOT touch the *blinker* subscriptions
    (bridge ‚Üê model events). Both are required for a clean shutdown; this
    method owns the blinker half. Must be called by the bridge's OWNER
    BEFORE the bridge is destroyed (e.g. `MainWindow.closeEvent`), NOT via
    `self.destroyed.connect(self.teardown)` ‚Äî see ¬ß5.7.
    """
    ship_moved.disconnect(self._on_ship_moved)
    docked.disconnect(self._on_docked)
    turn_advanced.disconnect(self._on_turn_advanced)
    state_changed.disconnect(self._on_state_changed)
```

`blinker.Signal.disconnect(receiver)` of a receiver that is not currently connected is a no-op (does not raise), so `teardown()` is safe to call more than once; the test fixture (¬ß6.1) always calls it once in teardown regardless of what a test did.

### 4.6 Error policy: the bridge catches nothing; Qt absorbs view-slot exceptions

The bridge wraps **nothing** in try/except. But ‚Äî verified empirically against the project's pinned **PySide6 6.11.0** (`uv.lock`; see the probe in ¬ß6.2) ‚Äî a Qt slot that raises during `self.<signal>.emit(payload)` does **NOT** propagate back through `emit()`. PySide6 routes the unhandled slot exception to `sys.excepthook` and `emit()` returns normally; the exception therefore does **not** travel `slot ‚Üí emit() ‚Üí _on_* ‚Üí blinker send() ‚Üí model method`. The blinker `send()` that the model called returns normally with the receiver-result list, oblivious to the view slot's failure.

The accurate v0.1 policy, then:

- The **bridge** never sees or suppresses a view-slot exception ‚Äî there is nothing for it to catch, because the exception never reaches the `_on_*` frame.
- The **model mutation** that justified the event has already committed before `send()` was called (umbrella invariants 3 + 9), and stays committed ‚Äî consistent with step-8 ¬ß6.5 "no rollback", though by a different mechanism (Qt's exception absorption, not synchronous propagation).
- A failing view slot surfaces through **Qt/Python exception reporting** (`sys.excepthook` in production; pytest-qt's automatic capture in tests ‚Äî see ¬ß6.2), **not** as an exception in model or controller code. Model/controller callers must **not** be promised synchronous propagation from view slots.
- Conversely, a **blinker-receiver** exception (i.e. a bug inside `_on_*` itself, before `emit()`) *does* propagate out of `send()` per blinker's contract ‚Äî but the bridge's `_on_*` bodies are a single `emit()` call and do not raise. The umbrella ¬ß9 synchronous-emit risk and the `_send_robust` deferral concern *blinker-side* receiver faults (a second model subscriber), not the Qt-slot faults discussed here; they remain v0.2 escalations unchanged.

App-level handling of view-slot exceptions (fatal-error dialog, structured logging via a `sys.excepthook` install, or a Qt message handler) is deferred to the future `app.py`/bootstrap spec. v0.1 fixes the behavior of the bridge only: it neither catches nor masks slot exceptions, and it makes no propagation promise to its blinker/model callers.

### 4.7 Worked example

```python
from PySide6.QtCore import QObject
from stmrr.controller.model_bridge import ModelBridge
from stmrr.model.events import ShipMovedPayload, ship_moved
from stmrr.model.state.game_state_manager import GameStateManager
from stmrr.model.state.states import MainMenuState

bridge = ModelBridge(GameStateManager(MainMenuState()))
assert isinstance(bridge.current_state, MainMenuState)  # initial render hook

received: list[object] = []
bridge.ship_moved.connect(received.append)              # a view slot would do this

payload = ShipMovedPayload(ship_id=..., from_position=..., to_position=...)
ship_moved.send(object(), payload=payload)              # the model fires the blinker event
assert received == [payload]                            # same instance, re-emitted on Qt side

bridge.teardown()                                       # releases the blinker subscriptions
ship_moved.send(object(), payload=payload)              # no longer re-emitted
assert received == [payload]                            # unchanged
```

## 5. Cross-cutting design decisions

### 5.1 Single `ModelBridge` QObject, not one QObject per signal

One `QObject` owning all four `Signal`s ‚Äî the `WorkerSignals(QObject)` idiom from Qt's own threading documentation. One object means one ownership point in `app.py`, one uniform thread affinity, and one teardown call. Per-signal QObjects would multiply ownership, parenting, and teardown surface for zero benefit at four signals. (OQ best-practice, research `‚Ä¶-open-questions.md`.)

### 5.2 Plain `QObject`, never `QObject + ABC`; `Signal`s are class-level

`class ModelBridge(QObject, ABC)` raises `TypeError: Shiboken.ObjectType.__new__(...) is not safe` ‚Äî Shiboken's metaclass conflicts with `ABCMeta`. The bridge is a plain `QObject` subclass. This is a real trap because the model layer leans heavily on ABCs (`GameState`), so the instinct to reach for `ABC` must be explicitly resisted. Likewise, a `Signal` declared inside `__init__` rather than at class scope raises `AttributeError` on first `.connect()` ‚Äî the descriptor protocol requires class-level declaration. (OQ footguns, both corroborated.)

### 5.3 `weak=False` on every blinker connection

blinker's `connect(receiver, weak=True)` default holds receivers via weak reference. The bridge's receivers are **bound methods** (`self._on_*`), whose weakref tracks the bridge instance; when the last external reference to the bridge is dropped, blinker silently stops calling them ‚Äî no error, no log (`has_receivers_for()` even keeps returning `True` for the dead receiver). For a long-lived singleton bridge this is a latent "events silently stop arriving" bug. `weak=False` makes blinker hold the bridge strongly; `teardown()` is the explicit release. The GC-survival test (¬ß6.2) pins this. (OQ headline footgun.)

### 5.4 `Signal(object)` carrying the intact frozen payload

The four payloads carry non-primitive fields (`EntityId`/`NewType`, `GridPosition`, `type[GameState]`) that cannot be expressed as Qt primitive signal types. `Signal(object)` carries the whole frozen dataclass; the receiving slot narrows it with `assert isinstance(payload, ShipMovedPayload)`. No field unpacking ‚Äî that would couple the bridge to each payload's shape and lose the single-`object` uniformity.

### 5.5 No typed `emit_*()` wrappers in v0.1 (OQ-1)

There is no parametric `Signal[T]` in PySide6 (official or `PySide6-stubs` 6.7.3.0). The bundled stub types the emit method as `def emit(self, /, *args: typing.Any) -> None` (`QtCore.pyi:11015`, verified in this venv) ‚Äî so `self.ship_moved.emit(anything)` type-checks regardless of argument type. **mypy `--strict` therefore provides NO type protection at the `.emit()` boundary.** A typed `emit_ship_moved(self, payload: ShipMovedPayload)` wrapper *would* add real static protection ‚Äî it would catch a cross-wiring bug like emitting a `DockedPayload` on `ship_moved` ‚Äî which the direct `.emit()` call cannot.

**Wrappers are nonetheless deferred** as a small-surface YAGNI tradeoff, not because mypy covers the gap (it does not). The surface is four one-line handlers, each emitting on the identically named signal with the payload it just received; the mapping is mechanical and visually verifiable in one screen. **Correctness is pinned by tests, not the type checker:** ¬ß6.2 fires the real blinker signal for each of the four and asserts the *exact payload object* arrives on the matching Qt signal (payload-identity, per-signal). The wrapper becomes worthwhile in v0.2 if a second, non-blinker caller needs to emit a bridge signal, or if the handler count grows enough that visual verification stops being reliable. View slots unavoidably annotate `(self, payload: object)` and narrow with `isinstance` ‚Äî a stub limitation, not a design choice.

### 5.6 No `@Slot` on `_on_*` in v0.1 (OQ-2)

`@Slot` is cosmetic on the `_on_*` handlers here, for two independent reasons: (a) they sit in blinker's receiver table and are invoked as plain Python callables ‚Äî Qt's meta-object slot dispatch never touches them; (b) v0.1 is single-threaded (`DirectConnection`), which needs no `@Slot` regardless. `@Slot` becomes load-bearing on the **view-slot** side (not the bridge) only under a cross-thread `QueuedConnection`, where an undecorated slot's proxy can segfault (confirmed PySide6 6.6‚Äì6.8). **v0.2 trigger to document:** *when a background model thread is introduced, add `@Slot(object)` to view slots invoked via `QueuedConnection` ‚Äî not to the bridge's blinker handlers.*

### 5.7 Two orthogonal teardown obligations + the future `app.py` wiring (OQ-4)

Shutdown has **two** independent cleanup paths that must not be conflated:

| Obligation | What it disconnects | Mechanism |
|---|---|---|
| Qt side | view slots ‚Üê bridge `Signal`s | Qt object-tree parenting: `~QObject` auto-disconnects when the parent (`MainWindow`) is destroyed |
| blinker side | bridge `_on_*` ‚Üê model events | **explicit** `bridge.teardown()` (¬ß4.5) ‚Äî Qt parenting does NOT touch blinker |

The future `app.py` owns both: it constructs the GSM, constructs `ModelBridge(manager, parent=main_window)` (Qt side handled by parenting), and calls **`bridge.teardown()` explicitly from the owner before the bridge is destroyed** ‚Äî e.g. in `MainWindow.closeEvent()`.

**Do NOT wire `bridge.destroyed.connect(bridge.teardown)`.** Verified against PySide6 6.11.0 (¬ß6.2 probe), that self-wiring does not perform the blinker disconnect: after `deleteLater()` + event processing, a subsequent blinker `send()` still invoked the bridge's `_on_*` handler (and, in the reviewer's environment, raised `RuntimeError: Signal source has been deleted` from the dead QObject). The root cause is that the blinker subscription is held with `weak=False` and is only released by the explicit `disconnect()` in `teardown()`; relying on the destroyed object's *own* bound method to run that disconnect during/after destruction is unreliable. The validated pattern is an **external owner-side call** while the bridge is still alive:

```python
# future app.py / MainWindow ‚Äî NOT built in step 9
def closeEvent(self, event):
    self._bridge.teardown()      # release blinker subs while bridge is alive
    super().closeEvent(event)
```

Qt object-tree parenting still handles the Qt-signal half automatically when the window is destroyed. This wiring is **specified, not built** in step 9 ‚Äî but ¬ß6.2 includes a lifetime test that proves the owner-side pattern: after an explicit `teardown()`, a real blinker `send()` invokes no bridge handler and raises nothing.

### 5.8 Manager injection over no-arg construction

Umbrella ¬ß8.1 locks *"`GameStateManager` ‚Ä¶ constructed once in `app.py` and passed into the bridge."* The bridge therefore takes `manager` as a required constructor argument (not the no-arg `ModelBridge()` the research sketch assumed). The `current_state` property (¬ß4.3) is the concrete consumer of that reference. No singleton, no module-level instance, no DI container ‚Äî the same rationale the umbrella gives for the GSM itself.

## 6. Testing

Bridge tests need a `QApplication`, so they live under `tests/integration/` (the home for *"pytest-qt tests for view + controller"*, `DESIGN.md` ¬ß9.1) ‚Äî **not** `tests/unit/`, which stays headless/model-only. Headless Qt is already configured (`QT_QPA_PLATFORM=offscreen` in `pyproject.toml`). **100% line + branch coverage** on `src/stmrr/controller/model_bridge.py`: `pytest --cov-reset --cov=stmrr.controller.model_bridge --cov-report=term-missing --cov-fail-under=100`. The `--cov-reset` is required: `pyproject.toml`'s `addopts` sets a repo-wide `--cov=stmrr` that would otherwise be inherited and dilute the targeted gate (same convention as the step-4 / step-8 targeted coverage gates).

### 6.1 `tests/integration/conftest.py` ‚Äî the `bridge` fixture (new; first conftest in the repo)

```python
from collections.abc import Iterator

import pytest
from pytestqt.qtbot import QtBot

from stmrr.controller.model_bridge import ModelBridge
from stmrr.model.state.game_state_manager import GameStateManager
from stmrr.model.state.states import MainMenuState


@pytest.fixture
def bridge(qtbot: QtBot) -> Iterator[ModelBridge]:
    """Fresh ModelBridge per test, with blinker teardown on exit.

    Function-scoped (NOT session): with weak=False, blinker holds each
    constructed bridge strongly, so a session-scoped or un-torn-down bridge
    would accumulate subscribers and fire the same event on multiple bridges
    across tests. `qtbot` provides the session-scoped QApplication singleton.
    A yield fixture is a generator ‚Äî annotate `Iterator[ModelBridge]`, not
    `ModelBridge` (mypy checks generator return types).
    """
    b = ModelBridge(GameStateManager(MainMenuState()))
    yield b
    b.teardown()
```

`qapp` is session-scoped by hard constraint (a `QApplication` cannot be destroyed and recreated within one process); `qtbot` is function-scoped and pulls `qapp` in implicitly. The `bridge` fixture is function-scoped with explicit `teardown()` to prevent cross-test subscriber accumulation (¬ß5.3).

## 7. Invariants

Properties step 9 contributes; a reviewer should probe these.

1. **The bridge is the sole model‚Üíview event path.** It is the only module importing both `stmrr.model.events` and `PySide6`. Enforced on both sides by import-linter: `view-does-not-import-model-events` blocks the view; the new `controller-events-via-bridge-only` contract (¬ß3) blocks every controller module except `model_bridge.py`. (DESIGN ¬ß9.1 / ADR-0003, *"mechanically enforced"*.)
2. **Qt signal names equal blinker signal names** (`ship_moved`, `docked`, `turn_advanced`, `state_changed`). A rename on either side without the other breaks the mechanical mapping; tests assert each pairing.
3. **Payloads cross the seam by identity, unmodified.** `_on_*` forwards the exact frozen dataclass instance; no copy, no field unpacking. Tested by an `is`-identity assertion.
4. **All four blinker connections use `weak=False`.** The long-lived bridge must not be silently GC-disconnected. Pinned by the GC-survival test.
5. **`teardown()` releases exactly the four blinker subscriptions and is idempotent.** It does not touch Qt-side connections (those are Qt-parenting's job). It must be called **owner-side, before the bridge is destroyed** (`MainWindow.closeEvent`), never via `self.destroyed.connect(self.teardown)` ‚Äî the latter does not perform the disconnect (¬ß5.7, SA-002).
6. **The bridge catches nothing, and a raising view slot does NOT propagate back to model/controller code.** On PySide6 6.11.0 (verified, ¬ß6.2/¬ß4.6), a slot exception during `emit()` is routed to `sys.excepthook` (pytest-qt's capture in tests); `emit()` and the model's blinker `send()` return normally. The model mutation stays committed (umbrella invariants 3 + 9; same end-state as step-8 ¬ß6.5, different mechanism). No synchronous-propagation promise is made to callers.
7. **`ModelBridge` is a plain `QObject`, not an ABC.** No `abc.ABC` in its MRO (Shiboken metaclass constraint, ¬ß5.2).
8. **`Signal`s are class-level attributes.** Not declared in `__init__` (¬ß5.2).
9. **`current_state` is read-only and delegates to the injected manager.** It is the only consumer of `self._manager`; the bridge performs no `isinstance`/`type` dispatch on the manager.
10. **One new `.importlinter` contract (`controller-events-via-bridge-only`); no umbrella amendment.** The new contract (¬ß3) conforms to ADR-0003's "enforced mechanically by import-linter" for the controller-side seam. No umbrella amendment: step 9 is the first controller module, and the model-layer umbrella spec's module table (¬ß4) and DAG (¬ß6) describe `stmrr.model.*` only ‚Äî they are not extended by a controller module. The bridge is described by `DESIGN.md` ¬ß9.1 and umbrella ¬ß3/¬ß5.2.
11. **`stmrr.controller` re-exports `ModelBridge`.** `controller/__init__.py` exposes `ModelBridge` with `__all__ == ["ModelBridge"]`, mirroring step-8's `state/__init__.py` discipline. Pinned by a ¬ß6.2 test.

## 8. Open questions

### 8.1 Locked decisions

- **Single `ModelBridge(QObject)` holding four `Signal(object)`** ‚Äî not per-signal QObjects (¬ß5.1).
- **Plain `QObject` subclass, no `ABC`; `Signal`s declared at class scope** (¬ß5.2).
- **`weak=False` on all four blinker connections; explicit `teardown()` for release** (¬ß5.3).
- **Payload carried intact as `object`; view narrows with `isinstance`** (¬ß5.4).
- **No typed `emit_*()` wrappers in v0.1** ‚Äî `.emit()` is typed `*args: Any`, so a wrapper *would* add static safety, but it is deferred as a small-surface YAGNI tradeoff with correctness pinned by per-signal payload-identity tests, not mypy (¬ß5.5, SA-004).
- **No `@Slot` on `_on_*`** ‚Äî blinker receivers, single-thread; v0.2 trigger (`QueuedConnection` across threads) documented (¬ß5.6).
- **Error policy: the bridge catches nothing; on PySide6 6.11.0 a raising view slot is routed to `sys.excepthook` (pytest-qt capture in tests) and `emit()`/`send()` return normally ‚Äî no propagation to model/controller** (¬ß4.6, SA-001). Model mutation stays committed (same end-state as step-8 ¬ß6.5).
- **`__init__(manager, parent=None)`; manager required** per umbrella ¬ß8.1; `current_state` is its consumer (¬ß5.8).
- **`current_state` read-only, delegates to manager** ‚Äî initial-render hook for the view (¬ß4.3).
- **Bridge-only scope; `app.py` wiring (owner-side `MainWindow.closeEvent` ‚Üí `bridge.teardown()`, GSM construction) specified but not built. NOT `destroyed.connect(teardown)`** (¬ß2, ¬ß5.7, SA-002).
- **One new `.importlinter` contract (`controller-events-via-bridge-only`, mirroring `blinker-only-in-events`); no umbrella amendment** ‚Äî conforms to ADR-0003's import-linter enforcement (¬ß3, invariant 10, SA-003).
- **Tests under `tests/integration/` (qtbot) via a function-scoped `bridge` fixture in a new `tests/integration/conftest.py`**; the single-seam controller rule is enforced by the import-linter contract (¬ß3), not a test (¬ß6.1).

### 8.2 Still open (v0.2 concerns, NOT v0.1 blockers)

1. **Parametric `Signal[T]` typing.** No path in official or `PySide6-stubs` stubs as of May 2026 (needs a mypy plugin or PEP 646 stubs). View slots eat `object` + `isinstance` for now. Track `python-qt-tools/PySide6-stubs`. (Research OQ-1.)
2. **`@Slot` + `QueuedConnection` when a model worker thread lands.** v0.2 must add `@Slot(object)` to *view* slots crossing threads (not the bridge handlers), or a non-decorated proxy may segfault on PySide6 6.6‚Äì6.8. (Research OQ-2.)
3. **`_send_robust` fault isolation.** When a second non-bridge subscriber connects to any signal, a raising receiver could skip the bridge (or vice versa). Implement the per-receiver try/except utility then (umbrella ¬ß9 / step-7 ¬ß9.2).
4. **Python 3.14 free-threading.** blinker's `receivers` dict is unlocked; safe under v0.1's single-thread assumption, a data race under a free-threaded multi-thread model. Confine model mutation/sends to one thread or lock `send()` in v0.2. (Research, compatibility.)
5. **A bridge registry / multi-bridge ownership.** If a second `ModelBridge` instance or multiple independent view subscribers appear, revisit single-instance ownership and the context-manager-vs-`teardown()` choice. (Research OQ-4.)
