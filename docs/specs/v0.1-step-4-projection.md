## 1. Purpose

`src/stmrr/view/scene/projection.py` is the single owner of isometric math: forward projection (world `(x, y, z)` → scene `(sx, sy)`), inverse projection (scene `(sx, sy)` + active z → world `GridPosition` or `None`), and painter-ordering (`zValue` formula for `QGraphicsItem`). Despite living under `src/stmrr/view/`, it imports zero PySide6 — pure Python, fully unit-testable headless, 100% line + branch.

It lands first among view-layer modules per `docs/design/tech-stack-pyside6.md` §11 because every subsequent rendering module (`map_view.py`, `grid_scene.py`, `grid_cell_item.py`, `starship_item.py`) depends on it. Build vertically through the stack early; iso math is the bottom of the view stack.

## 2. Scope

**In-scope:**

- `src/stmrr/view/scene/projection.py` — three public functions, four module-scope constants.
- `src/stmrr/view/scene/__init__.py` re-exports.
- `src/stmrr/view/__init__.py` and `src/stmrr/view/scene/` package markers if not already present.
- Unit tests under `tests/unit/view/scene/test_projection.py`.

**Out-of-scope (deliberately deferred):**

- `QTransform` adapter — `view/scene/map_view.py` (step 6) decides between scene-level transform and per-item precompute. Both paths consume the pure-function API defined here; neither is locked in this spec. See §9.1 for the rationale.
- `setSceneRect()` math, zoom/pan, mouse hit-testing, click → controller dispatch — all step 6 (`map_view.py` / `grid_scene.py`).
- Grid line rendering and the §3.3 question of whether grid lines benefit from a scene-level skew transform. Step 6's call.
- Z-level opacity (active 1.0, inactive 0.35 per `tech-stack-pyside6.md` §3.1) — `grid_scene.py`'s job to apply per item; projection only computes the `zValue` for painter ordering.
- Sector-bounds checking. `SectorMap` owns "is `(x, y, z)` inside the active sector"; projection only enforces non-negativity, the same boundary `GridPosition` enforces in step 3.
- Camera/viewport/scroll/zoom transforms — `MapView` (step 6).
- Any Qt code, signal wiring, controller bridging.

## 4. Constants and formulas

### 4.1 Constants

```python
TILE_WIDTH: int = 64       # iso diamond width in scene pixels
TILE_HEIGHT: int = 32      # iso diamond height in scene pixels (2:1 dimetric)
Z_OFFSET: int = 32         # vertical scene-pixel rise per +1 world z
MAX_Z_DEPTH: int = 10      # painter-ordering headroom; > max GridPosition.z
```

`TILE_WIDTH:TILE_HEIGHT = 2:1` is the standard dimetric (commonly called "isometric" in game dev) ratio. `Z_OFFSET = TILE_HEIGHT` gives one-tile-height vertical rise per z-step — the boring-correct convention used by Diablo, RimWorld, Project Zomboid, and every other tile-iso engine — so a sprite at `(0, 0, 1)` sits visually directly above `(0, 0, 0)`'s diamond top vertex.

`MAX_Z_DEPTH = 10` is locked headroom over the §4.3 sector ceiling of 7. Any future ADR raising the sector z-ceiling past 9 must confront this constant; see invariant §8.5.

### 4.2 Iso layout choice

**Diamond / screen iso, x-axis northeast, y-axis southeast.** `GridPosition(1, 0, 0)` projects northeast of the origin; `(0, 1, 0)` projects southeast. Higher z renders higher on screen (subtracts from `sy` because Qt's scene-y axis points down). This is the standard `world_to_scene` shape used in every tile-iso reference engine; named explicitly here so a future reader knows immediately which iso variant they're looking at and doesn't try to map staggered-iso intuition onto the formula.

### 4.3 Forward formula

```
sx = (x - y) * TILE_WIDTH / 2
sy = (x + y) * TILE_HEIGHT / 2 - z * Z_OFFSET
```

`world_to_scene` returns `(sx, sy)` as `tuple[float, float]`. `sx` can be negative for cells with `y > x` — that is correct and intended (origin sits at world `(0, 0, 0)`, not at the leftmost visible pixel). `MapView`'s `setSceneRect()` in step 6 handles negative scene coordinates natively.

### 4.4 Inverse formula

```
adjusted_sy = sy + z * Z_OFFSET          # undo the z-lift
raw_x = sx / TILE_WIDTH + adjusted_sy / TILE_HEIGHT
raw_y = adjusted_sy / TILE_HEIGHT - sx / TILE_WIDTH
rounded_x = round(raw_x)
rounded_y = round(raw_y)
if rounded_x < 0 or rounded_y < 0:
    return None
return GridPosition(rounded_x, rounded_y, z)
```

**Round-then-check.** Negativity is checked on `rounded_x` and `rounded_y`, never on `raw_x` or `raw_y`. A click at `sx = -1.0, sy = 0.0` is visually 99% inside the origin tile; it must snap to `(0, 0, z)`, not vanish. Only clicks more than half a tile west or north of the origin's iso position at the active z-level actually fall out of bounds.

**Active-layer offset gotcha.** The "origin" of inverse projection is the active z-level's projected position, not scene `(0, 0)`. At `z = 3`, world `(0, 0, 3)` projects to scene `(0, -96)` because `Z_OFFSET * 3 = 96`. A mouse click at scene `(0, 0)` while the active layer is z=3 inverse-projects to `(3, 3, 3)`, not `(0, 0, 3)` — that's mathematically correct (the click is 96 scene-pixels below the projected layer origin, which is 3 tiles down the iso anti-diagonal). Step 6's mouse handler must pass `sx`/`sy` in scene coordinates, not viewport coordinates, and must understand that "click at scene origin" depends on which layer is active.

**Round-to-nearest.** Python's `round()` (banker's rounding for `.5` ties) is acceptable because the iso-cell hit area is symmetric about its center; the rounding-mode choice doesn't bias hit-testing in practice and the formula is documented as "nearest cell to the click point."

## 5. Function contracts

### 5.1 `world_to_scene`

```python
def world_to_scene(pos: GridPosition) -> tuple[float, float]: ...
```

| Aspect | Detail |
|---|---|
| Semantics | Returns `(sx, sy)` per §4.3. |
| Complexity | O(1). |
| Errors | None. `GridPosition` validation already enforced non-negative integer coordinates upstream. |
| Examples | `world_to_scene(GridPosition(0, 0, 0))` → `(0.0, 0.0)` |
| | `world_to_scene(GridPosition(1, 0, 0))` → `(32.0, 16.0)` |
| | `world_to_scene(GridPosition(0, 1, 0))` → `(-32.0, 16.0)` |
| | `world_to_scene(GridPosition(0, 0, 1))` → `(0.0, -32.0)` |
| | `world_to_scene(GridPosition(19, 19, 6))` → `(0.0, 416.0)` (max-sector corner) |

### 5.2 `scene_to_world`

```python
def scene_to_world(sx: float, sy: float, z: int) -> GridPosition | None: ...
```

| Aspect | Detail |
|---|---|
| Semantics | Inverse-projects `(sx, sy)` against the active z-level per §4.4. Round-to-nearest snapping; `None` if the rounded result has any negative axis. |
| Complexity | O(1). |
| Errors | `ValueError` if `z < 0`. Negative-z input is a caller bug at the function boundary, not a "click missed" — same taxonomy as `GridPosition.__post_init__` (`ValueError` on invalid value, `TypeError` on wrong type, see step 3 spec §4.2). `TypeError` if `z` is not `int` (or is `bool`); `sx`/`sy` accept `int` or `float` since `QPointF.x()` returns `float`. |
| Examples | `scene_to_world(0.0, 0.0, 0)` → `GridPosition(0, 0, 0)` |
| | `scene_to_world(32.0, 16.0, 0)` → `GridPosition(1, 0, 0)` |
| | `scene_to_world(40.0, 18.0, 0)` → `GridPosition(1, 0, 0)` (rounds to nearest) |
| | `scene_to_world(-1.0, 0.0, 0)` → `GridPosition(0, 0, 0)` (raw x ≈ -0.016, rounds to 0 — demonstrates post-rounding negativity rule) |
| | `scene_to_world(-64.0, 0.0, 0)` → `None` (rounds to negative x) |
| | `scene_to_world(0.0, -96.0, 3)` → `GridPosition(0, 0, 3)` (z=3 layer's projected origin sits at scene `(0, -96)`, not `(0, 0)` — see §4.4 note) |
| | `scene_to_world(0.0, 0.0, -1)` → raises `ValueError` |

### 5.3 `z_value_for`

```python
def z_value_for(pos: GridPosition) -> int: ...
```

| Aspect | Detail |
|---|---|
| Semantics | Returns `(pos.x + pos.y) * MAX_Z_DEPTH + pos.z`. The integer is intended for `QGraphicsItem.setZValue()`; higher values paint in front. |
| Complexity | O(1). |
| Errors | None. |
| Examples | `z_value_for(GridPosition(0, 0, 0))` → `0` |
| | `z_value_for(GridPosition(1, 0, 0))` → `10` |
| | `z_value_for(GridPosition(0, 1, 0))` → `10` (intentional tie — see §5.3 note below) |
| | `z_value_for(GridPosition(0, 0, 1))` → `1` |
| | `z_value_for(GridPosition(2, 1, 3))` → `33` |

**Note on intentional ties.** Two distinct positions with the same `(x + y, z)` produce identical `zValue` (e.g., `(1, 0, 0)` and `(0, 1, 0)`). This is **permitted by design**. Same-cell occupancy is forbidden upstream (`SectorMap` enforces one entity per `(x, y, z)` cell), so the only way two items collide on `zValue` is if they sit in the same iso anti-diagonal at the same z — a rare arrangement where Qt's stable insertion-order tie resolution suffices. The painter-correctness invariant is "no occluding pair inverts" (§8.3, §8.4), not "no ties period." Trying to enforce per-cell uniqueness in `z_value_for` would duplicate `SectorMap`'s job and couple projection to a constraint outside its concern.

## 8. Invariants

Properties the rest of the view layer relies on; reviewer should probe these.

1. `world_to_scene` is total — defined for every `GridPosition` (which already guarantees non-negative integer coordinates). Never raises.
2. `world_to_scene(GridPosition(0, 0, 0)) == (0.0, 0.0)` exactly. Origin sits at scene origin.
3. **Painter ordering by anti-diagonal:** for any `p1, p2` with `p1.x + p1.y < p2.x + p2.y`, `z_value_for(p1) < z_value_for(p2)`. (Cells further "back" in iso paint behind cells further "forward.")
4. **Painter ordering by z at same anti-diagonal:** for any `p1, p2` with `p1.x + p1.y == p2.x + p2.y` and `p1.z < p2.z`, `z_value_for(p1) < z_value_for(p2)`. (Higher altitude paints in front at the same depth.)
5. **z-headroom invariant:** for every `GridPosition` constructible under `DESIGN.md` §4.3's sector ceiling of `z ≤ 6`, `pos.z < MAX_Z_DEPTH`. A future ADR raising the ceiling past z=9 must update `MAX_Z_DEPTH` or the ordering invariants (§8.3, §8.4) silently break.
6. **Round-trip identity:** `scene_to_world(*world_to_scene(p), p.z) == p` for every `GridPosition p`. (This is the core formula-consistency property; §7 makes it a parametrized test over the full 20×20×7 sector.)
7. `scene_to_world` returns `None` only when post-rounding `x` or `y` is negative; never returns a `GridPosition` with negative coordinates (impossible — `GridPosition.__post_init__` would reject it).
8. `scene_to_world` raises `ValueError` exactly when `z < 0`, and `TypeError` exactly when `z` is non-int (matching step 3's `GridPosition` validation taxonomy).
9. The module imports zero PySide6 / Qt symbols. Verifiable by `import-linter` and by inspection: only `stdlib` and `stmrr.model.world.GridPosition` appear in imports.

## 9. Open questions for review

### 9.1 Locked decisions

- **Pure-function module, no PySide6 import.** First view-layer module deliberately keeps Qt out. The §3.3 "apply iso transform once at scene level via `QTransform`" recommendation is a step 6 (`map_view.py`) decision — it applies to the grid-line layer where vector skewing is fine, but is wrong as a blanket scene-level transform for sprite items (sprites would skew unless individually counter-transformed). Per-item precompute via `world_to_scene` is the likely step 6 conclusion for the sprite layer; either way, both paths consume this module's pure-function API. No `as_qtransform()` adapter in this spec.
- **`(TILE_WIDTH, TILE_HEIGHT) = (64, 32)`, `Z_OFFSET = 32`.** A 20×20 sector spans ~1216×608 scene-pixels at 1× zoom — fits comfortably in the 1600×1000 minimum window with chrome budget intact. `Z_OFFSET = TILE_HEIGHT` is the standard one-tile-rise convention.
- **`MAX_Z_DEPTH = 10`.** Locked headroom over the §4.3 ceiling of 7. Invariant §8.5 pins it.
- **Diamond / screen iso (x → NE, y → SE).** Formula in §4.3, §4.4 is explicit and documented; not staggered-iso.
- **World origin, negative `sx` permitted.** Decouples projection from sector dimensions. `MapView`'s `setSceneRect()` in step 6 handles negative scene coordinates natively.
- **Constants live in `projection.py` as their permanent location**, not deferred-pending-step-6. Tile pixel dimensions are *geometric* parameters of the iso formula, not theme. Theme (`view/theme/palette.py`) holds colors and fonts; projection holds geometry. See §10 cross-ref note.
- **`scene_to_world` signature: `(sx: float, sy: float, z: int) -> GridPosition | None`.** Single function, returns `None` when post-rounding result has any negative axis. Sector-bounds checking is `SectorMap`'s job.
- **Round-then-check negativity.** Round first, then check `rounded_x < 0 or rounded_y < 0`. Clicks just outside the origin tile snap to `(0, 0)` rather than vanishing.
- **Negative z is a `ValueError` at the function boundary**, not a `None`. Caller bug, not a "click missed."
- **Painter formula `(x + y) * MAX_Z_DEPTH + z`, ties permitted.** The painter-correctness invariant is "no occluding pair inverts" (§8.3, §8.4), not "no ties." Same-anti-diagonal-same-z ties are resolved by Qt's stable insertion order; `SectorMap`'s per-cell uniqueness rule prevents the only case where ties would actually be visible.
