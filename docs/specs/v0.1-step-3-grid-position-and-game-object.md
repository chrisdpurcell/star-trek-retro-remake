## 1. Purpose

Two foundational model modules: `model/world/grid_position.py` (immutable 3-D coordinate value type) and `model/entities/game_object.py` (base class every active entity ‚Äî `Starship`, `Station`, `Anomaly`, `Projectile` ‚Äî eventually inherits from). They land first because every other model module depends on at least one. The spec locks the smallest API that supports v0.1's vertical slice (selection, click-to-move, Dock, AP debit, `ShipMoved`) while leaving room for v0.2 combat and pathfinding to slot in without renames. Cross-cutting concerns (events, save/load ID lifecycle, state machine, action contracts) live in [`v0.1-model-layer.md`](v0.1-model-layer.md) ¬ß5.

## 2. Scope

**In-scope:**

- The two modules above.

## 4. `GridPosition` specification

### 4.1 Class declaration

```python
from __future__ import annotations

from dataclasses import dataclass
from math import sqrt
from typing import Iterator


@dataclass(frozen=True, slots=True)
class GridPosition:
    x: int
    y: int
    z: int
```

`frozen=True` gives equality, hashing, and immutability for free; `slots=True` drops the per-instance `__dict__` so thousands of positions stay cheap. `DESIGN.md` ¬ß9.1: *"Logical coordinates are cartesian `(x, y, z)` stored on items."*

### 4.2 Validation contract

`__post_init__` runs two checks per axis:

1. **Type check:** `isinstance(value, int) and not isinstance(value, bool)` ‚Äî raises `TypeError` if it fails. Python type hints (`x: int`) do **not** enforce `int` at runtime; `bool` is an `int` subclass and slips through unchecked. `numpy.int64`, `Fraction`, and `Decimal` would also pass an `int` annotation in mypy but fail this runtime check. Caught fast at the boundary.
2. **Range check:** `value >= 0` ‚Äî raises `ValueError` if it fails. Sector dimensions (max 20√ó20√ó7 per `DESIGN.md` ¬ß4.3) are `SectorMap`'s responsibility; negatives are nonsense at the position level regardless of containing sector.

**Why both:** `ValueError` (not `TypeError`) for negatives because the type is correct ‚Äî the value is wrong. `TypeError` for non-int because the type is wrong. The two checks are intentionally separate so error messages name the actual failure mode. Resolves SA-008.

### 4.3 Method contracts

| Method | Signature | Semantics | Complexity | Errors |
|---|---|---|---|---|
| `manhattan_distance` | `(self, other: GridPosition) -> int` | `\|Œîx\| + \|Œîy\| + \|Œîz\|` | O(1) | none |
| `chebyshev_distance` | `(self, other: GridPosition) -> int` | `max(\|Œîx\|, \|Œîy\|, \|Œîz\|)` | O(1) | none |
| `euclidean_distance` | `(self, other: GridPosition) -> float` | `sqrt(Œîx¬≤ + Œîy¬≤ + Œîz¬≤)` | O(1) | none |
| `is_adjacent` | `(self, other: GridPosition) -> bool` | `chebyshev_distance(other) == 1` | O(1) | none ‚Äî explicitly `False` for `self` |
| `neighbors` | `(self) -> Iterator[GridPosition]` | yields up-to-26 valid (non-negative) adjacent positions; deterministic order sorted by `(dz, dy, dx)` lexicographically over `{-1, 0, +1}`¬≥ excluding `(0, 0, 0)` | O(1) (‚â§26 items) | none |
| `with_x` | `(self, x: int) -> GridPosition` | returns a copy with new `x` | O(1) | `ValueError` if `x < 0` (re-validated via `__post_init__`) |
| `with_y` | `(self, y: int) -> GridPosition` | returns a copy with new `y` | O(1) | `ValueError` if `y < 0` |
| `with_z` | `(self, z: int) -> GridPosition` | returns a copy with new `z` | O(1) | `ValueError` if `z < 0` |
| `origin` (classmethod) | `cls() -> GridPosition` | returns `cls(0, 0, 0)` | O(1) | none |

**Examples:**

```python
GridPosition(0, 0, 0).manhattan_distance(GridPosition(1, 2, 3))   # 6
GridPosition(0, 0, 0).chebyshev_distance(GridPosition(1, 2, 3))   # 3
GridPosition(0, 0, 0).euclidean_distance(GridPosition(2, 3, 6))   # 7.0
GridPosition(5, 5, 0).is_adjacent(GridPosition(5, 5, 0))          # False
list(GridPosition(0, 0, 0).neighbors())                            # 7 entries (corner)
GridPosition(3, 4, 5).with_z(6)                                    # GridPosition(3, 4, 6)
```

`neighbors()` order is non-negotiable: AI pathfinding tie-breaks on iteration order, and replay tests need bit-stable behavior. The yielded count depends on which axes touch zero ‚Äî **interior 26, face 17 (one axis at 0), edge 11 (two axes at 0), corner 7 (all three at 0)**. These are the four boundary classes the test suite must cover; counts derive directly from the rule "skip dz/dy/dx that would push an axis below 0".

### 4.4 Dunder behavior

`__repr__`, `__eq__`, `__hash__` come from `@dataclass(frozen=True)` ‚Äî no custom code. Equality is *value*-based: same `(x, y, z)` ‚áí equal. This is the **opposite** of `GameObject` equality (¬ß5.4); the contrast is intentional.

## 5. `GameObject` specification

### 5.1 Module-level types and ID generator

```python
from __future__ import annotations

from typing import NewType

from stmrr.model.world.grid_position import GridPosition

EntityId = NewType("EntityId", int)

# TODO(v0.2+): wrap _next_entity_id in threading.Lock once AI processing moves
# to QThreadPool. See docs/design/DESIGN.md ¬ß9.4 "Threading". Single-threaded
# turn loop means v0.1 does not need synchronization.
_NEXT_ENTITY_ID: int = 0


def _next_entity_id() -> EntityId:
    global _NEXT_ENTITY_ID
    _NEXT_ENTITY_ID += 1
    return EntityId(_NEXT_ENTITY_ID)
```

`EntityId` is a `NewType` ‚Äî a zero-cost mypy hint that catches `int` ‚Üî `EntityId` confusion while staying TOML-serializable as a plain integer (relevant to ADR-0004, [`docs/adr/0004-toml-not-pickle.md`](../adr/0004-toml-not-pickle.md)). The TODO sits at module scope above the counter so it is the first thing a reader sees, not unreachable code after a `return` (resolves SA-009).

**v0.1 decision (locks SA-005):** `EntityId` is a monotone integer for v0.1. The counter resets on process restart, which is acceptable because v0.1 has no save/load (v0.2 scope per `DESIGN.md` ¬ß10.1). The migration path to UUID4 (or to int IDs persisted in the save file) is a v0.2 persistence-spec problem ‚Äî separate spec, separate review. The model layer's `EntityId` *type* stays `NewType("EntityId", int)` either way; only the generator implementation changes.

### 5.2 Why `GameObject` is *not* a frozen dataclass

`Starship`, `Station`, and `Projectile` need mutable position plus component composition (`DESIGN.md` ¬ß9.1: *"Components are simple objects with their own state and behavior, composed onto entities rather than inherited"*). Frozen dataclasses force `dataclasses.replace` for every position update, breaking reference semantics the controller bridge relies on for animation and cascading through every component. Plain class wins.

### 5.3 Class declaration and method contracts

```python
class GameObject:
    """Base class for all active entities on a sector grid.

    Identity is per-instance (lifetime ID), not per-position. See ¬ß5.4 for
    the equality contract.
    """

    def __init__(self, position: GridPosition) -> None:
        self.id: EntityId = _next_entity_id()
        self.position: GridPosition = position
        self.active: bool = True

    def move_to(self, position: GridPosition) -> None: ...
    def deactivate(self) -> None: ...
```

| Method | Semantics | Errors |
|---|---|---|
| `__init__(self, position)` | assigns a fresh `EntityId`, sets `position`, sets `active=True`. No validation ‚Äî `position` is already validated. | none |
| `move_to(self, position)` | reassigns `self.position`. Does **not** emit events; that's the subclass's job. | none |
| `deactivate(self)` | sets `self.active = False`. Idempotent ‚Äî calling on an already-inactive object is a no-op, no error, no warning. | none |

Idempotent `deactivate` matches turn-based reality: the same destruction event may be observed by multiple subscribers; a second call must not throw.

### 5.4 Equality contract

Identity-based:

```python
def __eq__(self, other: object) -> bool:
    return self is other

def __hash__(self) -> int:
    return id(self)
```

This matches default `object` behavior, but writing it explicitly documents the contract loudly so a future subclass author doesn't override `__eq__` with value-equality (a footgun once entities enter pathfinding open-lists). Two `Starship` instances with identical state are **not** equal ‚Äî different ships.

### 5.5 `__repr__`

```python
def __repr__(self) -> str:
    return (
        f"{type(self).__name__}(id={self.id}, "
        f"position={self.position}, active={self.active})"
    )
```

`type(self).__name__` ensures subclass reprs read correctly: `Starship(id=‚Ä¶, ‚Ä¶)`, not `GameObject(‚Ä¶)`.

### 5.6 Subclass implications

Anticipated subclasses (`Starship`, `Station`, `Anomaly`, `Projectile`) inherit a module-scope ID generator (no reinvention), a mutable `position` (so subclass `move_to` overrides can layer cost logic, AP debit, and event emission around the base assignment), and `active` as the universal soft-delete flag. Hard removal from a `SectorMap` is a separate responsibility.

## 8. Invariants

Properties the rest of the model layer relies on; reviewer should probe these.

1. `GridPosition.x`, `.y`, `.z` are non-negative integers for any constructed instance.
2. A `GridPosition` is immutable post-construction; assignment raises `dataclasses.FrozenInstanceError`.
3. The three distance methods are symmetric: `a.f(b) == b.f(a)`.
4. All three distance methods return `0` (or `0.0`) for self-distance.
5. `is_adjacent(self, self)` is `False`.
6. Every position yielded by `neighbors()` has non-negative coordinates and Chebyshev distance 1 from `self`.
7. `EntityId` values from `_next_entity_id()` are strictly monotonically increasing within a process lifetime.
8. `GameObject` equality is reference identity: `a == b` iff `a is b`. Two instances are never equal regardless of attribute values.
9. `deactivate` is idempotent.
10. `move_to` does not emit events, does not validate sector bounds, does not interact with other entities.

## 9. Open questions for review

### 9.1 Decisions locked from review pass 1

- **EntityId for v0.1: monotone `int`.** UUID4 migration is a v0.2 persistence-spec concern. (Was Q3, locks SA-005.)
- **Validation taxonomy: `TypeError` for non-int, `ValueError` for negatives.** Two distinct checks per axis (was Q5; see ¬ß4.2 above). The earlier "either-or" framing is wrong ‚Äî both apply.

### 9.2 Still open

1. **`int` vs. `Coordinate` `NewType` for axis fields.** A `Coordinate = NewType("Coordinate", int)` would surface mixing with AP/hull values, but every literal becomes `Coordinate(0)`. Worth it, or noise? (No persistence implications either way.)
2. **`slots=True` and third-party validator friction.** pydantic v2 handles slotted dataclasses, but other validators may not when persistence lands. Is the per-instance memory saving worth the long-tail risk, or drop `slots=True` now?
3. **`neighbors()` as method or free function?** `tcod`'s graph adapter wants a callable `(GridPosition) -> Iterable[GridPosition]`, which `obj.neighbors` satisfies ‚Äî leaning method, but a free function `neighbors(pos)` would keep the dataclass dependency-free of iteration logic.
4. **Should `_next_entity_id()` accept a seed/reset hook for tests?** Tests currently rely on monotonic increase across the test session. A `reset_for_test()` helper would let each test start at 1, which is more readable but adds an export that production code must never call.
