## 1. Purpose

Two foundational model modules: `model/world/grid_position.py` (immutable 3-D coordinate value type) and `model/entities/game_object.py` (base class for every active entity). They land first because every other model module depends on at least one. The spec locks the smallest API supporting v0.1's vertical slice while leaving room for v0.2. Cross-cutting concerns live in [`v0.1-model-layer.md`](v0.1-model-layer.md) В§5.

## 2. Scope

**In-scope:**

- The two modules above.
- Their `__init__.py` re-exports under `model/world/` and `model/entities/`.
- Unit tests under `tests/unit/model/world/` and `tests/unit/model/entities/`.

**Out-of-scope (deliberately deferred):**

- Subclasses (`Starship`, `Station`). Later steps; this spec only fixes the base contract.

## 4. `GridPosition` specification

### 4.1 Class declaration

```python
from __future__ import annotations

from dataclasses import dataclass
from math import sqrt
from typing import Iterator


@dataclass(frozen=True, slots=True)
class GridPosition:
    x: int
    y: int
    z: int
```

`frozen=True` gives equality, hashing, and immutability for free; `slots=True` drops the per-instance `__dict__` so thousands of positions stay cheap. `DESIGN.md` В§9.1: *"Logical coordinates are cartesian `(x, y, z)` stored on items."*

### 4.2 Validation contract

`__post_init__` runs two checks per axis, in this order:

1. **Type check:** `isinstance(value, int) and not isinstance(value, bool)` вЂ” raises `TypeError` if it fails. mypy strict will reject `Fraction`, `Decimal`, or `float` against an `int` parameter, but untyped callers, TOML deserializers, and external data can still hand in anything at runtime; the guard catches them at the boundary. `bool` is the residual footgun вЂ” an `int` subclass that slips past mypy.
2. **Range check:** `value >= 0` вЂ” raises `ValueError` if it fails. Sector dimensions (max 20Г—20Г—7 per `DESIGN.md` В§4.3) are `SectorMap`'s responsibility; negatives are nonsense at the position level regardless of containing sector.

Separate exception types so messages name the actual failure mode.

### 4.3 Method contracts

| Method | Signature | Semantics | Complexity | Errors |
|---|---|---|---|---|
| `manhattan_distance` | `(self, other: GridPosition) -> int` | `\|О”x\| + \|О”y\| + \|О”z\|` | O(1) | none |
| `chebyshev_distance` | `(self, other: GridPosition) -> int` | `max(\|О”x\|, \|О”y\|, \|О”z\|)` | O(1) | none |
| `euclidean_distance` | `(self, other: GridPosition) -> float` | `sqrt(О”xВІ + О”yВІ + О”zВІ)` | O(1) | none |
| `is_adjacent` | `(self, other: GridPosition) -> bool` | `chebyshev_distance(other) == 1` | O(1) | none вЂ” explicitly `False` for `self` |
| `neighbors` | `(self) -> Iterator[GridPosition]` | yields up-to-26 valid (non-negative) adjacent positions; deterministic order sorted by `(dz, dy, dx)` lexicographically over `{-1, 0, +1}`Ві excluding `(0, 0, 0)` | O(1) (в‰¤26 items) | none |
| `with_x` | `(self, x: int) -> GridPosition` | returns a copy with new `x` | O(1) | `ValueError` if `x < 0` (re-validated via `__post_init__`) |
| `with_y` | `(self, y: int) -> GridPosition` | returns a copy with new `y` | O(1) | `ValueError` if `y < 0` |
| `with_z` | `(self, z: int) -> GridPosition` | returns a copy with new `z` | O(1) | `ValueError` if `z < 0` |
| `origin` (classmethod) | `cls() -> GridPosition` | returns `cls(0, 0, 0)` | O(1) | none |

**Examples:**

```python
GridPosition(0, 0, 0).manhattan_distance(GridPosition(1, 2, 3))   # 6
GridPosition(0, 0, 0).chebyshev_distance(GridPosition(1, 2, 3))   # 3
GridPosition(0, 0, 0).euclidean_distance(GridPosition(2, 3, 6))   # 7.0
GridPosition(5, 5, 0).is_adjacent(GridPosition(5, 5, 0))          # False
list(GridPosition(0, 0, 0).neighbors())                            # 7 entries (corner)
GridPosition(3, 4, 5).with_z(6)                                    # GridPosition(3, 4, 6)
```

`neighbors()` order is non-negotiable: AI pathfinding tie-breaks on iteration order; replay tests need bit-stable behavior. Counts depend on which axes touch zero вЂ” **interior 26, face 17 (one axis at 0), edge 11 (two), corner 7 (three)**. These are the four boundary classes the test suite must cover.

### 4.4 Dunder behavior

`__repr__`, `__eq__`, `__hash__` come from `@dataclass(frozen=True)` вЂ” no custom code. Equality is *value*-based: same `(x, y, z)` в‡’ equal. This is the **opposite** of `GameObject` equality (В§5.4); the contrast is intentional.

## 5. `GameObject` specification

### 5.1 Module-level types and ID generator

```python
from __future__ import annotations

from typing import NewType

from stmrr.model.world.grid_position import GridPosition

EntityId = NewType("EntityId", int)

# TODO(v0.2+): wrap _next_entity_id in threading.Lock once AI processing moves
# to QThreadPool. See docs/design/DESIGN.md В§9.4 "Threading". Single-threaded
# turn loop means v0.1 does not need synchronization.
_NEXT_ENTITY_ID: int = 0


def _next_entity_id() -> EntityId:
    global _NEXT_ENTITY_ID
    _NEXT_ENTITY_ID += 1
    return EntityId(_NEXT_ENTITY_ID)
```

`EntityId` is a `NewType` вЂ” a zero-cost mypy hint catching `int` в†” `EntityId` confusion while staying TOML-serializable (ADR-0004, [`docs/adr/0004-toml-not-pickle.md`](../adr/0004-toml-not-pickle.md)).

**v0.1 decision: monotone integer.** Counter resets on process restart; acceptable because v0.1 has no save/load. v0.2 persistence spec decides whether to persist the counter, switch to UUID4, or rebuild ID maps at load. The type stays `NewType("EntityId", int)` either way.

### 5.2 Why `GameObject` is *not* a frozen dataclass

Subclasses need mutable position plus component composition (`DESIGN.md` В§9.1). Frozen dataclasses force `dataclasses.replace` per update, breaking reference semantics the controller bridge needs for animation. Plain class wins.

### 5.3 Class declaration and method contracts

```python
class GameObject:
    """Base class for all active entities on a sector grid.

    Identity is per-instance (lifetime ID), not per-position. See В§5.4 for
    the equality contract.
    """

    def __init__(self, position: GridPosition) -> None:
        self.id: EntityId = _next_entity_id()
        self.position: GridPosition = position
        self.active: bool = True

    def deactivate(self) -> None: ...
```

| Method | Semantics | Errors |
|---|---|---|
| `__init__(self, position)` | assigns a fresh `EntityId`, sets `position`, sets `active=True`. No validation вЂ” `position` is already validated. | none |
| `deactivate(self)` | sets `self.active = False`. Idempotent вЂ” calling on an already-inactive object is a no-op, no error, no warning. | none |

Idempotent `deactivate`: the same destruction event may be observed by multiple subscribers; a second call must not throw.

**Why no base `move_to`.** Every subclass that moves does so under different action semantics вЂ” AP cost, adjacency check, event emission, sometimes a sector-map lookup. A base `move_to(self, position)` would either be empty boilerplate or get overridden by `Starship.move_to(target, sector_map)` with a different signature, which mypy `--strict` rejects as `[override]`. The cleanest contract is: the base owns identity (`id`, `position` field, `active`), subclass action methods own mutation (assigning `self.position` directly inside their own precondition + AP-debit + event-emit sequence). Companion spec [`v0.1-model-layer.md`](v0.1-model-layer.md) В§5.6.1 specifies `Starship.move_to` against this contract.

### 5.4 Equality contract

Identity-based:

```python
def __eq__(self, other: object) -> bool:
    return self is other

def __hash__(self) -> int:
    return id(self)
```

This matches default `object` behavior; writing it explicitly stops a future subclass author from overriding `__eq__` with value-equality (a footgun once entities enter pathfinding open-lists). Two `Starship` instances with identical state are **not** equal вЂ” different ships.

### 5.5 `__repr__`

```python
def __repr__(self) -> str:
    return (
        f"{type(self).__name__}(id={self.id}, "
        f"position={self.position}, active={self.active})"
    )
```

`type(self).__name__` ensures subclass reprs read correctly: `Starship(id=вЂ¦, вЂ¦)`, not `GameObject(вЂ¦)`.

### 5.6 Subclass implications

Subclasses (`Starship`, `Station`, `Anomaly`, `Projectile`) inherit the module-scope ID generator, the mutable `position` attribute (subclass action methods assign it directly inside their own precondition + AP-debit + event-emit sequence вЂ” there is no base `move_to` to override), and `active` as the universal soft-delete flag. Hard removal from `SectorMap` is separate.

## 8. Invariants

Properties the rest of the model layer relies on; reviewer should probe these.

1. `GridPosition.x`, `.y`, `.z` are non-negative integers for any constructed instance.
2. A `GridPosition` is immutable post-construction; assignment raises `dataclasses.FrozenInstanceError`.
3. The three distance methods are symmetric: `a.f(b) == b.f(a)`.
4. All three distance methods return `0` (or `0.0`) for self-distance.
5. `is_adjacent(self, self)` is `False`.
6. Every position yielded by `neighbors()` has non-negative coordinates and Chebyshev distance 1 from `self`.
7. `EntityId` values from `_next_entity_id()` are strictly monotonically increasing within a process lifetime.
8. `GameObject` equality is reference identity: `a == b` iff `a is b`. Two instances are never equal regardless of attribute values.
9. `deactivate` is idempotent.
10. `GameObject` exposes no action methods вЂ” only identity (`id`, `position`, `active`) plus `deactivate`. State mutation, event emission, sector-bounds validation, and inter-entity interaction live on subclass action methods exclusively.

## 9. Open questions for review

### 9.1 Locked decisions

- **EntityId for v0.1: monotone `int`.** UUID4 migration is a v0.2 persistence-spec concern.
- **Validation taxonomy: `TypeError` for non-int, `ValueError` for negatives.** Two distinct checks per axis вЂ” both apply.
- **Axis fields stay `int`, not a `Coordinate` `NewType`.** Wrapping would noise-up every literal (`Coordinate(0)` everywhere) for marginal type-safety benefit at v0.1's scale. Revisit only if axis/AP/hull confusion shows up in practice.
- **`slots=True` on `GridPosition`.** pydantic v2 handles slotted dataclasses cleanly; the persistence boundary is its own concern, not a reason to drop the per-instance memory win.
- **`neighbors()` is a method, not a free function.** `tcod`'s graph adapter wants `(GridPosition) -> Iterable[GridPosition]`; `obj.neighbors` satisfies it without an extra layer.
- **No `_next_entity_id()` test reset hook.** Tests assert relative ordering (`id_2 > id_1`), not absolute values вЂ” there is no need to expose a knob production code must never call.
