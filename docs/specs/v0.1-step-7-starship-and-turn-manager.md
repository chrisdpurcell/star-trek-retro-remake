## 1. Purpose

Two modules that close the v0.1 model layer's **action surface** — the player entity plus the turn orchestrator:

- `model/entities/starship.py` — `Starship(GameObject)`: kwarg-only constructor with `kind` discriminator, `ship_class`, `hull`, `ap_max`, `ap_remaining`. Implements all three v0.1 action methods (`move_to`, `dock_at`, `restore_ap`) plus the internal atomic check-then-debit `_debit_ap(cost)` helper named in umbrella §5.4.
- `model/combat/turn_manager.py` — `TurnManager(player_id, *, current_turn=1)`: turn-counter + orchestrator. `advance_turn(sector_map)` resolves the player, restores AP, increments the turn, emits `turn_advanced` per umbrella §5.6.3. Owns no AP, no NPC roster, no initiative.

They land together because (a) `combat/turn_manager.py` runtime-imports `entities.starship.Starship` to do `isinstance(player, Starship)` before `restore_ap()` (umbrella §6) — `TurnManager` literally cannot ship without `Starship`. (b) Both implement umbrella §5.6 action contracts which were under-specified at step-6 time; bundling them surfaces ordering and error-type decisions a per-module spec would miss. (c) Together they provide the **model-layer prerequisites** for `DESIGN.md` §10.1 DoD items #5, #6, #7. The DoD items themselves require view/controller wire-up (rendering, click-to-move handlers, Dock button state, End Turn button, comm log) which is out of scope for step 7 — that lands in step 8+ when `controller/model_bridge.py`, the action handlers, and the QGraphicsScene integration arrive. Step 7's completion advances the v0.1 model-side surface; the user-visible DoD remains gated on the later UI integration.

This spec also surfaces and amends five inconsistencies in `v0.1-model-layer.md` (the umbrella spec) — including one **blocker** found via research at [`docs/research/2026-05-25-v0.1-step-7-spec-assumptions.md`](../research/2026-05-25-v0.1-step-7-spec-assumptions.md): umbrella §6's DAG table lists `entities.station` as `TYPE_CHECKING`-only for `entities.starship`, but `dock_at`'s `isinstance(target, Station)` is a runtime expression — `Station` must be a runtime import. Full amendment list in §10.

## 3. Layering and constraints

- Both modules inherit ADR-0003 ([`docs/adr/0003-model-qt-free.md`](../adr/0003-model-qt-free.md)); the existing `model-is-qt-free` `import-linter` contract covers them statically. The step-5 `tests/unit/model/test_no_qt_imports.py` subprocess-isolated runtime guard rediscovers them via dynamic walk (no test changes required).
- `from __future__ import annotations` at the top of both modules per `DESIGN.md` §10.6. mypy `--strict` per `pyproject.toml`'s `[[tool.mypy.overrides]]` for `stmrr.model.*`.
- **Imports** (after the umbrella §6 amendments described in §10):
  - `entities.starship` **runtime**: `entities.game_object`, `entities.station`, `world.grid_position`, `events`, `exceptions`. **`TYPE_CHECKING` only**: `world.sector_map` (action-method parameter annotation; the actual `sector_map.get(...)` / `sector_map.bounds_check(...)` calls are duck-typed at runtime).
  - `combat.turn_manager` **runtime**: `entities.starship`, `events`, `exceptions`. **`TYPE_CHECKING` only**: `world.sector_map` (parameter annotation), `entities.game_object` (the `EntityId` NewType is referenced only in annotations; `TurnManager` does not mint or unwrap IDs).
- **Two new runtime edges in the model DAG**: (a) `combat.turn_manager → entities.starship` for `isinstance(player, Starship)`; (b) `entities.starship → entities.station` for `isinstance(target, Station)`. Both are forward edges in the DAG. **No cycles introduced** — `entities.station` does not import `entities.starship` (step 6 §5.1 locked this via the inline `_Dockable: Protocol`), and `entities.starship` does not import `combat.turn_manager`.
- The `isinstance(target, Station)` nominal-typing choice is research-grounded per [`docs/research/2026-05-25-v0.1-step-7-spec-assumptions.md`](../research/2026-05-25-v0.1-step-7-spec-assumptions.md) §C: `@runtime_checkable` Protocol `isinstance` only verifies attribute *existence*, not method signatures, creating a false-safety trap. Step 6 deliberately used Protocol for `Station.accepts_dock(ship)` (one-direction structural typing); step 7's reverse direction (`Starship.dock_at` resolving a target) uses nominal `isinstance` because `accepts_dock` is a `Station`-specific method called after the check. The asymmetry is intentional.
- Module + class docstrings mandatory on public types. Method docstrings only where the *why* is non-obvious (action precondition ordering, `_debit_ap` atomicity, `advance_turn` mutation order).
- No new `.importlinter` contracts. The existing four contracts (`model-is-qt-free`, `view-does-not-import-model-events`, `projection-is-qt-free`, `blinker-only-in-events`) cover step-7 surface without additions.

### 4.1 Class declaration

from stmrr.model.entities.station import Station

class Starship(GameObject):

### 4.2 Constructor signature

```python
def __init__(
    self,
    *,
    position: GridPosition,
    ship_class: str,
    hull: int,
    ap_max: int = 5,
    ap_remaining: int | None = None,
) -> None:

```

**Keyword-only `__init__`** (leading `*` after `self`): five parameters, four of which are scalars whose positional order is not obvious at the call site. `Starship(pos, "constitution", 100, 5, 5)` reads worse than `Starship(position=pos, ship_class="constitution", hull=100)`. Same rationale as step-6 `Station.__init__` (step-6 §5.1).

**`ap_remaining: int | None = None`** — sentinel pattern. If the caller omits or passes `None`, `__init__` sets `self.ap_remaining = ap_max` after validation. v0.1 callers always omit (turn 1 starts with a full pool per umbrella §4 "defaults to `ap_max` at construction so turn 1 starts with a full pool without first calling End Turn"). v0.2 save/load passes the persisted value explicitly (`Starship(..., ap_remaining=3)`) without API break.

**`ship_class: str` is open**, not a `Literal["constitution", "miranda", ...]`. Research finding ([open-questions §Q-F → cross-ref spec-assumptions §F]) corroborated by step-6's wide-Literal/narrow-runtime pattern: `Literal` is appropriate for closed enumerations (`Station.station_type`), but fights open-registry fields like `ship_class` where the set is extensible by mods, v0.2 ship-class registry, future ship classes. v0.1 validates only non-emptiness; the wide-Literal/narrow-runtime pattern is deliberately *not* used here.

### 4.3 Validation pipeline + EntityId-non-consumption invariant

All five validation checks run **before** `super().__init__(position)` — preserves step 3 §5.1's contiguous-monotone `_NEXT_ENTITY_ID` property (step-6 Station invariant 12 + Codex pass-1 SA-002). The full constructor body:

```python
def __init__(
    self,
    *,
    position: GridPosition,
    ship_class: str,
    hull: int,
    ap_max: int = 5,
    ap_remaining: int | None = None,
) -> None:
    # All validation BEFORE super().__init__() so failed construction does
    # NOT consume an EntityId (mirrors step-6 Station invariant 12).
    if not isinstance(ship_class, str):
        raise TypeError(
            f"ship_class must be str, got {type(ship_class).__name__}"
        )
    if len(ship_class) == 0:
        raise ValueError("ship_class must be non-empty")

    if not isinstance(hull, int) or isinstance(hull, bool):
        raise TypeError(f"hull must be int, got {type(hull).__name__}")
    if hull < 1:
        raise ValueError(f"hull must be >= 1, got {hull}")

    if not isinstance(ap_max, int) or isinstance(ap_max, bool):
        raise TypeError(f"ap_max must be int, got {type(ap_max).__name__}")
    if ap_max < 1:
        raise ValueError(f"ap_max must be >= 1, got {ap_max}")

    if ap_remaining is not None:
        if not isinstance(ap_remaining, int) or isinstance(ap_remaining, bool):
            raise TypeError(
                f"ap_remaining must be int or None, got {type(ap_remaining).__name__}"
            )
        if not (0 <= ap_remaining <= ap_max):
            raise ValueError(
                f"ap_remaining must be in [0, {ap_max}], got {ap_remaining}"
            )

    super().__init__(position)  # ID allocation happens here
    self.ship_class: str = ship_class
    self.hull: int = hull
    self.ap_max: int = ap_max
    self.ap_remaining: int = ap_max if ap_remaining is None else ap_remaining
```

| Field | Type check | Range check | Type-error | Range-error |
|-------|-----------|-------------|------------|-------------|
| `ship_class` | `isinstance(ship_class, str)` (`str` excludes `bytes` natively) | `len(ship_class) > 0` | `TypeError(f"ship_class must be str, got {type(...).__name__}")` | `ValueError("ship_class must be non-empty")` |
| `hull` | non-bool `int` | `hull >= 1` | `TypeError(f"hull must be int, got {type(...).__name__}")` | `ValueError(f"hull must be >= 1, got {hull}")` |
| `ap_max` | non-bool `int` | `ap_max >= 1` | `TypeError(f"ap_max must be int, got {type(...).__name__}")` | `ValueError(f"ap_max must be >= 1, got {ap_max}")` |
| `ap_remaining` (when not `None`) | non-bool `int` | `0 <= ap_remaining <= ap_max` | `TypeError(f"ap_remaining must be int or None, got {type(...).__name__}")` | `ValueError(f"ap_remaining must be in [0, {ap_max}], got {ap_remaining}")` |

**Why `hull >= 1`** ([open-questions §H → spec-assumptions §H]): a `hull == 0` ship at construction is incoherent — `active=True` from `GameObject.__init__` but semantically dead. The `tcod` tutorial uses the same `max_hp >= 1` invariant. Combat (v0.2) handles damage-to-zero via a setter that flips `active=False`; that path is umbrella invariant 2-compatible.

**Why `ap_max >= 1`**: parallel reasoning — an `ap_max == 0` ship can never act, so accepting it at construction would let the wire-up layer spawn a dead ship that fails every action method silently.

**Why `ValueError`, not a model exception**: constructor validation = `ValueError`, matching `GridPosition.__post_init__` (step 3 §4.2) and `Station.__init__` (step 6 §5.2). No model exception class fits — `IllegalActionError` subclasses are for action preconditions on already-constructed entities; this is "the value you passed me to construct with is wrong."

### 4.4 `_debit_ap(cost)` helper — atomic check-then-debit

```python
def _debit_ap(self, cost: int) -> None:
    """Internal AP-debit primitive. Atomic check-then-debit.

    cost must be a non-bool int >= 1 (the action layer's narrowest
    sensible domain; cost <= 0 has no game meaning and cost=True would
    silently work via Python's bool-is-int subtyping — the same
    footgun mitigation step-6 SectorMap.__init__ applies to its axes).

    Raise BEFORE mutation: if cost > self.ap_remaining, raise
    InsufficientAPError and leave self.ap_remaining unchanged. Else
    decrement. v0.1 callers always pass cost=1; v0.2's variable-cost
    moves (Pythagorean diagonal per DESIGN.md §5.1) pass the computed
    cost — at which point an off-by-one or sign bug in the cost
    formula MUST surface as ValueError, not silently restore AP.

    Leading underscore: action methods (move_to, dock_at) are the only
    intended callers. The defensive cost validation costs ~4 lines
    and shipped now because the helper is the v0.2 variable-cost
    extension point (review finding SA-003).
    """
    # v0.2 TODO: if Starship instances are mutated from multiple
    # QThreadPool workers (AI move tasks), this read-modify-write becomes
    # a race. Two paths (see DESIGN.md §9.4):
    #   (a) Confine all model mutations to a single "model thread" and
    #       use Qt queued signal/slot for cross-thread handoff
    #       (lock-free, idiomatic Qt — preferred).
    #   (b) Wrap with threading.Lock()/QMutex around the
    #       check + decrement.
    # PEP 703 free-threading makes this real even without QThreadPool
    # once 3.14 free-threaded builds are adopted.
    if not isinstance(cost, int) or isinstance(cost, bool):
        raise TypeError(f"cost must be int, got {type(cost).__name__}")
    if cost < 1:
        raise ValueError(f"cost must be >= 1, got {cost}")

    if cost > self.ap_remaining:
        raise InsufficientAPError(required=cost, available=self.ap_remaining)
    self.ap_remaining -= cost
```

**Why raise-before-mutate** (research-grounded per [spec-assumptions §A](../research/2026-05-25-v0.1-step-7-spec-assumptions.md)): universal convention across `esper`, `tcod-ecs`, and the roguelike-development corpus. Debit-then-check-negative is broken and undocumented as valid anywhere.

**Why a helper, not inline check + decrement in each action method** (Q3 from brainstorming): two action methods (`move_to`, `dock_at`) would duplicate the check+decrement pattern. Single point of truth eliminates a "future caller forgets to check" footgun. v0.1 callers always pass `cost=1`; the parameter exists as the v0.2 variable-cost-move extension point.

**Why cost is validated as non-bool int `>= 1`** (review finding SA-003): the helper is the v0.2 variable-cost extension point. A future buggy cost computation that produces `0` (no-op), `-1` (silently restores AP via `ap_remaining -= -1`), `True` (silently 1 via bool-is-int), or `1.5` (TypeError on subtraction in v0.1 but undefined under PEP 604 union widening in v0.2) must surface as a deterministic exception at this seam, not silently corrupt AP. The cost validation mirrors step-6 SectorMap's axis-validation pattern (int/bool/range, same exception types).

### 4.5 `move_to(target, sector_map)` — implements umbrella §5.6.1

```python
def move_to(self, target: GridPosition, sector_map: SectorMap) -> None:
    """Move this ship one cell. Implements umbrella §5.6.1.

    Preconditions (checked in this order; first failure raises):
      1. self.active                     → InactiveEntityError
      2. sector_map.bounds_check(target) → OutOfBoundsError
      3. self.position.is_adjacent(target) → NotAdjacentError
      4. self.ap_remaining >= 1            → InsufficientAPError (via _debit_ap)

    Effects (ordered):
      1. Capture from_position before mutation.
      2. _debit_ap(1) — may raise; if so, no mutation occurs.
      3. self.position = target.
      4. Emit ship_moved with payload carrying from_position + target.

    Zero-distance moves (target == self.position) raise NotAdjacentError
    because is_adjacent requires chebyshev_distance == 1 (chebyshev=0 is
    not adjacent). This is roguelike convention per research
    spec-assumptions §B.
    """
    if not self.active:
        raise InactiveEntityError(entity_id=self.id)
    if not sector_map.bounds_check(target):
        raise OutOfBoundsError(
            position=target,
            sector_dims=(sector_map.width, sector_map.height, sector_map.depth),
        )
    if not self.position.is_adjacent(target):
        raise NotAdjacentError(from_position=self.position, to_position=target)

    from_position = self.position
    self._debit_ap(1)  # may raise InsufficientAPError; no mutation if so
    self.position = target
    ship_moved.send(
        self,
        payload=ShipMovedPayload(
            ship_id=self.id,
            from_position=from_position,
            to_position=target,
        ),
    )
```

**Why `from_position` is captured before `_debit_ap`**: if `_debit_ap` raises, the local has no effect (Python doesn't leak names from a raising statement back to the caller frame's local namespace). Capturing earlier than strictly necessary also documents intent — `from_position` is the *pre-mutation* position, full stop.

**Why preconditions in this order** (umbrella §5.6.1 locked): `active` is the cheapest check (one boolean read) and the most fundamental ("can this entity act at all?"). `bounds_check` is the next-cheapest. `is_adjacent` is geometric. AP debit is *last* because (a) it's the only one that mutates state on success, and (b) failing earlier checks should never consume AP-checking work. The umbrella locks this order; the implementation follows it.

### 4.6 `dock_at(station_id, sector_map)` — implements umbrella §5.6.2

```python
def dock_at(self, station_id: EntityId, sector_map: SectorMap) -> None:
    """Dock at a station. Implements umbrella §5.6.2.

    Preconditions (checked in this order; first failure raises):
      1. self.active                                       → InactiveEntityError
      2. target resolvable AND isinstance(Station) AND active → NotDockableError (conflated)
      3. self.position.is_adjacent(target.position)        → NotAdjacentError
      4. target.accepts_dock(self) is True                 → NotDockableError
      5. self.ap_remaining >= 1                            → InsufficientAPError (via _debit_ap)

    Effects (ordered):
      1. _debit_ap(1) — may raise; if so, no mutation occurs.
      2. Emit docked with payload carrying ship_id + station_id.

    No state on the Station mutates (umbrella §5.6.2). Service application
    is mission-system-side in later versions; v0.1 only proves the action
    pipeline. The conflated NotDockableError (covering stale-ID, wrong-type,
    and inactive-station) is intentional per umbrella §5.6.2 — the
    controller does not distinguish the sub-failures in v0.1 UI messaging.
    """
    if not self.active:
        raise InactiveEntityError(entity_id=self.id)

    target = sector_map.get(station_id)
    if target is None or not isinstance(target, Station) or not target.active:
        raise NotDockableError(ship_id=self.id, station_id=station_id)

    if not self.position.is_adjacent(target.position):
        raise NotAdjacentError(from_position=self.position, to_position=target.position)

    if not target.accepts_dock(self):
        raise NotDockableError(ship_id=self.id, station_id=station_id)

    self._debit_ap(1)
    docked.send(self, payload=DockedPayload(ship_id=self.id, station_id=station_id))
```

**Why nominal `isinstance(target, Station)`, not a `_DockTarget: Protocol`** (research [spec-assumptions §C](../research/2026-05-25-v0.1-step-7-spec-assumptions.md)): `accepts_dock` is a `Station`-specific method called immediately after the check. `@runtime_checkable` Protocol `isinstance` checks attribute *existence* only, not method signatures — anything with an `accepts_dock` attribute (even a wrong-signature one) would pass. Nominal typing here documents the actual contract more honestly than structural typing. The Station-side asymmetry (`accepts_dock(ship: _Dockable)`) is correct in its direction because the predicate genuinely reads only `ship.active`.

**Why `target.accepts_dock(self)` is called** even though v0.1 reduces it to `ship.active` (which is already True at this point — precondition 1 verified `self.active`): umbrella §5.6.2 + step-6 §5.4 lock the predicate as the v0.2 extension point. Calling it now means the v0.2 faction/reputation widening is purely conjunctive on the Station side (`return ship.active and faction_ok(ship)`); no `dock_at` change is needed when v0.2 ships.

### 4.7 `restore_ap()`

```python
def restore_ap(self) -> None:
    """Reset ap_remaining to ap_max. Called by TurnManager.advance_turn
    on the player at the start of each turn (umbrella §5.6.3).

    Unconditional: there is no v0.1 mechanism for partial restoration or
    decay. Per DESIGN.md §3.1, unused AP does not carry over.
    """
    self.ap_remaining = self.ap_max
```

### 4.8 What's deliberately not on `Starship`

- **No `Faction` field, no shields/weapons/sensors/hull-damage handling.** Umbrella §4 + `DESIGN.md` §10.1 — v0.2 surface.
- **No `__eq__` override.** Inherits `GameObject.__eq__` (identity-based per step 3 §5.4). Two Starships at the same position with identical attrs are not equal — different ships.
- **No `combat.turn_manager` import.** The `TurnManager → Starship` runtime edge is one-directional per umbrella §6 (and the no-back-edge invariant 8 below). A back-edge would cycle.
- **No `Faction` validation against a registry.** Open-registry pattern per research [open-questions §F → spec-assumptions §F].
- **No `damage(amount)`, `heal(amount)`, or `set_hull(value)` methods.** v0.2 combat owns hull mutation.

### 5.2 Constructor + validation

```python
def __init__(self, player_id: EntityId, *, current_turn: int = 1) -> None:
    # EntityId is NewType("EntityId", int) — int at runtime.
    if not isinstance(player_id, int) or isinstance(player_id, bool):
        raise TypeError(f"player_id must be int, got {type(player_id).__name__}")
    if player_id < 1:
        raise ValueError(f"player_id must be >= 1, got {player_id}")

    if not isinstance(current_turn, int) or isinstance(current_turn, bool):
        raise TypeError(
            f"current_turn must be int, got {type(current_turn).__name__}"
        )
    if current_turn < 1:
        raise ValueError(f"current_turn must be >= 1, got {current_turn}")

    self.player_id: EntityId = player_id
    self.current_turn: int = current_turn
```

| Field | Type check | Range check | Why |
|-------|-----------|-------------|-----|
| `player_id` | non-bool `int` | `>= 1` | `_next_entity_id()` returns `1, 2, ...` so `0` is never a valid live ID. Catches programmer error (forgetting to assign an ID) at construction. |
| `current_turn` | non-bool `int` | `>= 1` | Turn-1 is the v0.1 start state; turn-0 has no semantic meaning. v0.2 save/load reconstructs at saved-turn (≥1). |

**Why no validate-before-super requirement** (research [open-questions §Q-E → spec-assumptions §E]): `TurnManager` is **not** a `GameObject` subclass — `object.__init__()` has no side effects, no monotone counter to leak. The step-6 Station invariant 12 (validate before `super().__init__()`) is specific to `GameObject` subclasses. Plain top-to-bottom validation is the conventional Python idiom here.

### 5.3 `advance_turn(sector_map)` — implements umbrella §5.6.3

```python
def advance_turn(self, sector_map: SectorMap) -> None:
    """End the current turn. Implements umbrella §5.6.3.

    Preconditions: none — End Turn is always available, including when
    the player ship is inactive (game-over state). The umbrella locks
    this: dead-ship End Turn restores AP (harmless — an inactive ship
    cannot consume AP via move_to/dock_at), increments the counter,
    and emits the event. The controller detects game-over via state
    transitions, not via advance_turn refusing to run.

    Effects (ordered, per umbrella invariant 9 — mutate → emit):
      1. Resolve player from sector_map.
      2. Verify the resolution succeeded AND the resolved entity is a
         Starship instance; raise InactiveEntityError if either fails.
         This is type-and-presence verification only — it does NOT
         check player.active. UNREACHABLE in v0.1 by construction (the
         player ship is added to SectorMap at session start and never
         removed); the check hardens against v0.2 persistence-layer
         foot-guns where a stale player_id could survive a save/load
         round-trip without its referent.
      3. player.restore_ap() — sets player.ap_remaining = player.ap_max.
         Runs even if player.active is False.
      4. self.current_turn += 1.
      5. Emit turn_advanced with payload carrying the NEW turn number.

    Subscribers reading manager.current_turn inside a turn_advanced
    handler see the new turn (umbrella invariant 3). See §9 risk note
    for the blinker synchronous-send exception-propagation behavior.
    """
    player = sector_map.get(self.player_id)
    if player is None or not isinstance(player, Starship):
        raise InactiveEntityError(entity_id=self.player_id)

    player.restore_ap()
    self.current_turn += 1
    turn_advanced.send(
        self,
        payload=TurnAdvancedPayload(turn_number=self.current_turn),
    )
```

**Why `InactiveEntityError` for the None / non-Starship case** (umbrella §5.6.3 locked; v0.2 split tracked as §9 open question 1): the semantic mismatch ("missing" or "wrong-type" framed as "inactive") is mild but real. The v0.1 collapse is acceptable because (a) both branches are unreachable by construction, (b) the controller's failure response is identical across the two sub-failures ("the player can't act this turn — save file likely corrupted"). The v0.2 split — if it happens — adds `MissingEntityError(IllegalActionError)` as a **peer** to `InactiveEntityError` under step 5's locked `IllegalActionError` root, **not** as a `LookupError`-rooted refactor (which would conflict with the locked `ModelError(Exception)` hierarchy). Note: an *inactive* Starship is NOT one of the collapsed sub-failures (it succeeds at End Turn per SA-001 lock in §5.3's docstring and §8 invariant 8); only None and non-Starship raise.

**Why effect order is `restore → increment → emit`** (umbrella invariant 9): subscribers reading `manager.current_turn` inside the `turn_advanced` handler must see the new value. Emitting before incrementing forces every subscriber to know "the payload's `turn_number` is the new turn, but `manager.current_turn` is the old turn until `send()` returns" — a footgun. Same rule the state-machine transition follows for `state_changed` (step 5 §4.5).

**Synchronous-send exception risk** (research [open-questions §Q2]): blinker's `send()` propagates a receiver's exception out and skips later receivers, while `current_turn` is already incremented. This is documented blinker behavior — *not* a bug. v0.1 has exactly one known-safe subscriber (controller bridge), so the bounded risk is acceptable. **Important factual correction**: `blinker.send_robust()` does **not** exist (PRs #3 and #27 in `pallets-eco/blinker` were proposed but never merged). The v0.2 mitigation is to implement a local `_send_robust(signal, sender, **kw)` utility that iterates `signal.receivers_for(sender)` with per-receiver try-except, appends `(receiver, value_or_exception)` tuples to a result list, and logs caught exceptions at `ERROR` level via `loguru`. Trigger: second non-bridge subscriber lands (e.g., replay log subscribes to `turn_advanced`). Tracked as v0.2 risk; see umbrella §9 amendment in §10.

### 5.4 What's deliberately not on `TurnManager`

- **No `debit_player_ap` method.** AP lives on the entity per umbrella §5.4. Action methods (`Starship.move_to`, `Starship.dock_at`) own AP mutation.
- **No NPC roster, no initiative.** Single-player v0.1; v0.2 generalizes per `DESIGN.md` §5.2 combat.
- **No event subscription.** `TurnManager` is a publisher only; it does not connect to any signal.
- **No try/except around `turn_advanced.send`.** v0.1 accepts blinker's synchronous-propagation contract because the subscriber set is one and known safe. v0.2 introduces the local `_send_robust` utility per §5.3.
- **No `tick()` / `time` field.** v0.1 turns are discrete; no real-time clock. `DESIGN.md` §3.1 locks discrete-turn semantics.

## 8. Invariants

Properties step 7 contributes that the rest of the model layer relies on; reviewer should probe these.

1. **`Starship.kind`** is the class-level string `"starship"` for every Starship instance — TOML save-format discriminator per umbrella §5.5.
2. **EntityId non-consumption on failed Starship construction**: `Starship.__init__` validates all arguments BEFORE calling `super().__init__(position)`. Failed construction (`ValueError`/`TypeError` on any field) does NOT consume an `EntityId` from `_next_entity_id()`'s monotone counter. Mirrors step-6 Station invariant 12.
3. **`_debit_ap` is atomic check-then-debit**: raises `InsufficientAPError` BEFORE mutation; `ap_remaining` is never decremented on a failed debit.
4. **`_debit_ap` cost domain is non-bool int `>= 1`**: any other input raises `TypeError`/`ValueError` BEFORE the check-then-debit body runs; `ap_remaining` is unchanged. Closes the v0.2-variable-cost extension point against silent AP-restoration via negative cost (SA-003 review finding).
5. **`Starship.ap_remaining ∈ [0, ap_max]`** after every action method returns (successful or failed). No action method can drive `ap_remaining` negative or above `ap_max`. Together with invariant 4, this means `_debit_ap` cannot widen the AP range under any call from any caller (including future v0.2 variable-cost callers with buggy cost formulas).
6. **`move_to` precondition order is locked**: active → bounds → adjacent → AP. First failure raises and no mutation occurs (no `ap_remaining` decrement, no `position` change). The order is enforced by the implementation, not just convention.
7. **`dock_at` precondition order is locked**: active → resolve-target-is-active-Station → adjacent → accepts_dock → AP. First failure raises and no mutation occurs. The Station's state never mutates during `dock_at` in v0.1.
8. **`advance_turn` effect order is locked**: resolve → restore → increment → emit (umbrella invariant 9). Subscribers reading `manager.current_turn` inside a `turn_advanced` handler see the new turn; subscribers reading `player.ap_remaining` see `ap_max`. Effect order holds for inactive players too (umbrella §5.6.3 "preconditions: none — always available"; SA-001 lock).
9. **`TurnManager → Starship` is the only new runtime DAG edge in step 7** (in addition to `Starship → Station`). No back-edge from `Starship` to `TurnManager`; the property is enforced by a regex-grep test in `test_starship.py` (no-back-edge row) and a positive-import assertion in `test_turn_manager.py` (import-structure row).
10. **`Starship` and `TurnManager` import nothing from `combat.combat_resolver` or `combat.ai_state_machine`** — those modules do not exist in v0.1 and any forward-reference (even `TYPE_CHECKING`) would break mypy `--strict` per step-6 invariant 11's general principle.
11. **`turn_advanced.send` is synchronous**, and any receiver exception propagates out while `current_turn` is already incremented. v0.1 has one known-safe subscriber so the bounded risk is accepted; v0.2 introduces a local `_send_robust` utility (NOT `blinker.send_robust`, which does not exist) when the subscriber set widens. See §9 risk + umbrella §10 amendment.

- **`InactiveEntityError` for None / non-Starship player in `advance_turn`** (umbrella §5.6.3 locked). Semantic stretch; v0.2 may add `MissingEntityError(IllegalActionError)` as a peer (NOT a `LookupError` refactor — see still-open 1).
- **`TurnManager` does not have validate-before-super** ([open-questions §E → spec-assumptions §E]). Not a `GameObject` subclass; `object.__init__` is side-effect-free. Plain top-to-bottom validation.
- **`v0.2 _send_robust` is a local utility, not `blinker.send_robust`** ([open-questions §Q2]). `blinker.send_robust` does not exist — PRs #3 and #27 in `pallets-eco/blinker` were never merged. Trigger: second non-bridge subscriber lands.
- **`kind: ClassVar[str]` (matching step 6), not `Final[str]`** ([open-questions §Q3]). `Final[str]` is research-recommended but inconsistent with step-6's locked pattern. See still-open 3.
- **v0.2 threading TODO comment in `_debit_ap`** ([open-questions §Q4]). PEP 703 free-threading + QThreadPool AI workers make AP-debit a v0.2 race condition. Comment in §4.4.
- **Test probe class shapes are implementation-defined**, not spec-locked ([open-questions §Q5]). The §7 obligations describe probe usage descriptively.

### 9.2 Still open (v0.2 concerns, NOT v0.1 blockers)

1. **`InactiveEntityError` vs. eventual `MissingEntityError(IllegalActionError)` split.** `TurnManager.advance_turn` collapses **two** sub-failures (player None / non-Starship) under `InactiveEntityError`. The exception name is a semantic stretch — these are "missing entity" or "wrong-type entity" conditions, not "inactive entity"; an *inactive* player ship is NOT a failure (it succeeds at End Turn per §5.3 SA-001 lock + invariant 8). The mismatch is mild and v0.1-acceptable because both collapsed sub-failures are unreachable by construction. **v0.2 path**: introduce `MissingEntityError(IllegalActionError)` as a **peer** to `InactiveEntityError` under step 5's locked `IllegalActionError` root — **NOT** a `LookupError`-rooted refactor (that would conflict with the umbrella's single-root `ModelError(Exception)` hierarchy). Trigger: bridge error-to-UI mapping needs to distinguish the two collapsed sub-failures for player-facing messaging (e.g., "no player ship in this sector" vs. "player_id points at a station, not a starship — save file corrupted"). Round-2 review (SA-001 partial) caught this paragraph still using stale three-sub-failure wording.

2. **`v0.2 _send_robust` utility design and trigger condition.** v0.1's plain `turn_advanced.send(...)` accepts blinker's documented exception-propagation contract. v0.2 must implement a local `_send_robust(signal, sender, **kw)` utility (since `blinker.send_robust` does not exist). The Django pattern: iterate `signal.receivers_for(sender)` with per-receiver `try/except Exception`, append `(receiver, value-or-exception)` tuples to a results list, log caught exceptions at `ERROR` level via `loguru`. **Trigger**: a second non-bridge subscriber connects to any model signal (e.g., replay log subscribes to `turn_advanced`, telemetry subscribes to `ship_moved`). The first cross-subscriber dispatch is when fault isolation matters.

3. **`kind: ClassVar[str]` → `kind: Final[str]` migration deferred** ([open-questions §Q3]). Research recommended `Final[str]` for stronger no-override enforcement (mypy blocks subclass override). Migrating just `Starship.kind` to `Final` while step-6 `Station.kind` stays `ClassVar[str]` is inconsistent. **Deferred trigger**: either (a) mypy bug #18235 is fixed (allowing `ClassVar[Final[str]]` nesting in non-dataclass code, useful for v0.2's likely `@dataclass` conversion), or (b) `Starship` and `Station` are converted to `@dataclass` (whichever comes first). Cross-cutting amendment migrates both classes together.

4. **Threading model selection for v0.2** ([open-questions §Q4]). Two options for v0.2 AP-race mitigation:
   - **(A) Single model-thread affinity (preferred)**: all entity state changes happen on one thread; AI workers compute decisions and post results back via Qt queued signals. Lock-free, idiomatic Qt.
   - **(B) Per-entity `threading.Lock` / `QMutex`**: wraps OS mutex; interoperable between Python threads and QThreads. Correct but adds contention.

   v0.1's `_debit_ap` TODO comment documents both paths. The choice is owned by the v0.2 combat spec.
