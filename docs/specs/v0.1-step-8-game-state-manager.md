## 1. Purpose

Two artifacts that close the v0.1 model layer's **controller-pivot surface** — the hand-rolled state machine plus its two concrete states:

- `model/state/game_state_manager.py` — `GameStateManager(initial_state: GameState)`: holds `current_state: GameState`; method `transition_to(target: GameState) -> None` runs the five-step `validate → exit → mutate → enter → emit` lifecycle per umbrella §5.3, raises `IllegalTransitionError` (already declared in `model/exceptions.py:76-82`) on disallowed transitions, emits `state_changed` (already declared in `model/events.py:76`) with `StateChangedPayload(from_state, to_state)` carrying class objects (`type[GameState]`), not instances.
- `model/state/states.py` — extends the existing ABC stub (step 5, HEAD `5420568`) with two concrete subclasses: `MainMenuState`, `SectorMapState`. Each declares `allowed_transitions: ClassVar[frozenset[type[GameState]]]` pointing at the *other* class. Mutual reference is resolved via post-class module-level patch (see §6.2).

They land together because (a) `state.game_state_manager` cannot be tested in isolation — its happy-path `transition_to` requires two concrete `GameState` subclasses with non-empty `allowed_transitions` sets that reference each other (a single concrete state with `allowed_transitions = frozenset()` can only test the rejection path); (b) the post-class patch idiom (§6.2) is a cross-module concern between the manager's validation contract and the states' attribute lifecycle — bundling them surfaces this in one review; (c) together they satisfy umbrella §2's "main-menu shell + sector load" DoD prerequisites (the controller bridge layer that drives the transitions lands at step 9+; step 8's completion is the model-layer surface, not the user-visible behavior).

This spec also identifies and amends three stale rows in `v0.1-model-layer.md`: two in §4 (the `state.game_state_manager` and `state.states` rows under-specify what step 8 ships) and one in §6 (the `state.game_state_manager` DAG row over-declares `state.states` as a runtime import — an artifact of an earlier design assuming `isinstance(target, GameState)` defensive check; the spec's §9.1 Q4 lock rejects that check). Full amendment list in §10.

## 4. `model/state/game_state_manager.py` specification

### 4.1 Class declaration

```python
from __future__ import annotations

from typing import TYPE_CHECKING

from stmrr.model.events import StateChangedPayload, state_changed
from stmrr.model.exceptions import IllegalTransitionError

if TYPE_CHECKING:
    from stmrr.model.state.states import GameState


class GameStateManager:
    """Hand-rolled state machine driving the v0.1 main-menu ↔ sector-map pivot.

    Per umbrella spec `docs/specs/v0.1-model-layer.md` §5.3, `transition_to`
    runs the five-step lifecycle `validate → exit → mutate → enter → emit`.
    `__init__` fires the initial state's `enter()` synchronously (no
    `state_changed` event for construction — see spec §9.1 Q2).

    World ownership rule (umbrella §5.8): instances of `SectorMap` and other
    runtime entities are NOT constructed in any `GameState.enter` body;
    `app.py` (or a wire-up session object) owns world construction.
    `GameStateManager` knows only about states.
    """

    def __init__(self, initial_state: GameState) -> None: ...
    @property
    def current_state(self) -> GameState: ...
    def transition_to(self, target: GameState) -> None: ...
```

### 4.2 `__init__(initial_state)` semantics

1. Store `initial_state` on a private attribute (`self._current_state`).
2. Call `initial_state.enter()` synchronously.
3. Do NOT emit `state_changed`. Lifecycle invariant: every observable `current_state` has been entered and not yet exited; construction is the only path that satisfies the invariant without an event.

**No defensive `isinstance(initial_state, GameState)` check** — the type annotation is the contract; mypy-strict at call sites enforces it. (Per §9.1 Q4, defensive `isinstance` adds runtime cost and a parallel exception type for a footgun the type system already catches.)

### 4.3 `current_state` property

Returns `self._current_state`. Read-only — no setter. The only mutation path is `transition_to`. This is enforced by convention (no `@current_state.setter`), not by `__setattr__` machinery; the underscore-prefixed `_current_state` signals "do not write" to readers.

### 4.4 `transition_to(target: GameState) -> None` semantics

Runs the five-step lifecycle per umbrella §5.3. The order is load-bearing; subscribers reading `manager.current_state` inside a `state_changed` handler MUST see the new state (umbrella invariant 9).

1. **Validate**: `if type(target) not in self._current_state.allowed_transitions: raise IllegalTransitionError(from_state=type(self._current_state), to_state=type(target))`. Note `type(target)` — the validation set holds class objects (`type[GameState]`), not instances. `IllegalTransitionError` is declared in `model/exceptions.py:76-82` (step 5) with the exact `from_state`/`to_state` kwarg signature this method uses.
2. **Exit**: `self._current_state.exit()`. The outgoing state is still bound to `self._current_state` at this point — `exit()` may read it via the manager if needed (though v0.1's `pass` bodies do not).
3. **Mutate**: `from_state_cls = type(self._current_state); self._current_state = target`. Capture `from_state_cls` BEFORE the assignment — needed for the payload in step 5.
4. **Enter**: `target.enter()`. The new state is now bound to `self._current_state`.
5. **Emit**: `state_changed.send(self, payload=StateChangedPayload(from_state=from_state_cls, to_state=type(target)))`. Sender is the manager instance (`self`); payload carries class objects.

**Idempotence is data-driven, not code-driven.** Self-transitions (target is the same class as `current_state`) raise `IllegalTransitionError` because neither `MainMenuState.allowed_transitions` nor `SectorMapState.allowed_transitions` includes its own class (§6.3). No explicit `if type(target) == type(current_state): ...` branch.

**Identity short-circuit is NOT performed.** `if self._current_state is target: return` is rejected (§9.1 Q6) — composition handles it: validation rejects with `IllegalTransitionError` because `type(target)` is not in the allowed set.

**Receiver exceptions during `state_changed.send`** propagate per blinker's synchronous-emit contract (umbrella §9 risk row). The state mutation has already committed before `send()` is called; a buggy receiver leaves the manager in a consistent state but raises out of the caller's `transition_to` call. v0.1 inherits step-7's deferral; mitigation (`_send_robust`) lands when a second non-bridge subscriber connects.

### 4.5 Worked example

```python
from stmrr.model.state.game_state_manager import GameStateManager
from stmrr.model.state.states import MainMenuState, SectorMapState

manager = GameStateManager(MainMenuState())  # enters MainMenu; no event
assert isinstance(manager.current_state, MainMenuState)

manager.transition_to(SectorMapState())  # exits Main, mutates, enters Sector, emits state_changed
assert isinstance(manager.current_state, SectorMapState)

# Disallowed transition raises:
from stmrr.model.exceptions import IllegalTransitionError
import pytest
with pytest.raises(IllegalTransitionError):
    manager.transition_to(SectorMapState())  # Sector → Sector not allowed
```

### 5.3 Post-class patch

Below both class definitions, at module level:

```python
# Resolve the mutual MainMenuState ↔ SectorMapState forward reference.
# Each class body declared `allowed_transitions = frozenset()` (satisfies
# the step-5 __init_subclass__ presence check); the assignments below
# replace those placeholders with the actual allowed sets now that both
# classes exist as names. Deleting either of these two lines silently
# reverts the allowed sets to empty — the smoke tests in
# tests/unit/model/state/test_states.py guard against that regression.
MainMenuState.allowed_transitions = frozenset({SectorMapState})
SectorMapState.allowed_transitions = frozenset({MainMenuState})
```

The two re-assignments are mypy-strict-compatible: `MainMenuState.allowed_transitions` is annotated `ClassVar[frozenset[type[GameState]]]`, and the assignment uses mypy's annotated-target inference — mypy honors the declared LHS type, so the attribute's revealed type after assignment is `frozenset[type[GameState]]` (not narrowed to `frozenset[type[SectorMapState]]`, which is what the bare `frozenset({SectorMapState})` literal would infer to in isolation). No `# type: ignore` needed. Empirically verified at spec-authoring time against the project's pinned mypy 1.20.2 with `--strict`: `Success: no issues found in 1 source file`; `reveal_type(MainMenuState.allowed_transitions)` reports `frozenset[type[GameState]]`. The pre-review verification report at [`docs/research/2026-05-26-v0.1-step-8-pre-review-research.md`](../research/2026-05-26-v0.1-step-8-pre-review-research.md) documents the reproduction.

**Verification check:** the step-5 `__init_subclass__` enforcement at `states.py:33-51` reads `'allowed_transitions' not in vars(cls)` at class-creation time. Because each concrete-class body explicitly assigns `allowed_transitions = frozenset()`, `vars(cls)` contains the key at `__init_subclass__` time — the check passes. The subsequent module-level re-assignment only updates the value; it does not re-trigger `__init_subclass__`. The existing `test_concrete_subclass_inheriting_allowed_transitions_from_parent_fails` test at `test_states.py:75-94` confirms this is the right reading: inheritance from a parent does NOT count, but a class-body declaration (even empty) does.

## 6. Cross-cutting design decisions

### 6.1 Why `pass` bodies for `enter`/`exit` in v0.1

Per umbrella §5.8, state lifecycle hooks are "for view/UI cueing, not constructors". The model layer does not do UI cueing — cueing happens via the `state_changed` event, which `controller/model_bridge.py` (step 9+) converts to a Qt signal. So `enter`/`exit` bodies on the model side may stay `pass` permanently for these two states.

Alternatives considered and rejected:

- **`loguru.logger.debug(...)`** — adds a runtime dep import to `state.states`, which is otherwise a pure-stdlib module (umbrella §6 leaf node). Marginal value; the `state_changed` event already gives observers a hook. **Defer**; if trace logging is wanted, add it once and globally in `GameStateManager.transition_to` rather than peppering every state class.
- **`raise NotImplementedError`** — these ARE the concrete classes; there is nothing further to delegate to. Wrong tool.
- **World construction (e.g., `self.sector_map = SectorMap(8, 8, 10)`)** — forbidden by umbrella §5.8. Spec must call this out as a non-negotiable.

### 6.2 Mutual-class-reference resolution (Q1)

Python has no forward-class declaration syntax. The two viable patterns are:

- **(A) Post-class module-level patch (chosen).** Empty `frozenset()` in each class body satisfies `__init_subclass__`; module-level assignments after both classes exist update to the real sets. Spec §5.3 above.
- **(B) `classmethod` resolving the set on each call.** Breaks the umbrella + step-5 contract (`allowed_transitions` is data, not a method) and the `__init_subclass__` `vars(cls)` check. Rejected.

Research per [`docs/research/2026-05-26-v0.1-step-8-open-questions.md`](../research/2026-05-26-v0.1-step-8-open-questions.md) §Q1.

### 6.3 Self-transition forbidden by composition (Q5)

`MainMenuState.allowed_transitions = frozenset({SectorMapState})` does NOT include `MainMenuState`. A caller doing `manager.transition_to(MainMenuState())` while `current_state` is already a `MainMenuState` instance hits the validation step's `type(target) not in current_state.allowed_transitions` branch and raises `IllegalTransitionError`. No special-case code in `transition_to`.

This is observable behavior — §7.2 includes a test asserting `MainMenu → MainMenu` raises. If a future PR adds `MainMenuState` to its own `allowed_transitions`, the test fails, forcing the author to justify the change.

### 6.4 Lifecycle invariant: every observable state is entered

`GameStateManager.__init__` calls `initial_state.enter()` synchronously. Lifecycle invariant: every state reachable as `manager.current_state` has had `enter()` called exactly once and has not yet had `exit()` called.

The asymmetry: `__init__` fires `enter()` but does NOT emit `state_changed`. `transition_to` fires BOTH. This is intentional — `state_changed` semantically means "a transition occurred", and construction is not a transition. Subscribers that need to react to the starting state read `manager.current_state` directly after constructing the manager (or after connecting to the signal, depending on the wire-up order).

A reviewer may want to "fix" this asymmetry by emitting `state_changed` in `__init__` with `from_state=None`. **Reject** — `StateChangedPayload.from_state` is typed `type[GameState]`, not `type[GameState] | None`; widening the type for one construction edge case ripples through every subscriber. The spec locks the asymmetry; tests in §7.1 + §7.3 pin it.

### 6.5 Lifecycle hook exception policy

v0.1 concrete states have `pass` bodies (§6.1), so the question of "what if `enter()` or `exit()` raises?" is moot for production code paths. But `GameStateManager` accepts any `GameState` subclass (§7.2 includes a probe-subclass test row), and a v0.2+ state with non-trivial hook bodies could raise. The spec locks the policy now so the v0.1 implementation cannot silently choose a different one.

**Policy: hook exceptions propagate; no rollback. Postconditions differ by hook.**

| Failing hook | `current_state` after raise | `state_changed` emitted? | `target.enter()` called? |
|---|---|---|---|
| `self._current_state.exit()` raises | Unchanged (still the outgoing instance) | No | No |
| `target.enter()` raises | The new `target` instance (mutation already committed) | No | (the raising call) |

**Rationale:**
- The five-step lifecycle `validate → exit → mutate → enter → emit` (§4.4) places `exit()` before mutation; if `exit()` raises, mutation hasn't happened, so leaving `current_state` pointing at the outgoing instance is consistent.
- `enter()` runs after mutation but before the event. If `enter()` raises, mutation has already committed; rolling back (`self._current_state = outgoing_state_capture`) would require a new local variable and a try/except — added complexity for a v0.1 scenario where the hook bodies are `pass`. The simpler "no rollback" policy is honest about the partial-transition reality: the caller sees an exception and the manager state matches what step 4 of the lifecycle just wrote.
- No `state_changed` emitted in either case — the event semantically means "a transition completed", which it did not.
- Receiver exceptions during `state_changed.send()` (step 5) are a separate concern handled by umbrella §9's synchronous-emit risk row; by the time `send()` runs, both hooks have completed successfully.

Comparison points: `transitions` (pytransitions) documents that exceptions raised by callbacks halt processing of further callbacks but do not roll back the state change itself; `python-statemachine` exposes a `catch_errors_as_events` flag for opt-in fault isolation. v0.1's policy aligns with `transitions`' default (propagate, no rollback) and explicitly defers any fault-isolation work to v0.2 (alongside the `_send_robust` deferral inherited from step 7).

Tested by two §7.2 rows (one per failing hook) that assert exception propagation, no event emission, and the documented `current_state` postcondition.

## 8. Invariants

Properties step 8 contributes that the rest of the model layer relies on; reviewer should probe these.

1. **`GameStateManager.current_state` is read-only at the public surface.** No `@current_state.setter`; the underscore-prefixed `_current_state` private attribute is the sole write site, and `transition_to` is the sole writer (with `__init__` performing the initial assignment).
2. **The five-step transition order `validate → exit → mutate → enter → emit` is locked.** `exit()` runs before `self._current_state = target`; `enter()` runs after the mutation but before `state_changed.send()`. Subscribers reading `manager.current_state` inside a `state_changed` handler see the post-mutation state (umbrella invariant 9; tested in §7.2).
3. **`from_state` in `StateChangedPayload` is captured BEFORE the mutation.** Captured into a local `from_state_cls` variable in step 3 of the lifecycle, before `self._current_state = target` runs. Tested via an inequality assertion on a MainMenu → SectorMap transition.
4. **`StateChangedPayload` field types are class objects (`type[GameState]`), not instances.** Subscribers compare by type, never by instance reference — keeps subscribers stateless. **Save/load alignment:** class objects are NOT directly representable in the TOML save schema (per ADR-0004, the writer is `tomli_w`, which serializes only primitives, lists, and dicts; stdlib `tomllib` is parse-only); persisting them is a v0.2 persistence-spec concern (per umbrella §5.5). The likely v0.2 mapping is fully-qualified-name strings (`"stmrr.model.state.states.MainMenuState"`) with a registry-driven reverse lookup on deserialization. v0.1 does not implement persistence; the design only ensures the payload shape is compatible with that future mapping. Instance-carrying alternatives (storing state instances in the payload) would force the persistence layer to deal with construction kwargs at load time — a harder problem.
5. **`__init__` does NOT emit `state_changed`.** Asymmetric with `transition_to` by design (§6.4). Lifecycle invariant: every observable `current_state` has had `enter()` called and not yet had `exit()` called; construction is the only path that satisfies the invariant without emitting an event.
6. **Self-transitions are forbidden by composition, not by code.** Neither `MainMenuState.allowed_transitions` nor `SectorMapState.allowed_transitions` includes its own class. `transition_to` has no `if type(target) is type(current_state): return` short-circuit; the validation step raises `IllegalTransitionError` naturally.
7. **Identity-equal targets are also forbidden by composition.** `manager.transition_to(manager.current_state)` raises `IllegalTransitionError` for the same reason as #6 — `type(current_state)` is not in its own `allowed_transitions`. No `if self._current_state is target: return` short-circuit.
8. **Step 5's `__init_subclass__` enforcement at `states.py:33-51` is untouched.** Step 8 only adds concrete subclasses below the ABC; the ABC body, the `__init_subclass__` method, and the abstract `enter`/`exit` declarations remain at their step-5 positions and behavior.
9. **`MainMenuState.allowed_transitions` and `SectorMapState.allowed_transitions` are non-empty `frozenset` instances at module import time.** The post-class patch (§5.3) is module-level; it runs at import. Smoke tests in §7.1 catch a regression where the patch lines are accidentally deleted.
10. **`state_changed.send` is synchronous** (inherited from umbrella §9 + step-7 invariant 11). Any receiver exception propagates out of `transition_to` while the state mutation is already committed. v0.1 accepts the bounded risk (one known-safe subscriber: the controller bridge); v0.2 mitigation is the `_send_robust` utility per step-7 §9.2.
11. **No back-edges from `state.*` to entities or combat.** `state.game_state_manager` runtime-imports only `events` and `exceptions`; `state.states` is imported under `TYPE_CHECKING` for annotations only. `state.states` runtime-imports nothing from `stmrr.model.*` (stdlib `abc`, `inspect`, `typing` only). Both rows in umbrella §6's DAG table accurately reflect this after the §10 amendments land.
12. **`MainMenuState` / `SectorMapState` do not declare `kind: ClassVar[str]`.** The `kind` discriminator is entities-only (`Starship.kind = "starship"`, `Station.kind = "station"`); states are not persisted as entities. Negative test in §7.1.
13. **Lifecycle hook exception policy is propagate-without-rollback** (§6.5). If `self._current_state.exit()` raises, no mutation occurs and `current_state` is unchanged. If `target.enter()` raises, mutation has already committed (`current_state` is the new instance) and no `state_changed` event fires. The asymmetry mirrors the five-step lifecycle's exit-before-mutate vs. enter-after-mutate placement. Tested by two §7.2 rows. Reviewed comparison: aligns with `transitions` (pytransitions) default; differs from `python-statemachine`'s optional `catch_errors_as_events` fault isolation, which is the same shape as the v0.2 `_send_robust` deferral inherited from step 7.

## 9. Open questions

### 9.1 Locked decisions

- **`transition_to(target: GameState)` takes an instance, not a class.** Umbrella §4 spec. Validation uses `type(target) in current_state.allowed_transitions` (set holds classes). Instance is required because `enter()` and `exit()` are instance methods.
- **`__init__(initial_state: GameState)` fires `enter()` synchronously; no `state_changed` event for construction.** Q2 lock. Lifecycle invariant: every observable state has been entered.
- **`MainMenuState` / `SectorMapState` `enter`/`exit` bodies are `pass`.** Q3 lock. v0.1 has no model-side cueing; future cueing flows through the `state_changed` event consumed by the controller bridge.
- **Mutual class reference resolved via post-class module-level patch.** Q1 lock. Class body declares `allowed_transitions = frozenset()` (satisfies `__init_subclass__`); module-level re-assignment after both classes exist updates the values. Tests pin the patched values.
- **No defensive `isinstance(target, GameState)` check in `transition_to`.** Q4 lock. mypy-strict at the boundary plus the validation step's `type(target) in ...` lookup handle non-`GameState` arguments cleanly.
- **No special-case self-transition logic; no identity short-circuit.** Q5 + Q6 locks. Composition (the allowed-transitions set) is the source of truth.
- **`from_state_cls` is captured BEFORE `self._current_state = target` in `transition_to`.** Invariant 3.
- **One runtime DAG edge removed, no additions.** Step 8 amends the umbrella §6 row for `state.game_state_manager` to move `state.states` from the runtime column to the TYPE_CHECKING column (the original runtime listing was an artifact of an assumed `isinstance(target, GameState)` check that the spec's Q4 lock rejects). No new edges introduced; `state.states` row unchanged (still a runtime leaf).
- **`state.states` remains a stdlib-only runtime leaf.** No `loguru` import; no `events` import; no `exceptions` import (states never raise model exceptions in v0.1 — they only have `pass` bodies). Confirmed by umbrella §6 row.
- **Tests use try/finally connect/disconnect** mirroring `test_starship.py` / `test_turn_manager.py`. No new pytest fixture introduced.
- **No `__repr__` override on `GameStateManager` or the concrete states for v0.1.** Default Python `<MainMenuState object at 0x...>` is sufficient for failure messages; `pytest` failure messages already include the class name. (See §9.2 still-open 1.)
- **Lifecycle hook exception policy: propagate without rollback** (§6.5). `exit()` failure → no mutation, no event. `enter()` failure → mutation already committed, no event. Documented and tested explicitly so v0.2's first non-trivial hook body can't be a surprise. The asymmetry between `exit` and `enter` postconditions mirrors the five-step lifecycle order, not an arbitrary choice.
- **Reentrancy from inside a `state_changed` receiver is NOT supported in v0.1.** A receiver calling `manager.transition_to(...)` would mutate `manager._current_state` while the previous `send()` is still on the stack. v0.1 has one known-safe subscriber (controller bridge) and no use case for reentrant transitions; the spec does not test it, does not document a policy, and treats it as undefined. v0.2 may revisit when the bridge or replay layer needs to chain transitions.

### 9.2 Still open (v0.2 concerns, NOT v0.1 blockers)

1. **`__repr__` for `GameStateManager`.** Default Python repr (`<GameStateManager object at 0x...>`) is uninformative in pytest failure messages. A `__repr__` returning `f"GameStateManager(current_state={type(self._current_state).__name__})"` is cheap and improves debugging. **Defer**: add when a debugging session feels the friction; not v0.1-blocking. Skip-trigger: spec author finds the default repr acceptable in step-9+ bridge debugging.

2. **Trace logging in `transition_to`.** A single `logger.debug(f"{type(from_state).__name__} → {type(to_state).__name__}")` line in `transition_to` would give state-machine trace visibility. **Defer**: requires importing `loguru` into `game_state_manager.py` (which currently has no logging imports) and creates a precedent for "should every action method log?". v0.2 may add a structured tracing layer (decoration-based, per umbrella invariant 6) that covers all model-layer transitions and action methods uniformly; better to wait for that decision than to scatter ad-hoc `logger.debug` calls.

3. **State count growth past ~7 nodes.** ADR-0005 names ~15 states as the threshold for revisiting the hand-rolled approach in favor of `transitions` or `python-statemachine`. v0.1 has 2 states; v0.2 combat likely adds 2-3 (`CombatState`, optionally `PausedState`, `LoadingState`); well below the threshold. No action.

4. **State-instance reuse vs. fresh construction.** `manager.transition_to(SectorMapState())` constructs a new instance on every transition. Reusing a single instance per state class (singleton-like) would save allocations and let states hold mutable inter-transition state. **Reject for v0.1**: state instances in v0.1 have zero state (empty `pass` bodies), so the optimization buys nothing and adds lifecycle complexity. v0.2 may revisit if a state needs to hold transition-spanning data (e.g., `CombatState` retaining initiative order across pause/resume cycles).
