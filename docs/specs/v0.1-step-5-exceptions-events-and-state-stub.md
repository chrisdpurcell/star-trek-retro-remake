## 1. Purpose

Three modules that anchor cross-cutting concerns for the rest of the v0.1 model layer:

- `model/exceptions.py` — single-file exception hierarchy used by every action method and the state-machine transition guard.
- `model/events.py` — the `blinker.Namespace`, four typed payload dataclasses, and four named signals that are the model layer's only sanctioned change-notification channel.
- `model/state/states.py` — the **minimal `GameState` ABC stub**. Concrete states (`MainMenuState`, `SectorMapState`) defer to the state-machine step where they are designed alongside their consumer (`GameStateManager`).

All three are runtime leaves per [`v0.1-model-layer.md`](v0.1-model-layer.md) §6. They land together because (a) `events.py` field annotations forward-reference `GameState`, so the ABC must exist for mypy `--strict` to resolve `type[GameState]`; (b) the exception hierarchy is what the next steps' action methods will raise; (c) all three are pure-Python no-deps, so reviewer cost is dominated by the design content, not module count.

This spec also adds one cross-cutting test, `tests/unit/model/test_no_qt_imports.py`, which closes the deferral noted in step 3 (umbrella §7 invariant 1) now that the model layer crosses three runtime modules.

- All three modules inherit ADR-0003 ([`docs/adr/0003-model-qt-free.md`](../adr/0003-model-qt-free.md)); the existing `model-is-qt-free` `import-linter` contract covers them statically. - Imports: stdlib, `blinker`, and other model modules only. Forward-referenced types in payload + exception annotations are imported under `if TYPE_CHECKING:` only — neither module makes a runtime import edge to the modules it references for typing.
- Module + class docstrings mandatory on public types. Method docstrings only where the *why* is non-obvious.
- Per umbrella §7 invariant 10: `events.py` is the **single grep target** for every signal and every payload dataclass in the model layer. No module-local signals; no payload classes outside `events.py`.

## 4. `model/exceptions.py` specification

### 4.1 Hierarchy

```
ModelError(Exception)                              # root; concrete but never raised directly by convention
├── IllegalActionError(ModelError)                 # action-precondition failures; concrete catch-all
│   ├── InsufficientAPError       (*, required: int, available: int)
│   ├── OutOfBoundsError          (*, position: GridPosition, sector_dims: tuple[int, int, int])
│   ├── NotAdjacentError          (*, from_position: GridPosition, to_position: GridPosition)
│   ├── InactiveEntityError       (*, entity_id: EntityId)
│   └── NotDockableError          (*, ship_id: EntityId, station_id: EntityId)
└── IllegalTransitionError(ModelError)
                                  (*, from_state: type[GameState], to_state: type[GameState])
```

`ModelError` and `IllegalActionError` are **concrete, not abstract**. ABC machinery for an exception class is gratuitous; the convention "always raise a subclass" is enforced by code review, not by the type system. The only intended use of the parent types is `except ModelError:` for layer-wide catches and `except IllegalActionError:` for action-precondition catch-all. **All exception classes inherit from `Exception` (through `ModelError`), not from `ValueError`** — callers use `except ModelError:` to distinguish model-layer failures from generic value errors raised by, e.g., `GridPosition.__post_init__` (which raises `ValueError` per spec A §4.2).

### 4.2 Module shape

```python
from __future__ import annotations

from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from stmrr.model.entities.game_object import EntityId
    from stmrr.model.state.states import GameState
    from stmrr.model.world.grid_position import GridPosition


class ModelError(Exception):
    """Root of the model-layer exception hierarchy. Never raised directly."""


class IllegalActionError(ModelError):
    """Raised when an action's preconditions fail. See umbrella spec §5.6."""


class InsufficientAPError(IllegalActionError):
    def __init__(self, *, required: int, available: int) -> None:
        self.required: int = required
        self.available: int = available
        super().__init__(
            f"action requires {required} AP, only {available} available"
        )


# ... remaining subclasses follow the same shape; signature contract in §4.3, locked __str__ formats in §4.4.

__all__ = [
    "ModelError",
    "IllegalActionError",
    "InsufficientAPError",
    "OutOfBoundsError",
    "NotAdjacentError",
    "InactiveEntityError",
    "NotDockableError",
    "IllegalTransitionError",
]
```

`exceptions.py` is a **runtime leaf**: the `EntityId`, `GridPosition`, and `GameState` imports live under `if TYPE_CHECKING:`, so PEP 563-stringified annotations on `__init__` parameters do not force a runtime import edge. This preserves the no-cycles property of umbrella §6's runtime DAG.

### 4.3 `__init__` signature contract

Every concrete subclass uses **keyword-only arguments** — the parameter list opens with `*` so positional construction raises `TypeError`. Two purposes:

1. **Self-documenting call sites.** `InsufficientAPError(required=2, available=0)` reads at the call site; `InsufficientAPError(2, 0)` would not.
2. **Forward-compatibility.** v0.2 may add fields (e.g., `entity_id` to `InsufficientAPError`) without breaking positional-arg callers — because none can exist by construction.

Each subclass:

1. Stores every kwarg as a public typed attribute on `self` (e.g., `self.required: int = required`).
2. Computes a formatted message inline (no helper functions, no template module).
3. Calls `super().__init__(formatted_message)` so `str(exc)` and `repr(exc)` both render meaningfully without a custom `__str__`.

`ModelError` and `IllegalActionError` themselves accept the default `Exception` signature (`*args`); they are catch-all parents and have no v0.1 callers raising them directly.

### 4.4 Locked `__str__` format strings

These are tested verbatim. Localization (v2.0+) will replace them with translation keys; v0.1 ships English literals so the bridge can render them as-is.

| Exception | Format string |
|---|---|
| `InsufficientAPError` | `f"action requires {required} AP, only {available} available"` |
| `OutOfBoundsError` | `f"position {position} is outside sector bounds {sector_dims}"` |
| `NotAdjacentError` | `f"position {to_position} is not adjacent to {from_position}"` |
| `InactiveEntityError` | `f"entity {entity_id} is inactive"` |
| `NotDockableError` | `f"ship {ship_id} cannot dock at station {station_id}"` |
| `IllegalTransitionError` | `f"transition not allowed: {from_state.__name__} → {to_state.__name__}"` |

`OutOfBoundsError` carries `sector_dims` as an attribute even though the bridge can re-derive it from `SectorMap`. The redundancy makes the exception self-contained for log emission and removes a `SectorMap` reference dependency from the error-handling path. Net win.

`IllegalTransitionError`'s message uses `__name__` rather than `__qualname__` — concrete states live at `stmrr.model.state.states.MainMenuState` and the leading prefix would noise up player-facing messages. `__name__` matches umbrella §5.3's "with both states named."

## 5. `model/events.py` specification

The umbrella spec §5.2 (lines 86-138) is the **canonical code listing**. This step-5 spec authors the file exactly as shown there, plus the additions in §5.1-§5.3 below. Any change between this spec and the umbrella's listing means the umbrella spec is being amended in the same PR — see §10 of this spec.

### 5.1 `__all__`

`events.py` exports the named signals and the payload dataclasses; the `stmrr_events` `Namespace` instance is **module-private** (no `__all__` entry, no leading-underscore name). Subscribers import named signals (`from stmrr.model.events import ship_moved`); the namespace is an implementation detail of how blinker scopes signals.

```python
__all__ = [
    "ShipMovedPayload",
    "DockedPayload",
    "TurnAdvancedPayload",
    "StateChangedPayload",
    "ship_moved",
    "docked",
    "turn_advanced",
    "state_changed",
]
```

### 5.2 PEP 563 footgun — locked invariant via test

The umbrella spec's §5.2 last paragraph calls out that `typing.get_type_hints()` on a payload class will raise `NameError` because the field-type imports (`EntityId`, `GridPosition`, `GameState`) are `TYPE_CHECKING`-only and absent from `events.py`'s runtime globals. **This is a feature, not a bug** — promoting those imports to runtime would create a cycle, which is why the leaf property exists.

The step-5 test obligation `test_get_type_hints_on_payload_raises_NameError` (in §7.3) locks this invariant. If a future contributor "fixes" the `NameError` by promoting the TYPE_CHECKING imports to runtime, the test forces them through code review, where the cycle implication can be flagged.

### 5.3 Re-exports under `model/__init__.py`

`model/__init__.py` already exists from earlier steps. Step 5 does **not** add re-exports for `exceptions.py` or `events.py` symbols at the package level. Callers import the leaf module directly (`from stmrr.model.events import ship_moved`, `from stmrr.model.exceptions import InsufficientAPError`). Rationale:

- Keeps the dependency edge visible at the call site (grep tells you who depends on `events`).
- Avoids an `__init__.py`-mediated re-export layer that has no consumer in v0.1.
- Matches step 3's choice (`grid_position`, `game_object` are re-exported under their `world/` and `entities/` subpackage `__init__.py`, but not under `model/__init__.py`).

## 6. `model/state/states.py` specification — minimal ABC stub

### 6.1 Class declaration

```python
from __future__ import annotations

from abc import ABC, abstractmethod
from typing import Any, ClassVar


class GameState(ABC):
    """Base class for all game states. Per ADR-0005 — hand-rolled state
    machine, no `transitions` library, no `QStateMachine`.

    Subclasses MUST declare `allowed_transitions` as a class-level
    `frozenset[type[GameState]]`. Omission is caught at class-definition
    time by `__init_subclass__` (see §6.2), not at instantiation.
    """

    allowed_transitions: ClassVar[frozenset[type["GameState"]]]

    def __init_subclass__(cls, **kwargs: Any) -> None:
        super().__init_subclass__(**kwargs)
        # Skip still-abstract intermediates: a subclass that inherits
        # @abstractmethods without overriding them is itself abstract,
        # and its concrete descendant will be checked at its own
        # __init_subclass__ time.
        if cls.__abstractmethods__:
            return
        if "allowed_transitions" not in vars(cls):
            raise TypeError(
                f"{cls.__name__} must declare class attribute "
                f"`allowed_transitions: ClassVar[frozenset[type[GameState]]]`"
            )

    @abstractmethod
    def enter(self) -> None: ...

    @abstractmethod
    def exit(self) -> None: ...


__all__ = ["GameState"]
```

### 6.2 Why `__init_subclass__`, not `@property @abstractmethod`, not a default

The `allowed_transitions` attribute is **data**, not behavior — it's a class-level lookup table consumed by `GameStateManager.transition_to`. Three enforcement options were considered:

1. **`ClassVar` annotation alone (no enforcement).** Rejected: the annotation is inherited but the value is not, so a subclass can omit the declaration and crash with `AttributeError` at first read. mypy `--strict` does **not** enforce subclass override of inherited class attributes — it only enforces that *provided* values match the type.
2. **`@property @abstractmethod`.** Rejected: turns a static lookup table into a per-instance method call. Every concrete state would override a property instead of declaring a class attr, which is more ceremony for no static-typing win (the dispatch is per-instance, but the data is per-class).
3. **`__init_subclass__` runtime check (chosen).** Fires at class-definition (i.e., import) time. Skips still-abstract intermediates via `if cls.__abstractmethods__: return`. Uses `"allowed_transitions" in vars(cls)` (not `hasattr`) so subclasses cannot satisfy the check by inheriting a sibling's value. Six lines, run-once cost, well-trodden pattern. The spec's test row pins the failure mode: subclass omission raises `TypeError` at class definition, not at instantiation.

A default of `frozenset()` (terminal-state semantics) was the simplest option, but it loses the defensive value: a concrete state that *forgets* to declare its transitions silently behaves as terminal. Forcing opt-in via `__init_subclass__` catches the omission.

### 6.3 Concrete subclass deferral

`MainMenuState` and `SectorMapState` are listed in umbrella §4 as v0.1 concretes. They land in the **state-machine step** alongside `model/state/game_state_manager.py`, where their `allowed_transitions` set is reviewed against gsm's transition contract in the same review. Designing them in step 5 — without their actual consumer (gsm) in view — would violate the design-with-consumer-in-context heuristic that ruled the maximal "pull all of states.py forward" option out at brainstorming time.

For step-5 test coverage of `StateChangedPayload`, tests use a `class _ProbeState(GameState)` pattern — same isolation trick as step 3's `_Probe(GameObject)` for the `GameObject.__repr__` test. See §7.4.

### 7.1 `test_no_qt_imports.py` — model-layer-wide runtime guard

Single test, dynamic walk. Closes umbrella §7 invariant 1's runtime obligation (deferred at step 3 pending ≥3 model modules; step 5 puts the layer at five — `grid_position`, `game_object`, `exceptions`, `events`, `state.states`).

```python
"""Runtime guard: no `stmrr.model.*` module pulls in PySide6 or shiboken6.

Complements the static `model-is-qt-free` import-linter contract by
catching transient imports inside function bodies that the static
analyser misses (umbrella §7 invariant 1).

Walks the `stmrr.model` package dynamically so the test never goes stale:
adding a new model module requires no edit here.
"""
```

| Test | Form |
|---|---|
| Walk every submodule under `stmrr.model.*` via `pkgutil.walk_packages(stmrr.model.__path__, "stmrr.model.")`, import each via `importlib.import_module`, then assert `"PySide6" not in sys.modules` and `"shiboken6" not in sys.modules`. | inline |

**Implementation note:** the test imports `stmrr.model` once at module scope to anchor `__path__`, then runs the walk inside the test function. `sys.modules` snapshot is taken **after** all imports complete, so transient imports inside imported modules' top-level code are caught. (Function-body imports inside *unexercised* code paths are still missed at this layer; controller-layer integration tests cover the action paths.)

**Stale-list footgun avoidance:** the test uses dynamic walk, not an explicit `import stmrr.model.X` per line. If a future Qt-importing module is added, the walk discovers it and the assertion fires. An explicit list would have a stale-list failure mode: a forgotten line means the violation hides behind absence.

## 8. Invariants

Properties the rest of the model layer relies on; reviewer should probe these.

1. `model/exceptions.py` is a runtime leaf — its only runtime imports are stdlib. `EntityId`, `GridPosition`, `GameState` resolve under `TYPE_CHECKING` only.
2. `model/events.py` is a runtime leaf — its only runtime imports are `dataclasses` (stdlib) and `blinker`. The four payload-field-type names resolve under `TYPE_CHECKING` only.
3. `model/state/states.py` is a runtime leaf — only stdlib imports (`abc`, `typing`).
4. `events.py` is the **single grep target** for every signal and every payload dataclass in the model layer (umbrella §7 invariant 10). `grep -rn "blinker.signal\|Namespace()\|class.*Payload" src/stmrr/model/` returns lines from `events.py` only.
5. Every `IllegalActionError` subclass and `IllegalTransitionError` accepts kwargs only — `cls(positional_arg)` raises `TypeError`.
6. Every concrete `GameState` subclass declares its own `allowed_transitions` (not inherited). Omission is a class-definition-time error, not a runtime read-time error.
7. `typing.get_type_hints()` on any payload class raises `NameError` — locking the runtime-leaf property of `events.py` against accidental "fixes."
8. After step 5, `stmrr.model` package contains exactly 5 importable modules: `world.grid_position`, `entities.game_object`, `exceptions`, `events`, `state.states`. The `test_no_qt_imports.py` walk discovers all 5 and finds no Qt symbol in `sys.modules` after the walk completes.

## 9. Open questions for review

### 9.1 Locked decisions

- **Bundling — three modules in one spec, one PR.** Step 3's two-module precedent justifies the shape; the `events.py` ↔ `state.states` forward reference forces the bundling.
- **`GameState` ABC stub only — concretes deferred to gsm step.** Smaller spec, concretes designed against their consumer.
- **`__init_subclass__` enforcement of `allowed_transitions`.** mypy `--strict` does not enforce subclass override of inherited class attributes. Property-abstractmethod is more ceremony for a static lookup table. `__init_subclass__` is the run-once-at-import-time check that matches the data-as-attribute shape. `vars(cls)` check (not `hasattr`) ensures the subclass declared the attribute itself.
- **Hybrid exception attribute richness (umbrella §5.6 + this spec §4.3-§4.4).** Each subclass carries the v0.1-UX-relevant attrs as kwargs-only fields with locked `__str__` formats. Bare exceptions force string-parsing; rich-everything is YAGNI for v0.2 concerns.
- **Locked `__str__` formats.** v2.0 localization replaces them with translation keys; v0.1 ships English literals.
- **`stmrr_events` namespace internal.** Subscribers import named signals; the `Namespace` instance is an implementation detail.
- **`IllegalActionError` and `ModelError` concrete, not abstract.** ABC machinery for an exception class is gratuitous.
- **`NotDockableError` no reason discriminator.** v0.2 problem; current attrs cover v0.1 + v0.2 debugging.
- **`test_no_qt_imports.py` dynamic walk, not explicit list.** No stale-list failure mode.
