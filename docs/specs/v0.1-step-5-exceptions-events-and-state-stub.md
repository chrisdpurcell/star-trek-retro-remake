## 1. Purpose

Three modules that anchor cross-cutting concerns for the rest of the v0.1 model layer:

- `model/exceptions.py` — single-file exception hierarchy used by every action method and the state-machine transition guard.
- `model/events.py` — the `blinker.Namespace`, four typed payload dataclasses, and four named signals that are the model layer's only sanctioned change-notification channel.
- `model/state/states.py` — the **minimal `GameState` ABC stub**. Concrete states (`MainMenuState`, `SectorMapState`) defer to the state-machine step where they are designed alongside their consumer (`GameStateManager`).

All three are runtime leaves per [`v0.1-model-layer.md`](v0.1-model-layer.md) §6. They land together because (a) `events.py` field annotations forward-reference `GameState`, so the ABC must exist for mypy `--strict` to resolve `type[GameState]`; (b) the exception hierarchy is what the next steps' action methods will raise; (c) all three are pure-Python no-deps, so reviewer cost is dominated by the design content, not module count.

This spec also adds one cross-cutting test, `tests/unit/model/test_no_qt_imports.py`, which closes the deferral noted in step 3 (umbrella §7 invariant 1) now that the model layer crosses three runtime modules.

## 3. Layering and constraints

- All three modules inherit ADR-0003 ([`docs/adr/0003-model-qt-free.md`](../adr/0003-model-qt-free.md)); the existing `model-is-qt-free` `import-linter` contract covers them statically. The new `tests/unit/model/test_no_qt_imports.py` adds runtime coverage for transient imports the static contract misses.
- `from __future__ import annotations` at the top of all three modules per `DESIGN.md` §10.6. Under Python 3.14 annotations are deferred by default (PEP 649); the explicit `from __future__ import annotations` (PEP 563) makes the deferred-evaluation behaviour unambiguous and matches the rest of the codebase. mypy `--strict` per the per-module rule pinned in `pyproject.toml`.
- Imports: stdlib, `blinker`, and other model modules only. Forward-referenced types in payload + exception annotations are imported under `if TYPE_CHECKING:` only — neither module makes a runtime import edge to the modules it references for typing.
- Module + class docstrings mandatory on public types. Method docstrings only where the *why* is non-obvious.
- Per umbrella §7 invariant 10: `events.py` is the canonical declaration site for every signal and every payload dataclass in the model layer. Step 5 enforces this in two complementary ways:
  1. **`.importlinter` forbidden contract `blinker-only-in-events`** — forbids any module under `stmrr.model.*` other than `stmrr.model.events` from importing `blinker` directly. **Must** be configured with `allow_indirect_imports = True` so future legitimate paths like `entities.starship → events → blinker` are not flagged. The contract bounds *direct imports*, not transitive ones. Required snippet shape:
     ```ini
     [importlinter:contract:blinker-only-in-events]
     name = blinker only in events
     type = forbidden
     source_modules =
         stmrr.model
     forbidden_modules =
         blinker
     ignore_imports =
         stmrr.model.events -> blinker
     allow_indirect_imports = True
     ```
     This contract enforces *import locations*, not *class declaration locations*. It catches "did someone create a new signal in another module?" because `blinker.signal()` requires importing `blinker`. It does **not** catch "did someone declare a `*Payload` dataclass elsewhere?" — payload dataclasses are plain `@dataclass` decorations with no required `blinker` import.
  2. **Payload-location grep test** — `tests/unit/model/test_payload_location.py` walks `src/stmrr/model/**/*.py` (excluding `events.py`) and asserts no class definition matches the regex `^class \w*Payload\b`. Cheap (regex per file, no AST machinery needed for v0.1's narrow naming convention), catches the only failure mode the import-linter contract can't.
  Mirrors how step 4 enforced Qt-free for the projection module via a dedicated contract.

## 4. `model/exceptions.py` specification

### 4.1 Hierarchy

```
ModelError(Exception)                              # root; concrete but never raised directly by convention
├── IllegalActionError(ModelError)                 # action-precondition failures; concrete catch-all
│   ├── InsufficientAPError       (*, required: int, available: int)
│   ├── OutOfBoundsError          (*, position: GridPosition, sector_dims: tuple[int, int, int])
│   ├── NotAdjacentError          (*, from_position: GridPosition, to_position: GridPosition)
│   ├── InactiveEntityError       (*, entity_id: EntityId)
│   └── NotDockableError          (*, ship_id: EntityId, station_id: EntityId)
└── IllegalTransitionError(ModelError)
                                  (*, from_state: type[GameState], to_state: type[GameState])
```

`ModelError` and `IllegalActionError` are **concrete, not abstract**. ABC machinery for an exception class is gratuitous; the convention "always raise a subclass" is enforced by code review, not by the type system. The only intended use of the parent types is `except ModelError:` for layer-wide catches and `except IllegalActionError:` for action-precondition catch-all. **All exception classes inherit from `Exception` (through `ModelError`), not from `ValueError`** — callers use `except ModelError:` to distinguish model-layer failures from generic value errors raised by, e.g., `GridPosition.__post_init__` (which raises `ValueError` per spec A §4.2).

### 4.2 Module shape

```python
from __future__ import annotations

from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from stmrr.model.entities.game_object import EntityId
    from stmrr.model.state.states import GameState
    from stmrr.model.world.grid_position import GridPosition


class ModelError(Exception):
    """Root of the model-layer exception hierarchy. Never raised directly."""


class IllegalActionError(ModelError):
    """Raised when an action's preconditions fail. See umbrella spec §5.6."""


class InsufficientAPError(IllegalActionError):
    def __init__(self, *, required: int, available: int) -> None:
        self.required: int = required
        self.available: int = available
        super().__init__(
            f"action requires {required} AP, only {available} available"
        )


# ... remaining subclasses follow the same shape; signature contract in §4.3, locked __str__ formats in §4.4.

__all__ = [
    "ModelError",
    "IllegalActionError",
    "InsufficientAPError",
    "OutOfBoundsError",
    "NotAdjacentError",
    "InactiveEntityError",
    "NotDockableError",
    "IllegalTransitionError",
]
```

`exceptions.py` is a **runtime leaf**: the `EntityId`, `GridPosition`, and `GameState` imports live under `if TYPE_CHECKING:`, so deferred-evaluation annotations on `__init__` parameters (Python 3.14 default per PEP 649; equivalently under the explicit `from __future__ import annotations`) do not force a runtime import edge. This preserves the no-cycles property of umbrella §6's runtime DAG.

### 4.3 `__init__` signature contract

Every concrete subclass uses **keyword-only arguments** — the parameter list opens with `*` so positional construction raises `TypeError`. Two purposes:

1. **Self-documenting call sites.** `InsufficientAPError(required=2, available=0)` reads at the call site; `InsufficientAPError(2, 0)` would not.
2. **Forward-compatibility.** v0.2 may add fields (e.g., `entity_id` to `InsufficientAPError`) without breaking positional-arg callers — because none can exist by construction.

Each subclass:

1. Stores every kwarg as a public typed attribute on `self` (e.g., `self.required: int = required`).
2. Computes a formatted message inline (no helper functions, no template module).
3. Calls `super().__init__(formatted_message)` so `str(exc)` and `repr(exc)` both render meaningfully without a custom `__str__`.

`ModelError` and `IllegalActionError` themselves accept the default `Exception` signature (`*args`); they are catch-all parents and have no v0.1 callers raising them directly.

### 4.4 Locked `__str__` format strings

These are tested verbatim. Localization (v2.0+) will replace them with translation keys; v0.1 ships English literals so the bridge can render them as-is.

| Exception | Format string |
|---|---|
| `InsufficientAPError` | `f"action requires {required} AP, only {available} available"` |
| `OutOfBoundsError` | `f"position {position} is outside sector bounds {sector_dims}"` |
| `NotAdjacentError` | `f"position {to_position} is not adjacent to {from_position}"` |
| `InactiveEntityError` | `f"entity {entity_id} is inactive"` |
| `NotDockableError` | `f"ship {ship_id} cannot dock at station {station_id}"` |
| `IllegalTransitionError` | `f"transition not allowed: {from_state.__name__} → {to_state.__name__}"` |

`OutOfBoundsError` carries `sector_dims` as an attribute even though the bridge can re-derive it from `SectorMap`. The redundancy makes the exception self-contained for log emission and removes a `SectorMap` reference dependency from the error-handling path. Net win.

`IllegalTransitionError`'s message uses `__name__` rather than `__qualname__` — concrete states live at `stmrr.model.state.states.MainMenuState` and the leading prefix would noise up player-facing messages. `__name__` matches umbrella §5.3's "with both states named."

### 5.2 Deferred-annotation footgun — locked per-payload invariants via test

The umbrella spec's §5.2 last paragraph calls out that `typing.get_type_hints()` on a payload class with TYPE_CHECKING-only field types raises `NameError`, because the field-type imports (`EntityId`, `GridPosition`, `GameState`) are absent from `events.py`'s runtime globals. **This is a feature, not a bug** — promoting those imports to runtime would create a cycle, which is why the runtime-leaf property exists.

The expected behaviour is **per-payload**, not blanket — `TurnAdvancedPayload` only contains `turn_number: int` (a builtin always in scope), so `get_type_hints(TurnAdvancedPayload)` succeeds. Locked expectations:

| Payload | `get_type_hints()` outcome |
|---|---|
| `ShipMovedPayload` | raises `NameError` (refs `EntityId`, `GridPosition`) |
| `DockedPayload` | raises `NameError` (refs `EntityId`) |
| `StateChangedPayload` | raises `NameError` (refs `GameState`) |
| `TurnAdvancedPayload` | succeeds; returns `{'turn_number': int}` |

The step-5 test obligation in §7.3 asserts each row explicitly. If a future contributor "fixes" the three `NameError` cases by promoting the TYPE_CHECKING imports to runtime, the affected tests force them through code review, where the cycle implication can be flagged.

Note on Python 3.14: under PEP 649 (default) and PEP 563 (`from __future__ import annotations`, used here), dataclass field annotations are not eagerly evaluated, so the TYPE_CHECKING imports stay safely deferred at class-definition time. `typing.get_type_hints()` is the runtime-resolution path that observes the missing names. For unresolved-but-safe inspection, `annotationlib.get_annotations(cls, format=annotationlib.Format.STRING)` returns string annotations without resolution — relevant if a future caller needs introspection without paying the resolution cost.

### 5.3 Re-exports under `model/__init__.py`

`model/__init__.py` already exists from earlier steps. Step 5 does **not** add re-exports for `exceptions.py` or `events.py` symbols at the package level. Callers import the leaf module directly (`from stmrr.model.events import ship_moved`, `from stmrr.model.exceptions import InsufficientAPError`). Rationale:

- Keeps the dependency edge visible at the call site (grep tells you who depends on `events`).
- Avoids an `__init__.py`-mediated re-export layer that has no consumer in v0.1.
- Matches step 3's choice (`grid_position`, `game_object` are re-exported under their `world/` and `entities/` subpackage `__init__.py`, but not under `model/__init__.py`).

## 6. `model/state/states.py` specification — minimal ABC stub

### 6.1 Class declaration

```python
from __future__ import annotations

import inspect
from abc import ABC, abstractmethod
from typing import Any, ClassVar


class GameState(ABC):
    """Base class for all game states. Per ADR-0005 — hand-rolled state
    machine, no `transitions` library, no `QStateMachine`.

    Subclasses MUST declare `allowed_transitions` as a class-level
    `frozenset[type[GameState]]`. Omission is caught at class-definition
    time by `__init_subclass__` (see §6.2), not at instantiation.
    """

    allowed_transitions: ClassVar[frozenset[type["GameState"]]]

    def __init_subclass__(cls, **kwargs: Any) -> None:
        super().__init_subclass__(**kwargs)
        # Skip still-abstract intermediates: a subclass that inherits
        # @abstractmethods without overriding them is itself abstract,
        # and its concrete descendant will be checked at its own
        # __init_subclass__ time. Use inspect.isabstract(cls), NOT
        # cls.__abstractmethods__ — under Python 3.14, ABCMeta has not
        # yet populated __abstractmethods__ on the new class object at
        # the moment __init_subclass__ runs (it is set after type.__new__
        # returns). inspect.isabstract reads the underlying _abc_impl
        # bookkeeping that is populated during class creation.
        if inspect.isabstract(cls):
            return
        if "allowed_transitions" not in vars(cls):
            raise TypeError(
                f"{cls.__name__} must declare class attribute "
                f"`allowed_transitions: ClassVar[frozenset[type[GameState]]]`"
            )

    @abstractmethod
    def enter(self) -> None: ...

    @abstractmethod
    def exit(self) -> None: ...


__all__ = ["GameState"]
```

### 6.2 Why `__init_subclass__`, not `@property @abstractmethod`, not a default

The `allowed_transitions` attribute is **data**, not behavior — it's a class-level lookup table consumed by `GameStateManager.transition_to`. Three enforcement options were considered:

1. **`ClassVar` annotation alone (no enforcement).** Rejected: the annotation is inherited but the value is not, so a subclass can omit the declaration and crash with `AttributeError` at first read. mypy `--strict` does **not** enforce subclass override of inherited class attributes — it only enforces that *provided* values match the type.
2. **`@property @abstractmethod`.** Rejected: turns a static lookup table into a per-instance method call. Every concrete state would override a property instead of declaring a class attr, which is more ceremony for no static-typing win (the dispatch is per-instance, but the data is per-class).
3. **`__init_subclass__` runtime check (chosen).** Fires at class-definition (i.e., import) time. Skips still-abstract intermediates via `if inspect.isabstract(cls): return` — **not** `cls.__abstractmethods__`, which raises `AttributeError` inside `__init_subclass__` under Python 3.14 because `ABCMeta` populates `__abstractmethods__` after `type.__new__` returns (i.e., after `__init_subclass__` has already run). `inspect.isabstract` reads the C-level `_abc_impl` bookkeeping that is populated during class creation and is therefore observable at this hook. Uses `"allowed_transitions" in vars(cls)` (not `hasattr`) so subclasses cannot satisfy the check by inheriting a parent's value. Run-once cost, well-trodden pattern. The spec's test row pins the failure mode: subclass omission raises `TypeError` at class definition, not at instantiation.

A default of `frozenset()` (terminal-state semantics) was the simplest option, but it loses the defensive value: a concrete state that *forgets* to declare its transitions silently behaves as terminal. Forcing opt-in via `__init_subclass__` catches the omission.

### 6.3 Concrete subclass deferral

`MainMenuState` and `SectorMapState` are listed in umbrella §4 as v0.1 concretes. They land in the **state-machine step** alongside `model/state/game_state_manager.py`, where their `allowed_transitions` set is reviewed against gsm's transition contract in the same review. Designing them in step 5 — without their actual consumer (gsm) in view — would violate the design-with-consumer-in-context heuristic that ruled the maximal "pull all of states.py forward" option out at brainstorming time.

For step-5 test coverage of `StateChangedPayload`, tests use a `class _ProbeState(GameState)` pattern — same isolation trick as step 3's `_Probe(GameObject)` for the `GameObject.__repr__` test. See §7.4.

### 7.1 `test_no_qt_imports.py` — model-layer-wide runtime guard (subprocess-isolated)

Single test, dynamic walk, **executed in a clean Python subprocess**. Closes umbrella §7 invariant 1's runtime obligation (deferred at step 3 pending ≥3 model modules; step 5 puts the layer at five non-package implementation modules — `world.grid_position`, `entities.game_object`, `exceptions`, `events`, `state.states` — out of nine total entries `pkgutil.walk_packages` discovers, the rest being scaffold package markers).

```python
"""Runtime guard: no `stmrr.model.*` module pulls in PySide6 or shiboken6
at import time.

Complements the static `model-is-qt-free` import-linter contract by
catching imports written at module scope, class body scope, decorator
scope, or default-argument scope — positions the static analyser may
miss when an import is conditional on `if TYPE_CHECKING:` style guards
or pulled in via dynamic patterns. Does NOT catch imports inside
function bodies that the probe doesn't call; those are caught by
controller-layer integration tests once the action methods land
(umbrella §7 invariant 1, three-layer enforcement).

The walk runs in a clean Python subprocess — running it in the parent
pytest process would race with `pytest-qt` (or any other view-layer
test) loading PySide6 first, which would make the assertion fail for
reasons unrelated to model code.
"""
```

| Test | Form |
|---|---|
| Spawn `subprocess.run([sys.executable, "-c", PROBE_SCRIPT], check=True, capture_output=True)`, where `PROBE_SCRIPT` walks every submodule under `stmrr.model.*` via `pkgutil.walk_packages(stmrr.model.__path__, "stmrr.model.")`, imports each via `importlib.import_module`, then asserts `"PySide6" not in sys.modules` and `"shiboken6" not in sys.modules`. The test itself only checks the subprocess's exit code and surfaces stderr on failure. | inline |

**Why subprocess isolation (not `sys.modules` snapshot in-process):** `sys.modules` is an additive global. `pytest-qt` fixtures, view-layer integration tests, or any fixture that imports PySide6 anywhere earlier in the same pytest run leaves PySide6 in `sys.modules` for the rest of the process. An in-process `"PySide6" not in sys.modules` assertion would then fail for reasons unrelated to model code, and the failure mode is order-dependent (works in isolation, fails in the full suite). A snapshot-and-diff approach (capture `sys.modules` before model import, assert no Qt symbol *added*) is closer but still misses the case where Qt was already loaded but a model module would have re-imported it. Subprocess is the only clean isolation boundary; the ~50 ms startup cost is one test per CI run.

**Coverage scope — what this test catches and what it doesn't:** the probe imports each model module top-to-bottom, which executes module-scope code (top-level statements, decorators, class bodies, default-argument expressions). It catches Qt imports written at any of those positions. It does **NOT** catch `def some_method(self): import PySide6` patterns where the import lives inside a function body that the probe never calls — those imports happen only when the function runs. The static `import-linter` `model-is-qt-free` contract has the same blind spot (it parses imports but doesn't reason about call graphs). The third layer — controller-layer integration tests that actually call the model's action methods (`Starship.move_to`, `Starship.dock_at`, `TurnManager.advance_turn` once those modules land) — is what catches function-body Qt imports inside live code paths. For v0.1, the three-layer combination (static import-linter + import-time runtime guard + integration-test exercise) is sufficient; an AST-based scan that walks every function body for forbidden imports is the v0.2 escalation if a regression slips through.

**Stale-list footgun avoidance:** the probe uses dynamic walk, not an explicit `import stmrr.model.X` per line. If a future Qt-importing module is added, the walk discovers it and the assertion fires. An explicit list would have a stale-list failure mode: a forgotten line means the violation hides behind absence.

**Why the walk discovers more than 5 entries:** `pkgutil.walk_packages` returns both packages (directories with `__init__.py`) and modules (`.py` files). The model layer has scaffold package markers (`combat/`, `missions/`, `resources/`, `systems/`) for layout completeness; importing an empty `__init__.py` doesn't load Qt, so the test passes regardless of how many empty packages exist. The test asserts the *property* (no Qt in `sys.modules`), not the *count* of entries.

### 7.3 `test_events.py`

### 7.4 `test_states.py`

| Case | Form |
|---|---|
| `GameState` cannot be instantiated directly (`TypeError` from ABC) | inline |
| A concrete subclass with `enter`, `exit`, and `allowed_transitions = frozenset()` instantiates cleanly | inline |
| A concrete subclass omitting `allowed_transitions` fails at **class-definition time** with `TypeError` carrying the class name | inline (use `pytest.raises(TypeError)` around an inline `class _Bad(GameState): ...` definition) |
| An intermediate abstract subclass (overrides `enter` but not `exit`) does **not** trigger the `allowed_transitions` check — it remains abstract; the check defers to its concrete descendant | inline |
| A concrete subclass that omits `allowed_transitions` and would otherwise inherit it from a concrete parent class fails the `vars(cls)` check at class-definition time (each concrete state must declare its own transition set; inheriting silently is a footgun) | inline |
| `enter` and `exit` are abstract methods on the ABC | inline (`assertEqual(GameState.enter.__isabstractmethod__, True)` shape) |

The "intermediate abstract" test guards against a future v0.2 hierarchy like `class _AbstractCombatState(GameState): ...` (overrides part of the contract, still abstract) being forced to declare a placeholder `allowed_transitions = frozenset()` just to import. The check defers correctly to the concrete descendant.

The `_ProbeState(GameState)` pattern used for `StateChangedPayload` rendering in `test_exceptions.py` is the same shape as the "concrete subclass instantiates cleanly" test here.

### 7.5 `test_payload_location.py` — payload-class declaration-site guard

Companion to the `blinker-only-in-events` import-linter contract. The contract enforces *import locations* (no module other than `events.py` imports `blinker`); this test enforces *class declaration locations* (no module other than `events.py` declares a `*Payload` class). Together they cover umbrella §7 invariant 10 in full.

| Test | Form |
|---|---|
| Walk every `*.py` file under `src/stmrr/model/` (excluding `events.py` itself), read its source, and assert no line matches the regex `^class \w*Payload\b`. Test fails with the offending file path + line if found. | inline |

Implementation note: regex over file source is sufficient for v0.1's narrow naming convention (`*Payload` suffix). An AST walk would be more rigorous (catches `MyPayload = type(...)` dynamic creation, conditional class defs) but is overkill: the convention is "name your payload classes `*Payload` and put them in `events.py`," and a contributor who routes around a regex by dynamic class creation is willfully bypassing the contract — that case is out of scope for v0.1 enforcement.

**Structure:** AAA with whitespace between phases, no phase comments. Test names: `test_<unit>_<scenario>_<expected>`. Use `pytest.mark.parametrize` when ≥3 cases share Arrange/Act shape; prefer `ids=` for readable CI output. Same conventions as step 3.

## 8. Invariants

Properties the rest of the model layer relies on; reviewer should probe these.

1. `model/exceptions.py` is a runtime leaf — its only runtime imports are stdlib. `EntityId`, `GridPosition`, `GameState` resolve under `TYPE_CHECKING` only.
2. `model/events.py` is a runtime leaf — its only runtime imports are `dataclasses` (stdlib) and `blinker`. The three TYPE_CHECKING-only payload-field-type names (`EntityId`, `GridPosition`, `GameState`) resolve under `TYPE_CHECKING` only; `int` (used by `TurnAdvancedPayload.turn_number`) is a builtin and resolves at runtime.
3. `model/state/states.py` is a runtime leaf — only stdlib imports (`abc`, `typing`, `inspect`).
4. `events.py` is the **canonical declaration site** for every signal and every payload dataclass in the model layer (umbrella §7 invariant 10). Enforced by two complementary checks: (a) the `blinker-only-in-events` `.importlinter` forbidden contract — direct imports of `blinker` from any `stmrr.model.*` module other than `stmrr.model.events` are CI failures; (b) `tests/unit/model/test_payload_location.py` — a regex sweep over model source asserting no `class \w*Payload` declaration exists outside `events.py`. The contract catches signal creation outside events (which requires a `blinker` import); the test catches payload-class declarations outside events (which don't require a `blinker` import and so escape the contract).
5. Every `IllegalActionError` subclass and `IllegalTransitionError` accepts kwargs only — `cls(positional_arg)` raises `TypeError`.
6. Every concrete `GameState` subclass declares its own `allowed_transitions` (not inherited). Omission is a class-definition-time error, not a runtime read-time error.
7. `typing.get_type_hints()` raises `NameError` for the three payload classes with TYPE_CHECKING-only field types (`ShipMovedPayload`, `DockedPayload`, `StateChangedPayload`) and **succeeds** for `TurnAdvancedPayload` (only `int` field, always in scope). This locks the runtime-leaf property of `events.py` against accidental "fixes" that would create import cycles, while staying honest about the per-payload behaviour.
8. After step 5, the model layer's runtime guard (`test_no_qt_imports.py`) imports every `stmrr.model.*` submodule via dynamic walk and finds no Qt symbol in the subprocess's `sys.modules`. The walk discovers both implementation modules and scaffold package markers (`combat`, `missions`, `resources`, `systems`, etc.) — the test asserts the Qt-free *property*, not a count of entries.

## 9. Open questions for review

### 9.1 Locked decisions

- **Bundling — three modules in one spec, one PR.** Step 3's two-module precedent justifies the shape; the `events.py` ↔ `state.states` forward reference forces the bundling.
- **`GameState` ABC stub only — concretes deferred to gsm step.** Smaller spec, concretes designed against their consumer.
- **`__init_subclass__` enforcement of `allowed_transitions` via `inspect.isabstract(cls)`.** mypy `--strict` does not enforce subclass override of inherited class attributes. Property-abstractmethod is more ceremony for a static lookup table. `__init_subclass__` is the run-once-at-import-time check that matches the data-as-attribute shape. `inspect.isabstract(cls)` is mandatory in place of `cls.__abstractmethods__` because the latter raises `AttributeError` inside `__init_subclass__` under Python 3.14 (ABCMeta populates the attribute after `type.__new__` returns, i.e. after the hook has run). `vars(cls)` check (not `hasattr`) ensures the subclass declared the attribute itself.
- **Hybrid exception attribute richness (umbrella §5.6 + this spec §4.3-§4.4).** Each subclass carries the v0.1-UX-relevant attrs as kwargs-only fields with locked `__str__` formats. Bare exceptions force string-parsing; rich-everything is YAGNI for v0.2 concerns.
- **Locked `__str__` formats.** v2.0 localization replaces them with translation keys; v0.1 ships English literals.
- **`stmrr_events` namespace internal.** Subscribers import named signals; the `Namespace` instance is an implementation detail.
- **`IllegalActionError` and `ModelError` concrete, not abstract.** ABC machinery for an exception class is gratuitous.
- **`NotDockableError` no reason discriminator.** v0.2 problem; current attrs cover v0.1 + v0.2 debugging.
- **`test_no_qt_imports.py` dynamic walk + subprocess isolation, not explicit list and not in-process.** Dynamic walk has no stale-list failure mode. Subprocess isolation is the only clean boundary against `sys.modules`-additive contamination from `pytest-qt` or other view-layer fixtures running earlier in the same pytest process.
- **Per-payload `get_type_hints` expectations.** Three TYPE_CHECKING-deps payloads raise `NameError`; `TurnAdvancedPayload` (int-only) succeeds. Locked verbatim per payload in §5.2 + §7.3.
- **Invariant 10 promoted from grep convention to `.importlinter` contract.** `blinker-only-in-events` forbidden contract; mirrors how step 4 promoted Qt-free for projection.
- **Annotation deferral lives under PEP 649 (Python 3.14 default) + PEP 563 (`from __future__ import annotations`)**, not "PEP 563 alone." Wording corrected throughout the spec.
