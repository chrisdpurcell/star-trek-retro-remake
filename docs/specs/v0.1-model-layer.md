## 1. Purpose

The model layer is the simulation: turn logic, AP, entities, sector geometry, AI state, mission state, resource state. It runs headless under `pytest` with no `QApplication`, no display server, and no Qt of any kind. The view renders what the model says is true; the controller is the seam that translates Qt input into model calls and model events into Qt signals. This spec covers the model surface that lands by the end of v0.1 and surfaces the cross-module decisions — events, IDs, state-machine contract, save/load lifecycle — that a per-module spec can't see. The companion module-level spec for the first two modules is [`v0.1-step-3-grid-position-and-game-object.md`](v0.1-step-3-grid-position-and-game-object.md).

## 3. Layering and the Qt-free rule

ADR-0003 ([`docs/adr/0003-model-qt-free.md`](../adr/0003-model-qt-free.md)): *"`src/stmrr/model/` must not import `PySide6`, `shiboken6`, or any other Qt module."* The rule is enforced mechanically by `import-linter`'s `model-is-qt-free` contract in `.importlinter`, run on every CI push and pre-commit. A secondary contract (`view-does-not-import-model-events`) forbids the view from subscribing to model events directly — the only path is through `controller/model_bridge.py`. `DESIGN.md` §9.1 calls this the *"single seam"*.

Every model module starts with `from __future__ import annotations`. Public APIs require complete mypy-strict type hints (`DESIGN.md` §10.6). Naming follows §10.6: `PascalCase` classes, `snake_case` functions, `UPPER_SNAKE_CASE` module constants, **`snake_case` past-tense events** (`ship_moved`, `turn_advanced`) so the bridge translation is mechanical.

## 5. Cross-cutting concerns

### 5.1 Entity ID space

- IDs are `EntityId = NewType("EntityId", int)` minted by a module-level monotone generator in `model/entities/game_object.py` (spec A §5.1).
- **Cross-module entity references are by ID, not by object reference.** `TurnManager` references the player as `player_id: EntityId`, not `player: Starship`. `SectorMap.get(id)` is the resolution mechanism.
- **Why ID-based references:** save/load (v0.2) must serialize a graph; object references would force the deserializer to fix up arbitrary cycles, while IDs round-trip through TOML as plain integers and re-resolve at load time via `SectorMap.get`. `DESIGN.md` §9.5 + ADR-0004.
- **Cost:** code that wants to act on an entity does an `O(1)` dict lookup. Acceptable; small entity counts (per `DESIGN.md` §9.4).
- ID stability across save/load is open — see §8 question 3.

### 5.2 Event vocabulary

- Names are `snake_case` past-tense, matching Qt's `Signal` convention so `controller/model_bridge.py` can rename mechanically (`DESIGN.md` §10.6).
- **Payload shape:** `blinker.signal(...).send(sender, **kwargs)` with explicit named kwargs. No tuples, no positional args. Each event in `events.py` carries a docstring listing the kwargs and their types so subscribers and the bridge stay in sync.

```python
# model/events.py
from blinker import Namespace

stmrr_events = Namespace()

ship_moved = stmrr_events.signal("ship_moved")
"""Sent when a Starship's position changes.
Kwargs: ship_id: EntityId, from_position: GridPosition, to_position: GridPosition
"""

docked = stmrr_events.signal("docked")
"""Sent when a Starship docks at a Station.
Kwargs: ship_id: EntityId, station_id: EntityId
"""

turn_advanced = stmrr_events.signal("turn_advanced")
"""Sent when the turn counter advances. Kwargs: turn_number: int"""

state_changed = stmrr_events.signal("state_changed")
"""Sent when GameStateManager transitions. Kwargs: from_state: type[GameState], to_state: type[GameState]"""
```

- **Where events are declared:** `model/events.py` only. No module-local signals. Single grep target.
- **Sync vs. queued:** open question, see §8 question 2.

### 5.3 State machine contract

Per ADR-0005 ([`docs/adr/0005-hand-rolled-state-machine.md`](../adr/0005-hand-rolled-state-machine.md)) — hand-rolled, ~200 lines, no `transitions` library, no `QStateMachine`.

`GameStateManager.transition_to(target_state)` semantics:

1. Validate that `type(target_state)` is in `current_state.allowed_transitions`. If not, raise `IllegalTransitionError` with both states named.
2. Call `current_state.exit()`.
3. Update `current_state` reference to `target_state`.
4. Call `target_state.enter()`.
5. Emit `state_changed(from_state=…, to_state=…)`.

The order matters: `exit()` before mutation, `enter()` before emission. Subscribers reading `manager.current_state` inside a `state_changed` handler see the new state. Mutating before exit, or emitting before enter, breaks observer semantics in subtle ways under v0.2's combat ↔ sector transitions.

`GameState` ABC: `enter()`, `exit()`, `allowed_transitions: frozenset[type[GameState]]`. Concrete v0.1 states: `MainMenuState` allows transition to `SectorMapState`; `SectorMapState` allows transition to `MainMenuState`. The full state set lands as features arrive.

### 5.4 Time, turns, AP

- `TurnManager` lives at `model/combat/turn_manager.py` (placement intentional — v0.2 combat extends, doesn't replace it). Owns `current_turn:int` (starts at 1), `player_ap_max:int` (5 per `DESIGN.md` §3.1, modifiable later by difficulty mode), and a callable hook to restore AP across all active entities on `advance_turn`.
- AP is debited inside the action method (e.g., `Starship.move_to` calls `debit_ap(1)` before mutating position). On insufficient AP, raise `InsufficientAPError`; the controller catches it and surfaces a UI message.
- `advance_turn()` is the only thing that emits `turn_advanced`. No other module emits it.
- Per `DESIGN.md` §3.1, AP fully restores at start of each turn; unused AP does not carry over. Restoration happens *inside* `advance_turn`, before the event is emitted.

### 5.5 Save/load round-trip

Out of scope to *implement* in v0.1, but every v0.1 model entity must be designed to support it without breaking changes:

- All entity state is plain attributes (no closures, no live references to lambdas, no Qt objects).
- All cross-entity references are `EntityId` (§5.1).
- `GridPosition` is a frozen dataclass with `(x, y, z)` ints — trivially TOML-serializable.
- `Starship`, `Station`: hull/shield/AP/services are plain ints, strings, sets of strings.
- ADR-0004 ([`docs/adr/0004-toml-not-pickle.md`](../adr/0004-toml-not-pickle.md)) bans `pickle` and `dill` outright.

The persistence module is its own spec (later session). v0.1 just keeps the door open.

## 6. Module dependency graph

A DAG. Foundation modules (`world/grid_position`, `entities/game_object`, `events`) sit at the bottom; dependency arrows point *up* from feature modules toward foundation. `events.py` is a pure leaf — every other module may import it; it imports nothing from inside `stmrr.model`.

```text
                +------------------------+
                | state/game_state_manager|
                +-----------+------------+
                            |
                +-----------v------------+
                |   state/states         |
                +-----------+------------+
                            |
       +--------------------+----------------------+
       |                                           |
+------v-----------+                     +---------v----------+
| combat/          |                     | world/sector_map   |
| turn_manager     |                     +---------+----------+
+------+-----------+                               |
       |                                           |
       |          +-------------------+            |
       +--------->| entities/starship |<-----------+
                  +---------+---------+
                            |
                  +---------v---------+
                  | entities/station  |<------+
                  +---------+---------+       |
                            |                 |
                  +---------v---------+       |
                  | entities/game_obj |       |
                  +---------+---------+       |
                            |                 |
                  +---------v---------+       |
                  | world/grid_pos    |       |
                  +-------------------+       |
                                              |
              +------------+                  |
              | events.py  |<-----------------+
              | (leaf)     |  (every module above imports events)
              +------------+
```

No cycles. `world/sector_map.py` imports `entities/game_object` (for the `EntityId` type) and `world/grid_position`; it does **not** import `entities/starship` or `entities/station` to avoid an upward dependency. `SectorMap` stores `GameObject` instances and trusts callers to pass concrete subclasses.

## 7. Invariants for the model layer

1. No module under `stmrr.model.*` imports `PySide6` or `shiboken6`. Mechanically enforced by `import-linter`.
2. Every active entity is reachable from a `SectorMap` (live entities) or the `GameStateManager`'s active state (transitional). No orphan references; no dangling IDs.
3. Every model event fires *after* the state mutation that justifies it, never before. Subscribers reading mutated state inside an event handler see the new value.
4. The blinker bus in `model/events.py` is the only allowed cross-module change-notification channel. Modules do not hold direct references to each other's mutable state for change notification.
5. All randomness in the model is seeded; given the same seed and input sequence, the simulation is deterministic. Required for replay tests, save/load fidelity, and AI determinism (`DESIGN.md` §5.6).
6. Model code does not call `print` or stdlib `logging` directly; it uses `loguru`.
7. Model code performs no file I/O; persistence is the persistence layer's job.
8. Cross-module entity references are by `EntityId`, never by object reference. Local references inside a single method are fine.
9. State transitions follow exit → mutate → enter → emit. No exceptions.
10. `model/events.py` is the single file declaring signals. No module-local signals.

## 8. Open questions for review

1. **`GameStateManager` lifecycle: singleton, DI, or module-level instance?** A singleton couples tests; DI is verbose; a module-level `manager = GameStateManager()` is convenient but hard to swap in tests. The controller bridge needs *some* way to subscribe; what is the canonical route?
2. **Synchronous vs. queued events.** Synchronous (handler runs in the emitter's frame) is simpler but couples emit-order to handler-execution-order; a deeply nested mutation that emits during another handler causes reentrancy bugs. Queued (drained at end-of-turn) decouples but introduces latency between mutation and view update. v0.1 likely needs synchronous; v0.2 combat may need queued. Lock the v0.1 choice now.
3. **`EntityId` stability across save/load.** Monotone-int IDs are stable within a process but the counter resets on reload, so external references (mission log, replay log, save metadata) must be re-resolved. UUID4 IDs survive reloads but lose ordering. Which constraint matters more? See also spec A §9.3.
4. **Does `GameStateManager` emit `state_changed`, or does it call a helper in `events.py` to emit?** The first is direct and cohesive; the second is testable in isolation and avoids tight coupling. v0.1 has only one emitter, so the question only matters once v0.2 adds more state-related events.
5. **Entity class authoring: `attrs`, `dataclasses`, or pure pydantic?** Pure pydantic gives free TOML schemas but couples the entity layer to a third-party validator at construction time. `dataclasses` keeps the layer dependency-light (matches `GridPosition`) but requires a separate persistence-time pydantic schema. `attrs` is in between. Pick one *now* before subclasses proliferate.
6. **Does `SectorMap.bounds_check` raise or return False?** Raising surfaces invariant violations loudly; returning False puts the burden on the caller. Adjacent issue: should `SectorMap.add(entity)` validate that `entity.position` is in-bounds, or trust the caller?

## 9. Risks and unknowns

- **If event ordering is ambiguous,** the view will render stale state. Subscribers reading `manager.current_state` mid-handler may see the old reference. Mitigation: invariant 3 + 9; lock §8 question 2 before v0.1 ships.
- **If entity IDs are not stable across save/load,** v0.2's mission log and replay tests break invisibly. Mitigation: resolve §8 question 3 before the persistence module lands.
- **If the state machine grows past ~7 states,** the hand-rolled approach (ADR-0005) starts to creak. Mitigation: ADR-0005 already defines the off-ramp ("revisit `transitions` if state count exceeds ~15").
- **If a model module accidentally takes on a Qt dependency,** headless tests die and CI fails — but only after the import-linter contract is in force. The contract is in force as of the v0.1 scaffold (CHANGELOG `[Unreleased]`); the risk is a transient test-time import that import-linter doesn't detect (e.g., conditional imports inside a function body). Mitigation: code review + a dedicated test that imports every model module and asserts `PySide6` is not in `sys.modules`.
- **If `SectorMap` stores object references and the persistence layer later wants ID-based serialization,** every entity needs auditing. Mitigation: invariant 8 + ID-based references from day one.
