## 1. Purpose

The model layer is the simulation — turn logic, AP, entities, sector geometry, state, events. It runs headless under `pytest` with no `QApplication` and no Qt. The view renders what the model says is true; the controller is the seam between Qt input/signals and model calls/events. This spec covers the v0.1 model surface and the cross-module decisions a per-module spec can't see (events, IDs, state-machine, save/load readiness, action contracts). Companion module-level spec: [`v0.1-step-3-grid-position-and-game-object.md`](v0.1-step-3-grid-position-and-game-object.md).

## 3. Layering and the Qt-free rule

ADR-0003 ([`docs/adr/0003-model-qt-free.md`](../adr/0003-model-qt-free.md)): *"`src/stmrr/model/` must not import `PySide6`, `shiboken6`, or any other Qt module."* The rule is enforced mechanically by `import-linter`'s `model-is-qt-free` contract in `.importlinter`, run on every CI push and pre-commit. A secondary contract (`view-does-not-import-model-events`) forbids the view from subscribing to model events directly — the only path is through `controller/model_bridge.py`. `DESIGN.md` §9.1 calls this the *"single seam"*.

Every model module starts with `from __future__ import annotations`. Public APIs require complete mypy-strict type hints (`DESIGN.md` §10.6). Naming follows §10.6: `PascalCase` classes, `snake_case` functions, `UPPER_SNAKE_CASE` module constants, **`snake_case` past-tense events** (`ship_moved`, `turn_advanced`) so the bridge translation is mechanical.

## 5. Cross-cutting concerns

### 5.1 Entity ID space

- IDs are `EntityId = NewType("EntityId", int)` minted by a module-level monotone generator in `model/entities/game_object.py` (spec A §5.1). **v0.1 locks monotone `int`** (spec A §9.1, this spec §8.1); UUID4 migration, if any, is a v0.2 persistence-spec concern.
- **Cross-module entity references are by ID, not by object reference.** `TurnManager` references the player as `player_id: EntityId`, not `player: Starship`. `SectorMap.get(id)` is the resolution mechanism.
- **Why ID-based references:** save/load (v0.2) must serialize a graph; object references would force the deserializer to fix up arbitrary cycles, while IDs round-trip through TOML as plain integers and re-resolve at load time via `SectorMap.get`. `DESIGN.md` §9.5 + ADR-0004.
- **Cost:** code that wants to act on an entity does an `O(1)` dict lookup. Acceptable; small entity counts (per `DESIGN.md` §9.4).

### 5.2 Event vocabulary

- Names are `snake_case` past-tense, matching Qt's `Signal` convention so `controller/model_bridge.py` can rename mechanically (`DESIGN.md` §10.6).
- **Payload shape:** `blinker.signal(...).send(sender, payload=Payload(...))`. The first positional argument to `send` is the `sender`; the single `payload` kwarg carries a frozen dataclass declared in `model/events.py`. The `sender` value is part of the event contract — subscribers can filter by sender, and bridge tests assert on it. Frozen dataclass payloads (vs. raw `**kwargs`) buy mypy-strict structural typing at the codebase's highest-risk seam — the bridge, the only place model and Qt meet — at the cost of one type per event. Four payloads in v0.1; landing them now spares v0.2's event explosion a migration.
- **Delivery is synchronous for v0.1:** each `send()` call invokes every connected receiver before returning. Queued delivery is a v0.2 problem if combat needs it.
- **Receiver invocation order is undefined.** Blinker calls receivers in connection order in current implementations, but its API does not document this as a contract; receivers **must** be order-independent — no receiver may rely on another running first or last. The `controller/model_bridge.py` is the single subscriber per event in v0.1, which makes the constraint trivial to satisfy; v0.2 must continue to honor it.

| Event | `sender` | Payload type | Emitted by |
|---|---|---|---|
| `ship_moved` | the `Starship` instance that moved | `ShipMovedPayload(ship_id: EntityId, from_position: GridPosition, to_position: GridPosition)` | `Starship.move_to` |
| `docked` | the `Starship` instance that docked | `DockedPayload(ship_id: EntityId, station_id: EntityId)` | `Starship.dock_at` |
| `turn_advanced` | the `TurnManager` instance | `TurnAdvancedPayload(turn_number: int)` | `TurnManager.advance_turn` |
| `state_changed` | the `GameStateManager` instance | `StateChangedPayload(from_state: type[GameState], to_state: type[GameState])` | `GameStateManager.transition_to` |

```python
# model/events.py
from __future__ import annotations

from dataclasses import dataclass
from typing import TYPE_CHECKING

from blinker import Namespace

if TYPE_CHECKING:
    from stmrr.model.entities.game_object import EntityId
    from stmrr.model.state.states import GameState
    from stmrr.model.world.grid_position import GridPosition


@dataclass(frozen=True, slots=True)
class ShipMovedPayload:
    ship_id: EntityId
    from_position: GridPosition
    to_position: GridPosition


@dataclass(frozen=True, slots=True)
class DockedPayload:
    ship_id: EntityId
    station_id: EntityId


@dataclass(frozen=True, slots=True)
class TurnAdvancedPayload:
    turn_number: int


@dataclass(frozen=True, slots=True)
class StateChangedPayload:
    from_state: type[GameState]
    to_state: type[GameState]


stmrr_events = Namespace()

ship_moved = stmrr_events.signal("ship_moved")
"""Sender: Starship. Payload: ShipMovedPayload."""

docked = stmrr_events.signal("docked")
"""Sender: Starship. Payload: DockedPayload."""

turn_advanced = stmrr_events.signal("turn_advanced")
"""Sender: TurnManager. Payload: TurnAdvancedPayload."""

state_changed = stmrr_events.signal("state_changed")
"""Sender: GameStateManager. Payload: StateChangedPayload."""
```

- **Where events and payloads are declared:** `model/events.py` only. No module-local signals; no payload classes outside `events.py`. Single grep target.
- **`events.py` stays a runtime leaf.** Payload field types are imported under `TYPE_CHECKING` only. PEP 563 (`from __future__ import annotations`) stringifies field annotations at class-definition time, so `@dataclass` does not eagerly resolve them — preserving the no-cycles property in §6's runtime DAG.
- **PEP 563 footgun — runtime type introspection.** `typing.get_type_hints()` on a payload class will raise `NameError` because field-type imports (`EntityId`, `GridPosition`, `GameState`) are `TYPE_CHECKING`-only and absent from `events.py`'s runtime globals. Direct attribute access (`payload.ship_id`) and `dataclasses.fields()` (which returns string annotations) are safe; runtime type *resolution* is not. The v0.1 bridge does plain attribute access and is unaffected. If a future caller (pydantic introspection, generic test helpers, debug tooling) needs resolved hints, expose a typed accessor in the bridge rather than relaxing the leaf-module constraint.

### 5.3 State machine contract

Per ADR-0005 ([`docs/adr/0005-hand-rolled-state-machine.md`](../adr/0005-hand-rolled-state-machine.md)) — hand-rolled, ~200 lines, no `transitions` library, no `QStateMachine`.

`GameStateManager.transition_to(target_state)` semantics:

1. Validate that `type(target_state)` is in `current_state.allowed_transitions`. If not, raise `IllegalTransitionError` (declared in `model/exceptions.py`, peer to `IllegalActionError` per §4) with both states named.
2. Call `current_state.exit()`.
3. Update `current_state` reference to `target_state`.
4. Call `target_state.enter()`.
5. Emit `state_changed(from_state=…, to_state=…)`.

The order matters: `exit()` before mutation, `enter()` before emission. Subscribers reading `manager.current_state` inside a `state_changed` handler see the new state. Mutating before exit, or emitting before enter, breaks observer semantics in subtle ways under v0.2's combat ↔ sector transitions.

`GameState` ABC: `enter()`, `exit()`, `allowed_transitions: frozenset[type[GameState]]`. Concrete v0.1 states: `MainMenuState` allows transition to `SectorMapState`; `SectorMapState` allows transition to `MainMenuState`. The full state set lands as features arrive.

### 5.4 Time, turns, AP — single source of truth

**AP lives on the entity, not on the `TurnManager`.** `Starship` owns:

```python
ap_remaining: int   # current pool, debited by actions
ap_max: int = 5     # cap, restored on turn advance, modifiable later by difficulty mode
```

`TurnManager` owns the turn counter and orchestrates restoration; it has no `debit_player_ap` method. Collaboration:

1. **Action side:** `Starship.move_to` and `Starship.dock_at` debit `self.ap_remaining` via an internal `_debit_ap(cost)` helper that raises `InsufficientAPError` if short. Action specs in §5.6.
2. **Turn-advance side:** `TurnManager.advance_turn(sector_map)` resolves `sector_map.get(self.player_id)`, asserts result is a `Starship` (concrete-class import per §6), calls `player.restore_ap()`, increments `self.current_turn`, emits `turn_advanced`.
3. **No reentrancy:** restoration completes before the event is emitted (invariant 3).

`TurnManager` takes a single `player_id: EntityId` for v0.1; v0.2 generalizes when NPC ships join. Per `DESIGN.md` §3.1, unused AP does not carry over; `restore_ap()` overwrites.

### 5.5 Save/load round-trip readiness

Out of scope to *implement* in v0.1, but every entity must be shaped so the round-trip can land in v0.2 without breaking changes:

- **Plain-attribute state.** No closures, lambdas, generators, or live Qt objects. Attributes are primitives, a frozen dataclass (`GridPosition`), an `EntityId`, a `frozenset[str]`, or another `GameObject` reference *resolvable by ID*.
- **Subclass discriminator.** Every `GameObject` subclass declares `kind: ClassVar[str]` (`"starship"`, `"station"`). The persistence spec uses it as the TOML table tag for subclass dispatch on load. v0.1 lands the field even though no save code reads it yet, avoiding a save-format migration later.
- **Cross-entity references by `EntityId`** (invariant 8). Persistence serializes `dict[EntityId, GameObject]` as a TOML array keyed by `kind`+`id`; `EntityId` round-trips as a plain integer.
- **Set normalization.** `Station.services: frozenset[str]` serializes as a sorted list, deserializes back into a `frozenset`. Hash-stable, TOML-compatible (TOML has no native set). Lock now so v0.2 doesn't migrate from `set`.
- **No file I/O in the model** (invariant 7).

ADR-0004 ([`docs/adr/0004-toml-not-pickle.md`](../adr/0004-toml-not-pickle.md)) bans `pickle` and `dill`.

### 5.6 v0.1 action contracts

The v0.1 vertical slice exercises three actions. Each names a single owner method, a precondition list, an ordered side-effect sequence, and the event it emits. `DESIGN.md` §10.1 DoD items #5, #6, #7 are not satisfied until all three pass tests.

**Error types** (declared in `model/exceptions.py` — see §4 module table): all action preconditions raise an `IllegalActionError` subclass (`InsufficientAPError`, `OutOfBoundsError`, `NotAdjacentError`, `InactiveEntityError`, `NotDockableError`). Subclasses let the controller produce precise error messages without inspecting a string `kind`.

#### 5.6.1 Move

- **Owner:** `Starship.move_to(target: GridPosition, sector_map: SectorMap) -> None`
- **Preconditions** (checked in this order; first failure raises):
  1. `self.active` (else `InactiveEntityError`)
  2. `sector_map.bounds_check(target)` is `True` (else `OutOfBoundsError`)
  3. `self.position.is_adjacent(target)` is `True` (else `NotAdjacentError`) — v0.1 is single-cell moves only; multi-cell pathing is v0.2.
  4. `self.ap_remaining >= 1` (else `InsufficientAPError`) — v0.1 cost is fixed at 1 AP per cell. The Pythagorean-diagonal cost rule (`DESIGN.md` §5.1) lands in v0.2 when movement extends beyond single-cell.
- **Effects** (ordered):
  1. Capture `from_position = self.position` (local, before mutation).
  2. `self.ap_remaining -= 1`.
  3. `self.position = target` — direct attribute assignment. There is no base `GameObject.move_to` to delegate to; `Starship.move_to` owns the action semantics end-to-end (precondition → AP debit → position assignment → event), and a base `move_to` with a different signature (`(self, position)` vs. `(self, target, sector_map)`) would be a mypy `--strict` `[override]` violation. See companion spec [`v0.1-step-3-grid-position-and-game-object.md`](v0.1-step-3-grid-position-and-game-object.md) §5.3.
  4. `ship_moved.send(self, payload=ShipMovedPayload(ship_id=self.id, from_position=from_position, to_position=target))`.

#### 5.6.2 Dock

- **Owner:** `Starship.dock_at(station_id: EntityId, sector_map: SectorMap) -> None`
- **Preconditions** (in order):
  1. `self.active` (else `InactiveEntityError`).
  2. `target = sector_map.get(station_id)` is not `None` and `isinstance(target, Station)` and `target.active` (else `NotDockableError` — combined check; the controller doesn't care which sub-failure triggered it). The conflation of "stale ID" / "wrong type" / "destroyed" under one error is fine for v0.1; v0.2 may split it once the UI distinguishes "stale ID" vs. "wrong faction" vs. "destroyed" for player-facing messaging.
  3. `self.position.is_adjacent(target.position)` (else `NotAdjacentError`).
  4. `target.accepts_dock(self)` is `True` (else `NotDockableError`). v0.1 always returns `True` for any active `Starship`; the predicate is the v0.2 extension point for faction/reputation gating.
  5. `self.ap_remaining >= 1` (else `InsufficientAPError`).
- **Effects** (ordered):
  1. `self.ap_remaining -= 1`.
  2. `docked.send(self, payload=DockedPayload(ship_id=self.id, station_id=station_id))`.
- No state on the `Station` mutates. Service application (repair, resupply, mission briefing) is UI-side / mission-system-side in later versions; v0.1 only proves the action pipeline, per ADR-0011.

#### 5.6.3 End Turn

- **Owner:** `TurnManager.advance_turn(sector_map: SectorMap) -> None`
- **Preconditions:** none — End Turn is always available.
- **Effects** (ordered):
  1. `player = sector_map.get(self.player_id)`. If `None` or not a `Starship`, raise `InactiveEntityError`. **This branch is unreachable in v0.1 by construction** — the player ship is added to `SectorMap` at session start and is never removed during a v0.1 session. The check lands now to harden against v0.2 persistence-layer foot-guns where a stale `player_id` could survive a save/load round-trip without its referent.
  2. `player.restore_ap()` — sets `player.ap_remaining = player.ap_max`.
  3. `self.current_turn += 1`.
  4. `turn_advanced.send(self, payload=TurnAdvancedPayload(turn_number=self.current_turn))`.

These three contracts are the v0.1 model's entire action surface. Any other "action" the UI offers in v0.1 (selection, camera pan, z-level switch) is view-side only.

### 5.7 `SectorMap.bounds_check` and `add` policy

**`bounds_check(position) -> bool`** returns `True` iff `0 <= position.x < self.width` and the same for `y < height` and `z < depth`. **Never raises** — it's a predicate, used in conditions.

**`add(entity)`** calls `bounds_check(entity.position)` and raises `ValueError` if it returns `False`, so an out-of-bounds entity never enters the map.

Predicate returns bool, mutator raises on contract violation — a stable pattern across the model layer.

### 5.8 World ownership lives at the wire-up layer

`SectorMap` instances are owned by `app.py` (or a session object outside the model layer), **not** by `GameStateManager` and **not** by any `GameState` subclass. `TurnManager.advance_turn(sector_map)` and `Starship.move_to(target, sector_map)` take the sector as a parameter precisely because the model layer does not own world construction; the wire-up layer does. `SectorMapState.enter()` is a state-machine hook for view/UI cueing, not a constructor — it does **not** instantiate `SectorMap`. Without this rule, `SectorMapState` accumulates world-construction logic and the model leaks lifecycle concerns into a state-machine hook, which v0.2 combat-vs-sector transitions then have to untangle.

## 6. Module imports — runtime DAG

Authoritative per-module import list. Runtime imports are evaluated at module load; `TYPE_CHECKING` imports are evaluated only by mypy and do not contribute to the runtime DAG.

| Module | Runtime imports from `stmrr.model.*` | `TYPE_CHECKING`-only |
|---|---|---|
| `events` | (none — runtime leaf) | `entities.game_object` (`EntityId`), `world.grid_position` (`GridPosition`), `state.states` (`GameState`) — all for payload field types only |
| `exceptions` | (none — leaf) | (none) |
| `world.grid_position` | (none — leaf) | (none) |
| `entities.game_object` | `world.grid_position` | (none) |
| `entities.station` | `entities.game_object`, `world.grid_position` | `entities.starship` (`accepts_dock` parameter type) |
| `entities.starship` | `entities.game_object`, `world.grid_position`, `events`, `exceptions` | `world.sector_map`, `entities.station` (action-method parameter types) |
| `world.sector_map` | `world.grid_position`, `entities.game_object`, `exceptions` | (none — duck types entities as `GameObject`) |
| `combat.turn_manager` | `entities.starship`, `world.sector_map`, `events`, `exceptions` | (none) |
| `state.states` | (none — ABC plus concrete states use no other model modules) | (none) |
| `state.game_state_manager` | `state.states`, `events`, `exceptions` | (none) |

**No cycles.** Runtime leaves: `events`, `exceptions`, `grid_position`. `events`'s payload types resolve under `TYPE_CHECKING` only (`entities.game_object`, `world.grid_position`, `state.states`); PEP 563 stringifies the annotations, so `@dataclass` does not import them at module-load time. `game_object` → `grid_position`. `station`/`starship` → `game_object` + leaves. `sector_map` → `game_object` + leaves. `turn_manager` → `starship` + `sector_map` (no back-edge). `state.*` is an isolated subtree.

**`TurnManager` imports `Starship` at runtime** so `advance_turn` can run `isinstance(entity, Starship)` before `restore_ap()`. No cycle — `Starship` does not import `turn_manager`. Invariant 8 (cross-module references by ID) is unaffected: `TurnManager` stores `player_id`, not a `Starship` reference.

**`SectorMap` does not import `Starship` or `Station`.** It stores `GameObject` instances by `EntityId`; subclass dispatch happens in the action methods.

## 7. Invariants for the model layer

1. No module under `stmrr.model.*` imports `PySide6` or `shiboken6`. Statically enforced by `import-linter`'s `model-is-qt-free` contract; *additionally* a model-layer-wide test obligation — `tests/unit/model/test_no_qt_imports.py` — imports every `stmrr.model.*` module and asserts `"PySide6" not in sys.modules` and `"shiboken6" not in sys.modules`, catching transient test-time imports (e.g., conditional inside a function body) the static contract misses.
2. Every active entity is reachable from a `SectorMap` (live entities) or the `GameStateManager`'s active state (transitional). No orphan references; no dangling IDs.
3. Every model event fires *after* the state mutation that justifies it, never before. Subscribers reading mutated state inside an event handler see the new value.
4. The blinker bus in `model/events.py` is the only allowed cross-module change-notification channel. Modules do not hold direct references to each other's mutable state for change notification.
5. All randomness in the model is seeded; given the same seed and input sequence, the simulation is deterministic. Required for replay tests, save/load fidelity, and AI determinism (`DESIGN.md` §5.6).
6. Model code does not call `print` or stdlib `logging` directly; it uses `loguru`.
7. Model code performs no file I/O; persistence is the persistence layer's job.
8. Cross-module entity references are by `EntityId`, never by object reference. Local references inside a single method are fine.
9. State transitions follow exit → mutate → enter → emit. No exceptions.
10. `model/events.py` is the single file declaring signals **and their payload dataclasses**. No module-local signals; no payload classes outside `events.py`.

## 8. Open questions for review

### 8.1 Locked decisions

- **Sync vs. queued events:** synchronous for v0.1 (§5.2). Receiver order is undefined; receivers must be order-independent.
- **`EntityId` stability:** monotone `int` for v0.1. UUID migration, if any, is a v0.2 persistence-spec concern (spec A §9.1).
- **`GameStateManager` emission:** GSM emits `state_changed` directly via `model.events` (§4).
- **Entity authoring:** plain Python classes (mutable) for entity subclasses; `GridPosition` stays a frozen dataclass; pydantic schemas land at the persistence boundary, not on entities.
- **`SectorMap.bounds_check`:** returns `bool`, never raises; `SectorMap.add` raises `ValueError` if `entity.position` fails `bounds_check` (§5.7).
- **`TurnManager` ↔ `Starship` coupling:** `combat/turn_manager.py` imports `Starship` at runtime and uses `isinstance` directly (§6). No `_starship_cls` injection, no `Protocol` indirection.
- **`GameStateManager` lifecycle:** constructed once in `app.py` and passed into the bridge. No singleton (couples tests); no module-level instance (awkward to swap in tests); no DI container in v0.1 (verbose for one consumer).
- **Typed event payloads:** frozen dataclasses (`ShipMovedPayload`, `DockedPayload`, `TurnAdvancedPayload`, `StateChangedPayload`) declared in `model/events.py`, passed via a single `payload=` kwarg. The bridge is the codebase's highest-risk seam (only place model and Qt meet); strong typing there is worth four small dataclasses in v0.1, and lands now so v0.2's event explosion does not have to migrate from raw kwargs.

### 8.2 Still open

1. **Replay-log integration: subscriber to model events, or a side-channel that records action invocations?** The first is decoupled but reconstructs state from events (lossy if events ever lack data); the second is precise but couples replay to every action method. v0.2 mission log + v1.0 replay both need this answered.

## 9. Risks and unknowns

- **State-mutation order.** Subscribers reading `manager.current_state` mid-handler may see the old reference if invariant 9 (exit → mutate → enter → emit) slips. Mitigation: invariants 3 and 9; bridge tests assert post-emit state.
- **`EntityId` migration when persistence lands.** v0.1 monotone `int` IDs reset on process restart, so v0.2 must choose between persisting the counter, switching to UUID4, or rebuilding ID maps at load time. The v0.2 persistence spec owns this decision; v0.1 just keeps the door open per §5.5.
- **State-machine growth past ~7 states.** ADR-0005 already defines the off-ramp ("revisit `transitions` if state count exceeds ~15").
- **Accidental Qt dependency.** `import-linter` is in force as of the v0.1 scaffold; the residual risk is a transient test-time import (e.g., conditional inside a function body) that the static contract misses. Mitigation owned in §7 invariant 1: `tests/unit/model/test_no_qt_imports.py` imports every `stmrr.model.*` module and asserts `PySide6`/`shiboken6` are not in `sys.modules`.
- **Object references vs. IDs.** If `SectorMap` ever stores object references and the persistence layer later wants ID-based serialization, every entity needs auditing. Mitigation: invariant 8 + ID-based references from day one.
