## 1. Purpose

The **first `QWidget`-class module in the codebase** and the first runnable entry point. `view/main_window.py` is the empty `QMainWindow` shell from the scaffold step order (`tech-stack-pyside6.md` §11 step 5 / `DESIGN.md` §10.1 vertical-build step 5 — *"empty `QMainWindow` shell with menu bar and dock placeholders. Verify it launches."*). `app.py` is the entry point that builds the `QApplication`, wires the MVC triad (`GameStateManager` → `ModelBridge` → `MainWindow`), and shows the window; `__main__.py` makes `uv run python -m stmrr` launch it.

This step **closes the `app.py` deferral** the step-9 spec recorded (`v0.1-step-9-model-bridge.md` §2, §5.7): the bridge's owner is now real, so its lifetime contract — construct the GSM, construct `ModelBridge(manager, parent=main_window)`, and call `bridge.teardown()` owner-side in `MainWindow.closeEvent()` (never `destroyed.connect`) — is implemented here, not just specified.

The window is a **shell**: `MapView`/`GridScene`/items do not exist yet (`tech-stack-pyside6.md` §11 steps 6–7), so the central area, dock contents, and most menu/toolbar actions are deliberate placeholders. The one piece of real behavior is the **minimal MVC seam consumer**: the window reads `bridge.current_state` for an initial state indicator and updates it on `bridge.state_changed`, proving the model→controller→view path end-to-end before any scene exists — which is the entire point of the vertical build order (`DESIGN.md` §10.1 *"catches MVC seam issues in week one"*).

## 3. Layering and constraints

- **Amends one existing `import-linter` contract (no NEW contract).** `main_window.py` annotates with `ModelBridge` under `TYPE_CHECKING`, creating a `stmrr.view.main_window → stmrr.controller.model_bridge` graph edge. **import-linter includes `TYPE_CHECKING` imports and checks *indirect* chains by default** (verified empirically 2026-05-29 + Import Linter docs), so the existing `view-does-not-import-model-events` `forbidden` contract would flag the indirect `view → controller.model_bridge → model.events` path and break CI (SA-001). Fix: add `allow_indirect_imports = True` to that one contract — it then forbids only a **direct** `view → model.events` import (the dangerous edge that actually bypasses the bridge), while permitting the legitimate indirect path through the bridge. **Empirically verified (2026-05-29):** with the amendment, a `TYPE_CHECKING` `ModelBridge` import yields `5 kept / 0 broken`, *and* a direct `from stmrr.model.events import …` in a view module still reports `BROKEN` — the contract keeps its teeth. This mirrors the two existing seam contracts (`blinker-only-in-events`, `controller-events-via-bridge-only`), which already use `allow_indirect_imports = True` for the same reason. (A local `Protocol` to avoid the import entirely was tried first and **rejected** — it satisfies mypy but fails Pyright on a `Signal`/`SignalInstance` descriptor mismatch; see §8.4.)
- **`view-does-not-import-model-events` (amended) is honored.** `main_window.py` imports **no** `stmrr.model.*` at all (its only cross-layer import is the `TYPE_CHECKING` `ModelBridge`). It treats the `state_changed` payload opaquely (`payload: object`) and reads the *class name* of `bridge.current_state` without importing `GameState`. This keeps the view decoupled from the model (conventions §1/§2).
- **`app.py` (`stmrr.app`) is governed by none of the view/controller `forbidden` contracts** (they scope `stmrr.view` / `stmrr.controller`). It may freely import `GameStateManager`, `MainMenuState` (model.state), `ModelBridge` (controller), and `MainWindow` (view). It does **not** import `model.events` — the bridge is the only events importer (the `controller-events-via-bridge-only` contract is unaffected).
- **Imports** (runtime vs `TYPE_CHECKING`) in `main_window.py`:
  - **Runtime:** `from PySide6.QtCore import Qt` (dock-area + alignment enums), `from PySide6.QtGui import QAction, QCloseEvent`, `from PySide6.QtWidgets import (QMainWindow, QWidget, QLabel, QDockWidget, QTextEdit, QToolBar, QPushButton, QMessageBox, QVBoxLayout)`, `from stmrr import __version__` (root package only — for the About text; **not** a `model.*` import, and `stmrr/__init__.py` pulls in no submodules).
  - **`TYPE_CHECKING` only:** `from stmrr.controller.model_bridge import ModelBridge` (annotation on `bind_bridge` / the `_bridge` attribute; the instance is injected at runtime — no hard runtime coupling to the controller).
  - `from __future__ import annotations` at the top per `DESIGN.md` §10.6.
- **mypy:** `stmrr.view.*` is `disallow_untyped_defs` (not full `strict`); `stmrr.app` and `stmrr.__main__` fall under the loose global (`disallow_incomplete_defs`). All three files are written **fully annotated and strict-clean regardless** (correctness over the config floor). No `pyproject.toml` mypy-override change is proposed — full annotations already satisfy the existing settings, and adding `stmrr.app` to `strict` is left as a future tightening, not bundled here.
- No new dependencies (`PySide6`, `loguru`, `pytest-qt` already declared).
- Module + class docstrings mandatory. Method docstrings on `bind_bridge` (seam-consumer rationale), `closeEvent` (owner-side teardown order — non-obvious), and `build_main_window` (the testable-seam rationale).

**`bind_bridge(bridge)`** (the minimal-but-real seam consumer):

```python
def bind_bridge(self, bridge: ModelBridge) -> None:
    """Attach the model→view seam: initial render + live state tracking.

    Stores the bridge for owner-side teardown (closeEvent), renders the initial
    mode from `current_state` (the step-9 initial-render hook, before any
    transition fires), and tracks subsequent transitions via `state_changed`.
    The payload is treated opaquely (`object`) so the view imports no model.events.

    Does NOT (re)parent the bridge — the caller (`app.py`) constructs it already
    parented via `ModelBridge(manager, parent=window)`; bind_bridge only stores
    the reference, renders, and connects. Calling it twice would stack a second
    `state_changed` connection — v0.1 binds exactly once at startup.
    """
    self._bridge = bridge
    self._state_indicator.setText(type(bridge.current_state).__name__)
    bridge.state_changed.connect(self._on_state_changed)
```

**`_on_state_changed(payload)`**:

```python
def _on_state_changed(self, payload: object) -> None:
    # View slot: payload is opaque (no model.events import). StateChangedPayload
    # carries `to_state: type[GameState]`; read it duck-typed.
    to_state = getattr(payload, "to_state", None)
    name = to_state.__name__ if isinstance(to_state, type) else repr(payload)
    self._state_indicator.setText(name)
```

**`_on_fullscreen_toggled(checked)`**: `self.showFullScreen()` if `checked` else `self.showNormal()`. (Tested via the action's `isChecked()` + a mock on `showFullScreen`/`showNormal`, **never** `isFullScreen()` — research #4.)

**`_show_about()`**: `QMessageBox.about(self, "About Star Trek Retro Remake", f"...{__version__}...")` using `stmrr.__version__`. (Tested by monkeypatching `QMessageBox.about` — research #8.)

**`closeEvent(event)`** (owner-side teardown — research #3, step-9 §5.7):

```python
def closeEvent(self, event: QCloseEvent) -> None:
    """Release the bridge's blinker subscriptions at window close (exit-only).

    Step-10 close is an EXIT path: v0.1 leaves `quitOnLastWindowClosed` at Qt's
    default (True), so closing the main window quits the event loop and the
    process exits — there is NO reopen-after-close. (Accepting a close event only
    HIDES the widget unless `WA_DeleteOnClose` is set, which step 10 does not set;
    the process exit is what tears everything down — SA-005.) teardown() runs
    before super().closeEvent() as defensive ordering: it releases the blinker
    subs (bridge ← model events) while the bridge is unambiguously alive; the
    Qt-signal half (view slots ← bridge signals) is Qt object-tree parenting's
    job. NOT wired via destroyed.connect (step-9 §5.7 / step-9 SA-002). Reopen
    support, if ever needed, would require re-binding a fresh bridge — out of
    scope for v0.1.
    """
    if self._bridge is not None:
        self._bridge.teardown()
    super().closeEvent(event)
```

### 4.2 `app.py` — entry point

```python
from __future__ import annotations

import sys

from loguru import logger
from PySide6.QtWidgets import QApplication

from stmrr import __version__
from stmrr.controller.model_bridge import ModelBridge
from stmrr.model.state.game_state_manager import GameStateManager
from stmrr.model.state.states import MainMenuState
from stmrr.view.main_window import MainWindow


def build_main_window() -> tuple[MainWindow, ModelBridge, GameStateManager]:
    """Construct and wire the MVC triad; return all three for lifetime ownership.

    The testable seam: no QApplication creation, no exec(). **Requires a GUI
    QApplication to already exist** (main() or pytest-qt creates it). Constructing
    a QWidget without one aborts the process at the C++ level, so this public,
    importable seam fails fast with a clear RuntimeError instead (CR-002). The
    single isinstance guard covers both no-app (instance() is None) and a non-GUI
    QCoreApplication singleton. Order matters — the window must exist before the
    bridge so the bridge can be Qt-parented to it (step-9 §5.7). Returns the bridge
    and manager so the caller holds Python refs that keep them alive (PySide6
    parenting does NOT pin the Python wrapper, and blinker holds the bridge with
    weak=False — research #5).
    """
    if not isinstance(QApplication.instance(), QApplication):
        raise RuntimeError(
            "build_main_window() requires a GUI QApplication; call stmrr.app.main() "
            "or construct a QApplication first."
        )
    manager = GameStateManager(MainMenuState())
    window = MainWindow()
    bridge = ModelBridge(manager, parent=window)
    window.bind_bridge(bridge)
    return window, bridge, manager


def main() -> int:
    """Build/reuse the QApplication, show the main window, run the event loop."""
    logger.info("Starting Star Trek Retro Remake v{}", __version__)
    app = QApplication.instance()
    if app is None:
        # Only when WE create the app: own it and set metadata. Under pytest-qt a
        # session QApplication already exists (SA-002); reusing it avoids the
        # "destroy the QApplication singleton before creating a new one"
        # RuntimeError, and NOT mutating its name/org keeps the shared test app
        # pristine for other tests.
        app = QApplication(sys.argv)
        app.setApplicationName("Star Trek Retro Remake")
        app.setOrganizationName("star_trek_retro_remake")  # conventional; see §5.5
    elif not isinstance(app, QApplication):
        # A non-GUI QCoreApplication already exists: instance() returns IT (not
        # None — verified 2026-05-29). Widgets cannot be constructed under it, and
        # a second QApplication cannot replace a live singleton. Fail clearly
        # instead of the cryptic C++ abort "QWidget: Cannot create a QWidget
        # without QApplication" (SA-NEW-001).
        raise RuntimeError(
            "stmrr requires a GUI QApplication, but a non-GUI QCoreApplication "
            "already exists; cannot start the main window."
        )
    # After the branches above, `app` is narrowed to QApplication.
    window, _bridge, _manager = build_main_window()
    window.show()
    return app.exec()
```

- **`build_main_window()` requires (does not create) a GUI `QApplication`** — it raises a clear `RuntimeError` if none exists or if the singleton is a non-GUI `QCoreApplication` (CR-002), rather than letting `MainWindow()` abort the process at the C++ level. `main()` satisfies this by ensuring the app first; tests satisfy it via `qtbot`. The guard's `raise` is exercised by a subprocess test (not countable by in-process coverage — §6.2 coverage note).
- **`main()` reuses an existing `QApplication`** via `QApplication.instance()`, constructing one only when absent (SA-002), and **rejects a non-GUI `QCoreApplication`** singleton with a clear `RuntimeError` rather than aborting on `MainWindow` construction (SA-NEW-001 — `instance()` returns the `QCoreApplication`, not `None`; verified 2026-05-29). The reuse-safety makes the `main()` smoke test safe under pytest-qt (whose session `qapp` exists); a second `QApplication(sys.argv)` would raise `RuntimeError: Please destroy the QApplication singleton…` (reviewer-probed). After the guard block, `app` is narrowed to `QApplication`, so `app.exec()` typechecks with no cast (this resolves the earlier A18 union-typing caveat).
- `build_main_window` keeps the bridge + manager referenced via the returned tuple; `main` binds them to `_bridge`/`_manager` locals that live for the duration of `exec()`. The window also holds `self._bridge`; the bridge holds `self._manager`. Both the Qt parent **and** the Python ref are present (research #5).
- `setApplicationName`/`setOrganizationName` are set **only on an app `main` itself constructs**, before any widget or `QSettings` — conventional startup hygiene (§5.5). They do not, alone, establish the design's exact INI path (SA-003).
- loguru's default stderr sink is used as-is (one `info` line) — sufficient for DoD #1 *"no errors in the loguru log."* No custom sink configured this step.

## 5. Cross-cutting design decisions

### 5.1 Window built bare, bridge attached via `bind_bridge` (chicken-and-egg)
Step-9 §5.7 specifies `ModelBridge(manager, parent=main_window)` — the bridge is parented to the window, so the **window must be constructed first**. A constructor that took the bridge would invert that order. `bind_bridge()` resolves it: build the window, build the bridge parented to it, then attach. `bind_bridge` is also the natural home for the initial-render hook and the `state_changed` connection, and it makes the seam independently testable (construct a window, call `bind_bridge(fixture_bridge)`, assert it reacts).

### 5.2 The minimal seam consumer is real, not ceremonial
With no scene or status panels yet, the window has little model state to render — but rendering *something* from `current_state` + `state_changed` is what proves the model→controller→view path in week one (`DESIGN.md` §10.1). The `_state_indicator` label is that consumer: it shows `"MainMenuState"` at startup (initial-render hook, step-9 §4.3) and updates on every transition. It is replaced/augmented by real dock panels in later steps, but the seam it exercises is permanent.

### 5.3 The view imports no `model.*`; payload is opaque
`view-does-not-import-model-events` forbids `stmrr.view → stmrr.model.events`. Rather than import the payload type from elsewhere, the slot annotates `payload: object` and reads `to_state` duck-typed (`getattr` + `isinstance(to_state, type)`), and the initial render reads `type(current_state).__name__`. No `stmrr.model.*` import appears in `main_window.py` at all — the strongest form of the decoupling, and it sidesteps any debate about whether `view → model.state` (a non-events model import) is acceptable.

### 5.4 Fullscreen as a checkable `QAction`; tested by intent, not WM outcome (research #4)
The `offscreen` QPA plugin (mandated for tests, conventions §4) has no window manager and does **not** reliably report `isFullScreen()`/`hasFocus()`/`isActiveWindow()`. So F11 is a **checkable** `QAction` whose `toggled(bool)` drives `showFullScreen()`/`showNormal()`, and the test asserts (a) the action's `isChecked()` flipped and (b) `showFullScreen`/`showNormal` was *called* (mock), never the realized WM state. `setMinimumSize` and programmatic geometry **are** reliable under offscreen and may be asserted directly.

### 5.5 `app/organizationName` now, `QSettings` later (SA-003 corrected)
Persistence (DoD #8) is deferred. Setting application/organization names at startup is conventional hygiene and avoids the `QSettings.status() == AccessError` a default-constructed `QSettings()` hits when they are unset (research #9). **But it does not establish the design's `~/.config/star_trek_retro_remake/window_state.ini` path** — default `QSettings()` on Unix uses NativeFormat (`.conf`), and the exact `.ini` filename requires the explicit `QSettings(path, QSettings.IniFormat)` constructor. That filename/format decision belongs to the persistence step (SA-003); step 10 only sets the names, and only on an app it constructs (§4.2).

### 5.6 `build_main_window()` / `main()` split for testability (research #2)
`exec()` blocks; pytest-qt already runs its own loop, so tests must never call the real `exec()`. Splitting wiring (`build_main_window`, no loop) from the loop (`main`) lets tests exercise all wiring directly and smoke-test `main()` by monkeypatching `QApplication.exec`. The split also keeps each function within one screen.

### 5.7 Disabled placeholders, not enabled no-ops (review: missing consideration)
Every unwired menu/toolbar/turn-bar action is created `setEnabled(False)`, not left enabled-but-inert. An enabled control that silently does nothing on click is a UX defect and an ambiguous test target; a disabled (greyed) control honestly reads "not yet available." Later steps flip `setEnabled(True)` as they wire each action to `action_handlers`. Only the four self-contained actions (Exit, Fullscreen, the two dock toggles, About) ship enabled.

## 7. Invariants

Properties step 10 contributes; a reviewer should probe these.

1. **`python -m stmrr` launches the window** (DoD #1) — `__main__.py` → `app.main()` → `QApplication` + `MainWindow().show()` + `exec()`, with a loguru startup line and no errors.
2. **The window is the bridge's owner.** `MainWindow.closeEvent` calls `bridge.teardown()` **before** `super().closeEvent()`; the bridge is Qt-parented to the window **and** referenced by `self._bridge`. Never `destroyed.connect(teardown)` (step-9 §5.7).
3. **The view imports no `stmrr.model.*`.** `main_window.py` consumes `current_state` by class name and `state_changed` payloads opaquely; its only cross-layer import is the `TYPE_CHECKING` `ModelBridge`. The amended `view-does-not-import-model-events` (`allow_indirect_imports = True`) still forbids any *direct* `view → model.events` edge — verified empirically (SA-001).
4. **`view → controller.model_bridge` (TYPE_CHECKING only) is the window's sole cross-layer import**, for the `ModelBridge` annotation. Five import-linter contracts stay green; one (`view-does-not-import-model-events`) is amended, none added. A local `Protocol` alternative was rejected (Pyright `Signal`/`SignalInstance` fragility, §8.4).
5. **The seam is live end-to-end.** Binding a bridge renders the initial state name; a real model `transition_to` updates it via blinker → bridge Qt signal → view slot. Pinned by §6.1 tests.
6. **Fullscreen is asserted by intent, not WM outcome.** Tests check `QAction.isChecked()` + a mocked `showFullScreen`/`showNormal`, never `isFullScreen()` (offscreen has no WM — research #4).
7. **Window minimum size is 1600×1000** (`DESIGN.md` §6.5) — reliable under offscreen, asserted directly.
8. **`stmrr.view` re-exports `MainWindow`** with `__all__ == ["MainWindow"]` (mirrors step-8/9 package-export discipline). Pinned by a §6.1 test.
9. **Every test-constructed bridge is torn down** (no cross-test blinker accumulation; step-9 §6.1 `weak=False` rationale).
10. **`main()` never constructs a second `QApplication`, and refuses a non-GUI singleton.** It reuses `QApplication.instance()` when present (SA-002) and raises a clear `RuntimeError` if the existing singleton is a non-GUI `QCoreApplication` (SA-NEW-001) rather than aborting on widget construction. **`build_main_window()` likewise refuses to construct widgets without a GUI `QApplication`** (CR-002). Pinned by the §6.2 same-instance test + the two subprocess guard tests (`main()` under `QCoreApplication`; `build_main_window()` with no app).
11. **Unwired controls are disabled, not enabled no-ops** — only Exit/Fullscreen/dock-toggles/About ship enabled (§5.7); pinned by the §6.1 disabled-placeholder test.
12. **Close is exit-only** — `closeEvent` tears down the bridge; v0.1 quits on last-window-close with no reopen path (§4.1, SA-005).
