## 1. Purpose

Two modules that complete the v0.1 model layer's **world surface** — the bounded grid container plus the first non-ship inhabitant:

- `model/world/sector_map.py` — `SectorMap(width, height, depth)`: dict-backed entity container with bounds checking, ID-keyed lookup, position-indexed lookup, and snapshot iteration. Owns no actions; owned by the wire-up layer per umbrella §5.8.
- `model/entities/station.py` — `Station(GameObject)`: dock-target subclass with a forward-compatible `station_type` Literal (v0.1 accepts only `"starbase"`), an immutable `services` set, and the `accepts_dock` predicate the action pipeline calls.

They land together because (a) `Station.accepts_dock` is what `Starship.dock_at` (step 7) will call, and `Starship.dock_at` will resolve the station via `SectorMap.get`; designing the container and the dock-target together surfaces ID-resolution and bounds-check decisions a per-module spec would miss. (b) Both are runtime leaves of the not-yet-landed action layer — no `Starship`, no `TurnManager` — so step 7 can build on a finished world surface without circular review pressure. (c) Both depend only on already-landed step-3 code (`GridPosition`, `GameObject`); reviewer cognitive load stays bounded.

This spec also surfaces and amends an inconsistency in the umbrella spec's §6 runtime-DAG table: `sector_map` is listed as runtime-importing `model.exceptions`, but §5.7 specifies `add(entity)` raises plain `ValueError` (not a model exception). Step 6 raises `ValueError` (research-grounded per [`docs/research/2026-05-23-python314-idioms-game-model-layer.md`](../research/2026-05-23-python314-idioms-game-model-layer.md) §3) and amends umbrella §6 to drop the stale import.

## 4. `model/world/sector_map.py` specification

### 4.1 Class declaration

```python
from __future__ import annotations

from stmrr.model.entities.game_object import EntityId, GameObject
from stmrr.model.world.grid_position import GridPosition


class SectorMap:
    """A bounded 3-D entity container indexed by EntityId.

    Owned by the wire-up layer (umbrella §5.8), passed by reference into
    action methods (Starship.move_to, Starship.dock_at, TurnManager.advance_turn).
    Owns no game logic — it is the storage + spatial-bounds primitive.
    """

    def __init__(self, width: int, height: int, depth: int) -> None:
        ...
```

`SectorMap` is a **plain class, not a dataclass**: it carries a mutable `_entities` dict, the public dimension attributes are read-only by convention (not by enforcement — see §4.2), and it has no need for the `frozen`/`eq`/`order` machinery a dataclass would provide. Equality is identity (the default `object.__eq__`); two `SectorMap` instances with identical contents are not equal — they're different worlds.

### 4.2 Construction validation

`__init__(width, height, depth)`:

1. **Type check** (each axis, in order `width`, `height`, `depth`): `isinstance(value, int) and not isinstance(value, bool)`. On failure: `TypeError(f"{axis_name} must be int, got {type(value).__name__}")`. Same `bool`-is-an-int-subclass footgun mitigation as `GridPosition.__post_init__` (step 3 §4.2).
2. **Range check** (each axis): `value >= 1`. On failure: `ValueError(f"{axis_name} must be >= 1, got {value}")`. A 0-dimension sector is degenerate — every `bounds_check` returns `False` and every `add` raises. `DESIGN.md` §4.3 caps dimensions at 20×20×7, but the lower bound `>= 1` is the only sanity check v0.1 enforces; the upper bound is a `SectorMap` *caller* concern (config validation), not a `SectorMap` invariant.
3. Assign `self.width: int = width`, `self.height: int = height`, `self.depth: int = depth`, `self._entities: dict[EntityId, GameObject] = {}`.

Public attributes are conventionally read-only — no `@property` machinery, no `__setattr__` overrides. Mutation of `self.width` after construction is undefined behaviour; reviewers should not add tests for it.

### 4.3 Method contracts

| Method | Signature | Semantics | Complexity | Errors |
|---|---|---|---|---|
| `bounds_check` | `(self, position: GridPosition) -> bool` | `True` iff `0 <= position.x < self.width` and same for `y < height` and `z < depth`. **Never raises** — predicate, not a guard. | O(1) | none |
| `add` | `(self, entity: GameObject) -> None` | Insert `entity` keyed by `entity.id`. See §4.3.1 for the precondition order. | O(1) | `ValueError` (see §4.3.1) |
| `remove` | `(self, entity_id: EntityId) -> None` | `del self._entities[entity_id]`. Mirrors `dict.__delitem__` semantics — strict, not idempotent. | O(1) | `KeyError` if `entity_id not in self._entities` (raised by `del`) |
| `get` | `(self, entity_id: EntityId) -> GameObject \| None` | `self._entities.get(entity_id)`. Returns `None` for missing IDs. | O(1) | none |
| `at` | `(self, position: GridPosition) -> list[GameObject]` | Linear scan; returns every entity whose `.position == position`, **including inactive** entities. Order matches dict insertion order (Python 3.7+ guarantee). Returns `[]`, never `None`. | O(n) over `len(self._entities)` | none |
| `entities` | `@property -> tuple[GameObject, ...]` | `tuple(self._entities.values())` — fresh snapshot, safe against concurrent mutation during iteration. Order matches dict insertion order. | O(n) | none |
| `__contains__` | `(self, entity_id: EntityId) -> bool` | `entity_id in self._entities`. Lets callers write `if some_id in sector:` instead of `if sector.get(some_id) is not None:`. | O(1) | none |
| `__len__` | `(self) -> int` | `len(self._entities)`. Lets callers write `len(sector)` for entity count without exposing the private dict. | O(1) | none |

#### 4.3.1 `add` precondition order

Checks in this order; first failure raises and the dict is not mutated:

1. **Out-of-bounds:** `self.bounds_check(entity.position)` is `False` → `ValueError(f"entity position {entity.position} is outside sector bounds ({self.width}, {self.height}, {self.depth})")`.
2. **Duplicate ID:** `entity.id in self._entities` → `ValueError(f"entity id {entity.id} already present in sector")`.
3. Insert.

**Why `ValueError` for both, not `KeyError` for duplicate-ID and not `OutOfBoundsError` for out-of-bounds:**

- Per CPython convention (`Built-in Exceptions` docs, ["passing arguments with the wrong value"](https://docs.python.org/3/library/exceptions.html#ValueError)), `ValueError` signals "the value is wrong"; `KeyError` signals "a key being looked up was not found." The duplicate-add case is "this value (entity) is wrong for the current state" — it's not a failed lookup. Research finding §3, corroborated across the Python docs + the `esper`/`tcod-ecs`/`mecs` ECS surveys: `KeyError` on missing remove (lookup-style), `ValueError` on wrong-shape add (value-style). The asymmetry is intentional.
- `OutOfBoundsError` (declared in step-5's `model/exceptions.py` as an `IllegalActionError` subclass) is specifically for **action preconditions** — `Starship.move_to`'s bounds check. `SectorMap.add` is a setup/spawn operation, not a player action; routing it through `IllegalActionError` would conflate setup failures with player-facing action failures in `except IllegalActionError:` blocks. Plain `ValueError` keeps the action-error namespace clean. This is the research-supported resolution to the umbrella §6 inconsistency — see §10.

#### 4.3.2 `at(position)` includes inactive entities

`at(position)` returns every entity at that position regardless of `active` state. Rationale: `at` is a spatial query primitive; callers know best whether they want to filter (`[e for e in sector.at(p) if e.active]` is one line at the call site). Filtering inside `at` would force every caller that *does* want inactive entities (debug overlays, post-mortem queries, persistence dumps) to bypass it — and there is no analogous "include inactive" flag on `dict.values()` that this method mirrors.

The same applies to the `entities` property: every entity in the dict is returned, inactive or not. `GameObject.deactivate` is soft-delete (step 3 §5.6); `SectorMap.remove(entity_id)` is the hard-delete primitive. The two are not the same operation, and `at`/`entities` reflect the dict state, not a filtered view of it.

### 4.4 Iteration and snapshot semantics

`SectorMap` does **not** implement `__iter__`. Callers iterate via `for e in sector.entities:` (snapshot tuple, safe to mutate during iteration) or `for entity_id in sector._entities:` (private — single-purpose, mostly tests). Explicit:

- **No `__iter__` on the public surface.** A hypothetical `for e in sector:` is ambiguous between "iterate by ID key" (dict behaviour) and "iterate by entity value." Avoiding it forces the call site to be specific.
- **`entities` is a `tuple`, not a `frozenset`.** Order matters for replay determinism (umbrella §7 invariant 5 — given the same seed and action sequence, the simulation is deterministic). Dict insertion order is the determinism anchor; `frozenset` would lose it.

### 4.5 What's deliberately not on `SectorMap`

- **No spatial index.** `at(position)` is O(n); v0.1 entity counts make this fine. Adding a `dict[GridPosition, list[GameObject]]` shadow index doubles the storage cost and adds a mutation hazard (every `add`/`remove`/position-mutation must touch both structures). v0.2 picks this up if AI pathfinding profiles it as hot.
- **No `move(entity_id, new_position)` helper.** Position assignment is the action method's job (`Starship.move_to` writes `self.position = target` directly per umbrella §5.6.1). A `SectorMap.move` would either (a) duplicate the bounds check action methods already do, or (b) skip it and become a footgun. Keeping the mutation responsibility on action methods is one of umbrella §5.4's invariants.
- **No event emission.** `SectorMap` is the storage primitive; events are the action layer's job. Mutation of `_entities` (via `add`/`remove`) does not emit. If v0.2 needs spawn/despawn events (e.g., for the bridge to track entity creation), the action layer adds them, not the container.
- **No `__hash__` override.** `SectorMap` inherits `object.__hash__` (identity-based). Putting a `SectorMap` in a set is a code smell (worlds are not values), but Python's default behaviour is fine.

## 5. `model/entities/station.py` specification

### 5.1 Class declaration

```python
from __future__ import annotations

from collections.abc import Iterable
from typing import ClassVar, Literal, Protocol, TypeAlias, get_args

from stmrr.model.entities.game_object import GameObject
from stmrr.model.world.grid_position import GridPosition


class _Dockable(Protocol):
    """Structural type for entities that can dock at a Station.

    Step 6 must NOT depend on stmrr.model.entities.starship (the module
    does not exist until step 7). A naive ``if TYPE_CHECKING: from
    stmrr.model.entities.starship import Starship`` would fail under
    ``mypy --strict`` because mypy resolves TYPE_CHECKING-guarded imports
    during static analysis even though they're skipped at runtime — and
    the import target does not yet exist.

    Protocol sidesteps the issue. v0.1 contract: any object with
    ``active: bool`` satisfies ``_Dockable``. Step 7's ``Starship``
    satisfies it structurally (the ``active`` attribute is inherited
    from ``GameObject``) without requiring an explicit subclass
    relationship or an import edge in either direction.

    Step-7 contract: ``_Dockable`` is the umbrella-locked
    contract (umbrella §5.6.2 is amended in the same PR as this
    spec — see §10). Step 7 inherits ``ship: _Dockable``
    unchanged. Tightening back to ``Starship`` is permitted ONLY
    via an explicit new step-7 amendment to umbrella §5.6.2 with
    its own rationale (e.g., if step 7 introduces a method that
    needs Starship-specific attributes beyond ``active``). Absent
    such an amendment, the Protocol stays — both for v0.1 and for
    v0.2+ (the v0.2 faction/reputation widening just adds
    attributes to ``_Dockable`` itself).

    v0.2 extension: if faction/reputation gating widens the predicate
    (umbrella §5.6.2), this Protocol grows new required attributes
    (``faction: str``, etc.) and the concrete ``Starship`` is checked
    structurally against the wider contract by mypy.
    """

    active: bool


StationType: TypeAlias = Literal["starbase", "civilian", "military", "neutral"]
"""Wide Literal for forward-compatible save-file deserialization (umbrella §4 +
§5.5). All four values may appear in saved game files from v0.2 onward; v0.1
runtime rejects three of the four (see §5.2). Annotated as ``TypeAlias =
Literal[...]`` (NOT PEP 695 ``type StationType = Literal[...]``) so
``typing.get_args(StationType)`` returns the Literal args directly — PEP 695's
``type`` syntax wraps the Literal in a ``TypeAliasType`` and breaks ``get_args``.
See research §1."""

_V1_ALLOWED_STATION_TYPES: frozenset[str] = frozenset({"starbase"})
"""v0.1 runtime-accepted subset of StationType. Membership-tested (not
narrowed) — mypy does not narrow on ``in`` against a runtime set, so widening
``_V1_ALLOWED_STATION_TYPES`` in v0.2 requires no guard-code changes. See
research §4."""

_STATION_TYPE_ARGS: tuple[str, ...] = get_args(StationType)
"""Evaluated at module import. Used by tests (§7.2) to parametrize the
v0.1-rejected cases as ``set(_STATION_TYPE_ARGS) - _V1_ALLOWED_STATION_TYPES``,

class Station(GameObject):
    """A dockable structure on the sector grid.

    v0.1 supports the starbase variant only; civilian/military/neutral are
    reserved at the type level for v0.2 save-format forward-compatibility
    (umbrella §4) but rejected at runtime by __init__ (§5.2).
    """

    kind: ClassVar[str] = "station"

    def __init__(
        self,
        *,
        position: GridPosition,
        station_type: StationType,
        services: Iterable[str],
    ) -> None:
        ...
```

**Keyword-only `__init__`** (the leading `*` after `self`): three params, two of which are domain literals; positional construction at the call site would obscure which string is which (`Station(pos, "starbase", {"repair"})` reads worse than `Station(position=pos, station_type="starbase", services={"repair"})`). Same rationale as step-5's exception-class kwarg-only init (step-5 §4.3). The `position` parameter is also kwarg-only despite being inherited from `GameObject.__init__` — consistency across the call site wins over preserving the parent's positional signature.

**`kind: ClassVar[str] = "station"`** is the TOML save-format discriminator per umbrella §5.5. v0.1 lands the field even though no save code reads it yet, avoiding a save-format migration when persistence ships in v0.2.

### 5.2 `station_type` runtime validation + constructor order

`__init__` validates **both** `station_type` and `services` element types **before** calling `super().__init__(position)`. The full constructor body:

```python
def __init__(
    self,
    *,
    position: GridPosition,
    station_type: StationType,
    services: Iterable[str],
) -> None:
    # Validate BEFORE super().__init__() so failed construction does not
    # consume an EntityId from the module-global monotone counter
    # (Codex pass-1 SA-002; see "Why validate before super()" below).
    if station_type not in _V1_ALLOWED_STATION_TYPES:
        raise ValueError(
            f"station_type {station_type!r} is reserved for v0.2; "
            f"v0.1 supports only: {sorted(_V1_ALLOWED_STATION_TYPES)}"
        )

    # Reject string-shaped containers BEFORE freezing — `frozenset("repair")`
    # would silently produce a character set (Codex pass-2 SA-003).
    if isinstance(services, (str, bytes)):
        raise TypeError(
            f"services must be an iterable of str, not a bare "
            f"{type(services).__name__} (which iterates as individual "
            f"characters/bytes); pass a list/tuple/set/frozenset of str"
        )
    # Materialize ONCE — `services` may be a generator that can only be
    # iterated once, and we need to scan elements before frozenset().
    services_tuple = tuple(services)
    # Element-type check BEFORE frozenset() so unhashable non-str elements
    # raise the project's TypeError with a useful message, not the stdlib
    # generic `unhashable type: '<X>'` from frozenset() (Codex pass-2 SA-003).
    bad_types = sorted({type(s).__name__ for s in services_tuple if not isinstance(s, str)})
    if bad_types:
        raise TypeError(
            f"services must contain only str elements; "
            f"got non-str types: {bad_types}"
        )

    super().__init__(position)  # ID allocation happens here
    self.station_type: StationType = station_type
    self.services: frozenset[str] = frozenset(services_tuple)
```

**Why validate before `super().__init__(position)` (Codex pass-1 SA-002):** `GameObject.__init__` calls `_next_entity_id()` (step 3 §5.1), the module-global monotone counter, which advances on every call regardless of whether the constructed instance is returned to the caller. If validation ran *after* `super().__init__()` and then raised, the counter would have advanced but no entity would be bound to the consumed ID — silently introducing a gap in the otherwise contiguous-monotone sequence. v0.1 tests assert *relative* ordering rather than absolute values (step 3 §9.1), so a gap is invisible to v0.1 test conventions, but the umbrella's TOML round-trip readiness (§5.5) and the v0.2 persistence + replay specs may rely on the contiguous-monotone property. Validate-first keeps the contract clean now rather than discovering the leakage during v0.2 spec work. Locked as invariant 12 in §8.

**Why `ValueError`, not a model exception:** constructor validation = `ValueError`, matching `GridPosition.__post_init__` (step 3 §4.2). No model exception class fits — `IllegalActionError` subclasses are for action preconditions on already-constructed entities; this is "the value you passed me to construct with is wrong for v0.1."

**Why `if station_type not in _V1_ALLOWED_STATION_TYPES`, not `if station_type == "starbase"`:**

Both work in v0.1 (set has one element). The membership form scales when v0.2 widens `_V1_ALLOWED_STATION_TYPES` — no code change to the guard. The equality form would force re-touching `__init__` for every v0.2 widening.

**Why NOT an exhaustive `if/elif` chain over `get_args(StationType)`:** mypy narrows on `if x == "literal_value":` and treats the `else` as `Never` (research §4 + research's footgun on the FastAPI break in different form). When v0.2 widens `StationType`, mypy would flag the now-missing branches as a `return` violation — a useful safety property in some codebases, but here the *intent* is "v0.1 rejects everything except the v0.1 subset; widening the Literal must NOT require touching this guard." The membership-test form preserves that intent; the if/elif form fights it.

### 5.3 `services` normalization + element-type validation

`services: Iterable[str]` accepts any iterable of strings; `__init__` performs **three** validation steps before freezing — full code listing in §5.2:

1. **Reject string-shaped containers** (`isinstance(services, (str, bytes))` → `TypeError`). A bare `str` is technically `Iterable[str]` because iterating a string yields characters — `frozenset("repair")` produces `frozenset({'a', 'e', 'i', 'p', 'r'})` without complaint. The caller almost always meant "a station providing repair", not "a station with 5 single-char services." Same for `bytes`. Catching this at the door eliminates a silent-corruption footgun that bypasses every downstream check (Codex pass-2 SA-003).
2. **Materialize the iterable to a tuple** (`tuple(services)`). Generators can only be iterated once; if we element-check before freezing, the subsequent `frozenset()` would see an empty iterator. Tuple materialization is O(n) and lets us scan + freeze independently.
3. **Element-type check before `frozenset()`**. Computing `bad_types` over the tuple lets the project's `TypeError("services must contain only str elements; got non-str types: ...")` fire *before* `frozenset()` would raise the stdlib's generic `TypeError: unhashable type: 'list'` for inputs like `services=[["repair"]]` — a useful-message-vs-generic-message reordering (Codex pass-2 SA-003).

Then `self.services: frozenset[str] = frozenset(services_tuple)`. Each step closes a footgun the prior one couldn't see.

**Iterable-and-normalize** is the dominant idiom across pydantic v2, attrs, and stdlib dataclasses (research §2). The TOML load path returns `list[str]`, so requiring `frozenset[str]` strictly would force every load-time call site to wrap. The normalization is one operation; the type is locked as `frozenset[str]` post-init (immutable, hashable, TOML-round-trippable per umbrella §5.5).

**Element-type runtime validation (Codex pass-1 SA-003):** mypy `--strict` catches typed callers that pass non-`str` elements, but `frozenset(["repair", 1])` happily produces a mixed frozenset under untyped callers (test fixtures, future TOML load paths, reflection). Worse, v0.2 save serialization will canonicalize via `sorted(self.services)`, and `sorted()` raises `TypeError` on heterogeneous types in Python 3 — a v0.2 time-bomb planted in v0.1 construction. The `isinstance` check seals it now. Same defensive pattern as step 3 §4.2's `int`/`bool` validation on `GridPosition` axes (mypy can catch most callers but the runtime check is the durable contract). All three steps live in §5.2's reorder so they run *before* `super().__init__()` for the same EntityId-non-consumption reason.

**No other validation on the elements.** Empty strings, whitespace, duplicates, unicode (any `str` subclass that isn't blocked by `isinstance(s, str)`) — all accepted. v0.1 has no service registry to validate against; v0.2 may add one when stations expose actual services. Cost of deferring: a typo in a service name silently becomes a unique service. Acceptable for v0.1.

**`bytes` is rejected at both levels.** A `bytes` *argument* is the str/bytes rejection in step 1 above. A `bytes` *element* inside an iterable — `services=[b"repair"]` — is rejected by the element-type check in step 3 because `bytes` is not a `str` subclass. TOML has no native bytes type, so neither rejection is gratuitous; both prevent the v0.2 persistence layer from inheriting a hidden mixed-type contract.

**Order is lost.** `frozenset` discards insertion order. Tests must not assert on order — they assert on set equality. Save serialization will canonicalize via `sorted(self.services)` per umbrella §5.5; this is the persistence spec's concern.

### 5.4 `accepts_dock(ship)` semantics

```python
def accepts_dock(self, ship: _Dockable) -> bool:
    """v0.1: returns ship.active. v0.2 will extend with faction/reputation
    gating per umbrella §5.6.2; the predicate shape is locked now so v0.2
    additions are conjunctive (return ship.active and faction_ok(ship))."""
    return ship.active
```

**Why `return ship.active`, not `return True`:** umbrella §4 + §5.6.2 say "v0.1 always returns True for any active Starship." Reading this literally: *active* is the precondition, and the predicate must return `False` for an inactive ship. Returning `ship.active` directly is the cleanest expression: it makes the v0.2 extension purely conjunctive (`return ship.active and faction_ok(ship)`) instead of requiring a refactor. Defensive value also: `accepts_dock` is callable outside the `Starship.dock_at` precondition chain (UI tooltips, debug overlays, future query APIs) where the caller has not pre-checked `ship.active`.

**Why `ship: _Dockable` (Protocol) and not `ship: Starship`:** umbrella §5.6.2 originally said `accepts_dock(ship: Starship)`, but step 7's `Starship` does not yet exist as a Python module. A `TYPE_CHECKING` import of a not-yet-existing module fails under `mypy --strict` (CI gate) because mypy resolves TYPE_CHECKING-guarded imports during static analysis. `_Dockable: Protocol` (defined in §5.1) captures the actual structural contract — any object with `active: bool` qualifies — and step 7's `Starship` satisfies it for free via `GameObject.active` inheritance. This amends umbrella §5.6.2; see §10. The Protocol is a *more honest* contract than `Starship` because the predicate genuinely depends only on `active`, not on Starship-specific attributes.

**Why `ship: _Dockable` and not `ship: GameObject`:** `_Dockable` is the *minimum* contract this method requires; `GameObject` is broader (carries `id`, `position`, `deactivate()`) than necessary. Using the narrower Protocol documents intent — any future GameObject subclass with `active` (whether or not it's a Starship) can dock against the predicate, but the contract is tied to the actual surface used. Step 7 may optionally tighten back to `Starship` once available; the current Protocol stays valid either way.

### 5.5 What's deliberately not on `Station`

- **No `dock(ship)` method that mutates state.** `accepts_dock` is a predicate; the actual dock action lives on `Starship.dock_at(station_id, sector_map)` (step 7, per umbrella §5.6.2). Mutating `Station` state on dock would couple step 6 to step 7 and create a back-edge in the runtime DAG.
- **No service registry validation.** See §5.3 — v0.2 concern.
- **No `Faction` field, no shield/weapon/sensor systems, no hull/health.** Umbrella §4 explicitly excludes these from v0.1 (v0.2 per `DESIGN.md` §10.1).
- **No `__eq__` override.** Inherits `GameObject.__eq__` (identity-based per step 3 §5.4). Two `Station` instances at the same position with identical `station_type` and `services` are not equal — they're different stations.

## 8. Invariants

1. `SectorMap.__init__` rejects `width`/`height`/`depth` that are not `int`, are `bool`, or are `< 1`. No `SectorMap` instance reaches code with degenerate dimensions.
2. `SectorMap.bounds_check(position)` is a total function: returns `bool` for any `GridPosition` input, never raises.
3. `SectorMap.add(entity)` is atomic on failure — neither the out-of-bounds case nor the duplicate-ID case mutates `_entities`.

5. `SectorMap.entities` returns a fresh snapshot tuple per access; iteration over the snapshot is safe under concurrent `add`/`remove`. Order matches dict insertion order (Python 3.7+ language guarantee).
6. `SectorMap.at(position)` and `SectorMap.entities` return entities regardless of `active` state. Soft-delete (`GameObject.deactivate`) and hard-delete (`SectorMap.remove`) are distinct operations; container queries reflect membership, not activation.

8. `Station.__init__` rejects every `StationType` value not in `_V1_ALLOWED_STATION_TYPES` at runtime, regardless of which v0.1 callers pass it. The `StationType` annotation is intentionally wider than the v0.1 runtime contract — forward-compatibility for the v0.2 persistence layer per umbrella §4.
9. `Station.services` is a `frozenset[str]` post-init regardless of the input iterable's concrete type. Element types are enforced at construction time: a `services` iterable containing any non-`str` element raises `TypeError` before `super().__init__()` runs (SA-003). Hash-stable, TOML-round-trippable per umbrella §5.5 because no non-`str` element can ever survive construction.
10. `Station.accepts_dock(ship)` returns `False` for any ship with `ship.active == False`. v0.2 extension is purely conjunctive (`return ship.active and faction_ok(ship)`) — predicate shape is locked.
11. `SectorMap` and `Station` import **nothing** from `entities.starship` or `combat.turn_manager` — neither at runtime nor under `TYPE_CHECKING`. ## 9. Open questions for review

### 9.1 Locked decisions

- **Two-module scope, one PR.** Both modules are runtime leaves of step 7 (`Starship`, `TurnManager`); landing them together unblocks step 7 without forcing step 7 to pick which container/dock-target conventions to use under review pressure. Stations need a container to live in; the container's `add` semantics inform what action methods later expect.
- **`SectorMap.add` raises `ValueError` on both out-of-bounds and duplicate-ID; `remove` raises `KeyError` on missing entity.** Research §3 + Python `ValueError` vs `KeyError` convention. Asymmetric, but each exception communicates the actual failure type. `OutOfBoundsError` (declared in step-5 `model/exceptions.py`) is reserved for *action preconditions* (`Starship.move_to`'s bounds check), not setup operations.
- **`SectorMap` does not import `model.exceptions` at runtime.** Resolves the umbrella §6 stale-import claim — see §10. Plain builtin `ValueError`/`KeyError` cover every error condition step-6 introduces.
- **`SectorMap.at` and `entities` include inactive entities.** Container queries reflect dict membership, not activation. Filtering belongs at the call site.
- **`SectorMap` exposes no `__iter__`.** Ambiguous (key vs value iteration); callers use `entities` or `_entities` explicitly. `__contains__` and `__len__` are unambiguous so they are exposed.
- **`StationType: TypeAlias = Literal["starbase", "civilian", "military", "neutral"]` (NOT PEP 695 `type StationType = Literal[...]`).** PEP 695 syntax wraps the Literal in a `TypeAliasType` and breaks `typing.get_args(StationType)` (returns the TypeAliasType args, not the Literal args). Research §1 footgun.
- **v0.1 runtime restriction enforced by `_V1_ALLOWED_STATION_TYPES: frozenset[str]` membership test, not by `if/elif` chain.** Membership test does not narrow under mypy; widening `_V1_ALLOWED_STATION_TYPES` in v0.2 requires zero guard-code changes. `if/elif` over Literal values would narrow and demand `assert_never`, forcing every Literal widening to touch the guard. Research §4.
- **`Station.__init__` rejects v0.2-reserved `station_type` values with plain `ValueError`, not a model exception.** Constructor validation = `ValueError`, matching `GridPosition.__post_init__` (step 3 §4.2). No `IllegalActionError` subclass applies — `IllegalActionError` is for action preconditions on already-constructed entities.
- **`Station.services` accepts `Iterable[str]`, normalizes to `frozenset[str]` in `__init__`.** Dominant idiom per research §2 (pydantic, attrs, dataclasses). TOML load path returns `list[str]`, so strict-`frozenset` typing would force every load-time call site to wrap.
- **`Station.accepts_dock(ship)` returns `ship.active` (not `True`).** Inactive ships cannot dock. v0.2 extension is conjunctive (`return ship.active and faction_ok(ship)`); the predicate shape is locked now.
- **`Station.accepts_dock`'s parameter type is `_Dockable: Protocol` (defined inline in `station.py`), not `Starship`.** Codex pass-1 (SA-001) flagged that the originally-drafted `TYPE_CHECKING` import of `entities.starship` would fail under `mypy --strict` because mypy resolves TYPE_CHECKING-guarded imports during static analysis even though they're skipped at runtime — and step 7's `Starship` module does not yet exist. `_Dockable` requires only `active: bool`, which is the actual surface `accepts_dock` reads; step 7's `Starship` satisfies it structurally via `GameObject.active` inheritance. Alternatives considered and rejected: (a) `ship: GameObject` — too broad, doesn't document intent; (b) minimal `Starship` stub in step 6 — scope creep into step 7; (c) `# type: ignore[import-not-found]` on the TYPE_CHECKING import — hack. The Protocol amends umbrella §5.6.2; see §10. **Step 7 inherits `_Dockable` unchanged** — tightening back to `Starship` is permitted only via an explicit new step-7 amendment to umbrella §5.6.2 with its own rationale (e.g., if step 7 introduces a method that needs Starship-specific attributes beyond `active`). Absent such an amendment, the Protocol stays — both for v0.1 and v0.2+. This closes the Codex pass-2 ambiguity finding about `_Dockable` permanence.
- **Keyword-only `__init__` on `Station`** (leading `*` after `self`). Matches step-5 exception kwarg-only pattern. Forces call sites to name the literal-typed `station_type` argument.
- **`SectorMap` is a plain class, not a dataclass.** Mutable `_entities` dict, identity equality, no need for `frozen`/`eq`/`order` machinery. Equivalent to step 3's choice for `GameObject` (step 3 §5.2).
