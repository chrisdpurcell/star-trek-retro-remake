## Open questions

### Q1: Mutual class reference for `MainMenuState.allowed_transitions = {SectorMapState}` and vice versa

**Problem.** `allowed_transitions: ClassVar[frozenset[type[GameState]]]` is enforced at class-creation time by `states.py` `__init_subclass__` (presence check via `'allowed_transitions' not in vars(cls)`). But at the moment `MainMenuState` is being created, `SectorMapState` does not yet exist as a name in the module namespace — and vice versa. Both classes need to reference each other in their `allowed_transitions` frozensets.

**Option A — Post-class module-level patch** (recommended).
```python
class MainMenuState(GameState):
    allowed_transitions: ClassVar[frozenset[type[GameState]]] = frozenset()  # patched below
    def enter(self) -> None: ...
    def exit(self) -> None: ...

class SectorMapState(GameState):
    allowed_transitions: ClassVar[frozenset[type[GameState]]] = frozenset()  # patched below
    def enter(self) -> None: ...
    def exit(self) -> None: ...

MainMenuState.allowed_transitions = frozenset({SectorMapState})
SectorMapState.allowed_transitions = frozenset({MainMenuState})
```
Pros: trivial; mypy-strict-clean; idiomatic Python (no metaclass tricks). `__init_subclass__` is satisfied because `vars(cls)` contains `allowed_transitions` at class-creation time (empty frozenset is still a valid `frozenset[type[GameState]]`). The post-class assignment is module-level, runs at import time, completes before any caller sees either class.
Cons: a reader who sees the class body in isolation thinks the set is empty. Mitigate with an inline comment naming the patch site.

**Option B — Lazy `classmethod` resolution**.
```python
class MainMenuState(GameState):
    @classmethod
    def allowed_transitions(cls) -> frozenset[type[GameState]]:
        from stmrr.model.state.states import SectorMapState
        return frozenset({SectorMapState})
```
Pros: no module-level mutation.
Cons: breaks the spec contract — `allowed_transitions: ClassVar[frozenset[type[GameState]]]` is data, not a method. Changing it to a method ripples through the `GameStateManager.transition_to` validation (`type(target) in current_state.allowed_transitions` becomes `type(target) in current_state.allowed_transitions()`) and through every test. Also breaks the step-5 `__init_subclass__` presence check, which inspects `vars(cls)` for a *data* attribute. **Not viable** without amending step-5's contract.

**Option C — String names + runtime lookup**.
```python
class MainMenuState(GameState):
    allowed_transitions = frozenset({"SectorMapState"})  # type: ignore[assignment]
```
Pros: avoids ordering problem entirely.
Cons: defeats the type system (`ClassVar[frozenset[type[GameState]]]` becomes `frozenset[str]`); the manager would need to resolve names at transition time; `mypy --strict` rejects the type mismatch. **Rejected.**

**Option D — `__init_subclass__` registry** (deferred to v0.2 if state count grows).
Maintain a class-attribute mapping `_pending_transitions: dict[str, set[str]]` and resolve after all subclasses register. Heavy machinery for two states. **Rejected for v0.1**, possibly viable in v0.5+ if the state graph balloons past ~7 nodes (ADR-0005's stated revisit point).

**Recommendation:** **Option A** (post-class patch). The comment must explicitly name the patch site so a future reader inspecting only the class body does not assume the empty frozenset is the final value.

**Test discipline:** the spec must include a smoke test asserting the patched values are non-empty at import time — guards against accidental deletion of the patch lines:
```python
def test_main_menu_state_can_transition_to_sector_map() -> None:
    assert SectorMapState in MainMenuState.allowed_transitions

def test_sector_map_state_can_transition_to_main_menu() -> None:
    assert MainMenuState in SectorMapState.allowed_transitions
```

### Q2: Should the initial state's `enter()` fire in `GameStateManager.__init__`?

**Problem.** Umbrella §5.3 specifies the transition lifecycle `exit → mutate → enter → emit` for `transition_to`. The constructor's behavior is unspecified. If the initial state's `enter()` does NOT fire in `__init__`, then:
- The very first transition runs `current_state.exit()` on a state that was never entered — semantically wrong.
- Or: subscribers connected before the first transition never observe an `enter()` event for the starting state.

If the initial state's `enter()` DOES fire in `__init__`, then:
- The "set up subscribers before constructing the manager" call ordering becomes load-bearing — wire up the bridge BEFORE constructing the GSM, or the initial `state_changed` is missed.
- But §5.3 says `state_changed` fires only on `transition_to`, not on construction — so even if `enter()` runs in `__init__`, no `state_changed` is emitted, and subscribers don't see the initial state via the event channel.

**Option A — Initial state's `enter()` runs in `__init__`, no `state_changed`** (recommended).
- Lifecycle invariant: every state observed as `current_state` has had `enter()` called and not yet had `exit()` called.
- No event emitted for construction — subscribers query `manager.current_state` directly at startup if they care.
- **Partial alignment with `python-statemachine` 3.1.2.** `python-statemachine` fires both initial `on_enter` AND a synthetic `__initial__` event that triggers `after_transition` callbacks. This spec fires initial `on_enter` but deliberately suppresses the event emission — preserves `StateChangedPayload.from_state: type[GameState]` strict typing without widening to `type[GameState] | None` for the one-time bootstrap edge. `transitions` (pytransitions) takes the opposite stance ("`on_enter_«state»` callback will not fire when a Machine is first initialized"); the spec's choice is closer to `python-statemachine`'s entry behavior but consciously diverges on event emission.

**Option B — `enter()` does NOT fire in `__init__`; first `transition_to` is special-cased to skip the initial state's `exit()`**.
- Cleaner symmetry: `enter()` always pairs with `exit()`, and both pair with a `state_changed` event.
- But: the special-case-on-first-transition logic is awkward. Easier to forget. The "every observed state has been entered" invariant is the more useful one.

**Option C — Initial state is `None`; caller must `transition_to(MainMenuState())` to bootstrap**.
- `current_state: GameState | None`. Every reader has to handle `None`. Pollutes the type system for one moment in the lifecycle.
- **Rejected** — bad ergonomics for a one-time bootstrap.

**Recommendation:** **Option A**. `__init__(initial_state: GameState)` calls `initial_state.enter()` after assignment. No event. The "subscribers see the new state" invariant becomes: "subscribers connected before any `transition_to` see the post-transition state in their handler".

**Spec phrasing:** "the initial state's `enter()` fires synchronously inside `__init__`, but no `state_changed` event is emitted for construction — `state_changed` is reserved for transitions. The lifecycle invariant is: every state visible as `current_state` has had `enter()` called exactly once and has not yet had `exit()` called."

### Q3: What goes in v0.1 `MainMenuState.enter()` / `SectorMapState.enter()` bodies?

**Problem.** Per umbrella §5.8, `enter()` is "a state-machine hook for view/UI cueing, not a constructor — it does not instantiate `SectorMap`". v0.1 has no UI cueing yet (step 8 is pre-view); so what literally goes inside the method bodies?

**Option A — `pass`** (recommended).
- Honest: there is nothing for the model to do at state entry/exit in v0.1.
- 100% line coverage is trivially satisfied; 100% branch coverage is trivially satisfied (no branches).
- View-side cueing will land later as either: (a) a v0.2+ amendment overriding `enter()` to do UI cueing through the controller bridge — but per §5.8 the *model* does not do UI cueing; cueing happens via the `state_changed` event, which the bridge converts to a Qt signal. So `enter()` body content may never grow — it may remain `pass` permanently for these two states.

**Option B — `loguru.logger.debug(f"Entering {type(self).__name__}")`**.
- Useful for trace-debugging in dev; matches model-layer logging convention (invariant 6).
- Cost: adds a `loguru` import to `states.py`, which was a pure-stdlib module after step 5. Crosses a runtime-deps boundary for marginal value (the `state_changed` event already gives observers a hook).
- **Defer.** If trace logging is wanted, add it once and globally in `GameStateManager.transition_to` rather than peppering every state class.

**Option C — `raise NotImplementedError`**.
- Forces every subclass override. But `MainMenuState` and `SectorMapState` ARE the concrete classes — they have nothing to delegate to. **Wrong tool.**

**Recommendation:** **Option A** — `pass` bodies for both `enter()` and `exit()` on both `MainMenuState` and `SectorMapState`. Add a one-line comment naming §5.8 to explain why the body is empty (forestalls a future reviewer adding "TODO: cueing here" without checking the spec).

### Q4: Defensive type checking on `transition_to(target)` argument

**Problem.** Umbrella §5.3 step 1 says "validate that `type(target_state)` is in `current_state.allowed_transitions`". But what if `target` is not a `GameState` at all — e.g., a caller passes a class, or a `None`, or an arbitrary object?

**Option A — Trust the type annotation; rely on `isinstance` only in the validation predicate** (recommended).
- `transition_to(target: GameState) -> None`. mypy-strict catches mis-typed callers at static-analysis time.
- The validation step (`type(target) in current_state.allowed_transitions`) implicitly rejects non-`GameState` instances — `type(some_other_object)` is unlikely to be in the frozenset.
- Matches step-7 `Starship` / `TurnManager` discipline: tight validation on construction kwargs (because user-supplied at session start), looser on internal method args (because they pass through typed plumbing).

**Option B — Eager `isinstance(target, GameState)` check at entry, raise `TypeError`**.
- Defensive against dynamic-typed callers.
- But: the existing `IllegalTransitionError` covers "this transition is not allowed". A separate `TypeError` for "argument is not a GameState" is a different failure mode that callers can already prevent by reading the type signature.
- **Reject.** Adds runtime cost and a second exception type for a footgun that the type system already catches.

**Option C — `if not isinstance(target, GameState): raise IllegalTransitionError(...)`**.
- Lumps wrong-type and wrong-target into one error. Caller can't distinguish.
- The error message would have to be carefully phrased ("target must be a GameState; got <type>"). Adds branching for a non-issue under mypy-strict.

**Recommendation:** **Option A**. Trust mypy-strict at the API boundary. The validation step rejects anything whose `type(...)` is not in the allowed set; that's sufficient.

### Q5: Self-transition semantics

**Problem.** What happens if a caller does `manager.transition_to(MainMenuState())` while `manager.current_state` is already an instance of `MainMenuState`?

**Composition check:** with `MainMenuState.allowed_transitions = frozenset({SectorMapState})` (does NOT include `MainMenuState` itself), the validation step raises `IllegalTransitionError`. Self-transitions are forbidden by the composition of the allowed sets, not by an explicit special case.

**Should there be an explicit special case?** Comparison points:
- `transitions` (pytransitions): self-transitions DO fire `on_exit` / `on_enter` by default (it's a re-entry).
- `python-statemachine`: configurable via `enable_self_transition_entries` flag; default `False` (no exit/enter on self-transitions per SCXML legacy default).

For v0.1, the question is moot — `MainMenuState` cannot transition to `MainMenuState` because it's not in the allowed set. The behavior falls out of the data, not the code.

**Recommendation:** **No special-case logic.** The allowed-transitions data is the single source of truth. If v0.2 wants to allow self-transitions for a particular state, that state's `allowed_transitions` includes its own class; the existing `validate → exit → mutate → enter → emit` lifecycle then runs (re-entering the same instance). No code change in GSM.

**Test discipline:** include a test asserting `MainMenuState → MainMenuState` raises `IllegalTransitionError`. Pins the behavior so a future "helpful" PR adding `cls` to the frozenset doesn't slip through.

### Q6: Identity check on transition target (short-circuit `if current_state is target`?)

**Problem.** What if a caller does `manager.transition_to(manager.current_state)` — passes the *exact same instance*?

This is distinct from Q5 (which is "same type, different instance"). Here, the instance identity matches.

**Option A — No special case; treat like any other transition** (recommended).
- The validation step checks `type(target) in current_state.allowed_transitions`. Since `current_state.allowed_transitions` does NOT include the current state's own class (per Q5), validation rejects this with `IllegalTransitionError`.
- Falls out of the data — no `is` check needed.

**Option B — Short-circuit: if `current_state is target`, return without effect**.
- Surprising silent no-op. Caller may have invoked `transition_to` expecting side-effects (event emission, lifecycle hooks); silently skipping them is a footgun.
- **Reject.**

**Option C — Short-circuit: if `current_state is target`, raise `IllegalTransitionError` with a specific "self-identity" message**.
- Distinguishes "you passed the wrong object" from "this transition is not in your allowed set".
- Adds one branch for a footgun that mypy-strict doesn't catch but composition (per Option A) handles cleanly.

**Recommendation:** **Option A.** Composition handles it. No identity branching.

### Q7: Test pattern for invariant 9 ("subscribers see post-transition state")

**Problem.** Umbrella invariant 9 (also umbrella §5.3): "Subscribers reading `manager.current_state` inside a `state_changed` handler see the new state." This is an *observable* invariant that the spec must include a test for.

**Recommended test pattern** (matches the established try/finally idiom in `tests/unit/model/entities/test_starship.py` and `tests/unit/model/combat/test_turn_manager.py` — no shared fixture; each test connects and disconnects its own receiver):
```python
def test_state_changed_handler_sees_new_state() -> None:
    captured: list[type[GameState]] = []
    initial = MainMenuState()
    manager = GameStateManager(initial)

    def handler(sender, payload):
        captured.append(type(manager.current_state))

    state_changed.connect(handler)
    try:
        manager.transition_to(SectorMapState())
    finally:
        state_changed.disconnect(handler)

    assert captured == [SectorMapState]
```
The handler reads `manager.current_state` (not `payload.to_state`) to specifically prove the mutation order — `mutate` ran before `emit`. If the order regresses (emit before mutate), `captured == [MainMenuState]` and the test fails.

**Coverage:** must run both directions (MainMenu → SectorMap AND SectorMap → MainMenu) to catch a one-way regression.

**No central fixture for signal isolation exists in v0.1** (no `tests/conftest.py` at the project root). The try/finally idiom above is the established pattern; the spec inherits it rather than introducing a new fixture. If a future session needs cross-test signal isolation, the fixture can be added once and consumed by all signal-touching tests; out of scope for step 8.

### Q8: Behavior when receiver raises during `state_changed.send()`

**Problem.** blinker's `send()` propagates the first receiver exception synchronously and skips remaining receivers. The state mutation (current_state assignment, prior `enter()` call) has already committed. So a buggy receiver leaves the manager in an inconsistent observable state from the caller's perspective: the transition "succeeded" but the caller sees an exception.

**Umbrella context.** Step-7 spec §9 / §9.2 already locked the mitigation strategy for this risk (synchronous-emit fault isolation): deferred to v0.2 when a second non-bridge subscriber lands. The trigger condition is the *second subscriber* on any signal.

**Step-8 inheritance.** `state_changed` is the fourth signal to land. v0.1 has exactly one subscriber per signal (controller bridge). No new mitigation work in step 8 — inherit the deferral.

**Test discipline:** include one explicit test asserting that `state_changed.send` propagates a receiver exception (pins the documented behavior), so a future change in blinker's contract or a "helpful" local wrapper that swallows exceptions is caught:
```python
def test_state_changed_receiver_exception_propagates() -> None:
    def bad_receiver(sender, payload):
        raise RuntimeError("subscriber blew up")
    manager = GameStateManager(MainMenuState())
    state_changed.connect(bad_receiver)
    try:
        with pytest.raises(RuntimeError, match="subscriber blew up"):
            manager.transition_to(SectorMapState())
    finally:
        state_changed.disconnect(bad_receiver)
    # Post-condition: transition still committed despite receiver failure
    assert type(manager.current_state) is SectorMapState
```
This is a *defensive* test in the same sense as step 7's `test_advance_turn_inactive_player_starship_succeeds` — it pins documented current behavior so that v0.2's `_send_robust` migration is a conscious break, not an accidental one.

## Locked-decisions draft (for spec §9.1)

1. **Q1: Post-class module-level patch.** `MainMenuState.allowed_transitions = frozenset({SectorMapState})` and vice versa, with inline comment naming the patch site. Smoke tests pin the patched values.
2. **Q2: Initial-state `enter()` fires in `__init__`; no `state_changed` for construction.** Lifecycle invariant: every observed `current_state` has been entered and not yet exited.
3. **Q3: Empty `pass` bodies** for `MainMenuState.enter/exit` and `SectorMapState.enter/exit` in v0.1. Comment naming §5.8.
4. **Q4: No defensive `isinstance(target, GameState)` check** in `transition_to`. Type system at the boundary + validation step handle it.
5. **Q5: No special-case self-transition logic.** Allowed-transitions data is the source of truth. Test pins `MainMenu → MainMenu` raises `IllegalTransitionError`.
6. **Q6: No identity short-circuit** on `current_state is target`. Composition handles it.
7. **Q7: Test pattern fixed** as shown in Q7 (handler reads `manager.current_state` to prove mutation order, run both directions).
8. **Q8: Receiver-exception behavior inherited from umbrella §9** (synchronous propagation; mitigation deferred). One defensive test pins current behavior.

## Still-open questions (for spec §9.2)

1. **What does `loguru` trace logging look like in `transition_to`?** Optional; could be added as a single `logger.debug(f"{type(from_state).__name__} → {type(to_state).__name__}")` line. v0.2 nice-to-have; v0.1 may ship without it.
2. **Does `GameStateManager` need a `__repr__`?** Useful for `pytest` failure messages. Cheap to add; spec should call it out one way or the other.

## Risks

- **Patch-site fragility.** A reviewer who deletes the post-class assignment lines silently reverts the allowed-transitions sets to empty frozensets. The smoke tests in Q1 catch this. Spec must require the smoke tests.
- **`enter()` running in `__init__` is asymmetric.** It's the one path where `enter()` fires without `state_changed`. Code reviewers may "fix" the asymmetry by adding the emit. The spec must explicitly name this asymmetry and forbid the emit.
- **State count growth past ~7 nodes.** ADR-0005 already covers this. v0.1 has 2 states; well below the revisit threshold.
