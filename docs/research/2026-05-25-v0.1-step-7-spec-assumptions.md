## A. `_debit_ap(cost)` — raise-before-mutate is the correct order

### Finding

Raise-before-mutate is universally established as the correct order for precondition helpers in Python game and transaction systems. The pattern is:

```python
def _debit_ap(self, cost: int) -> None:
    if self.ap_remaining < cost:
        raise InsufficientAPError(...)
    self.ap_remaining -= cost
```

The inverse pattern (debit then check for negative) is documented as broken:

```python
# BROKEN — do not use
self.ap_remaining -= cost
if self.ap_remaining < 0:
    self.ap_remaining += cost  # partial-rollback is fragile
    raise InsufficientAPError(...)
```

The inverse requires a compensating rollback that is not atomic under exception unwind and is invisible to any concurrency or re-entrant code path. Even in single-threaded Python, the window between the debit and the raise is observable to any debugger or assertion.

### Supporting evidence

- The tcod v2 roguelike tutorial (official tutorial source for Python roguelike development) implements all action preconditions as guard-first: the action method checks `if not self.entity.active:` etc. before any state mutation. The guard fires an exception; mutation follows. [community] (https://rogueliketutorials.com/tutorials/tcod/v2/part-6/)
- esper ECS and tcod-ecs both document that component read/write happens after system precondition checks; they do not expose a "debit with rollback" primitive — check-and-debit is the whole primitive. [community] (https://github.com/HexDecimal/python-tcod-ecs) (https://github.com/benmoran56/esper)
- Python stdlib convention (`dict.__delitem__`, `list.pop`, `int.__sub__`) uniformly raises before mutation when the precondition is knowable from arguments alone. [official] (https://docs.python.org/3/library/stdtypes.html)
- Prior research doc §3 for this project (2026-05-23) corroborates this as the established ECS pattern for dict-backed containers. [community] (docs/research/2026-05-23-python314-idioms-game-model-layer.md)

### Verdict for the spec

Spec is correct: raise `InsufficientAPError` before the debit, never after. The helper body is `if self.ap_remaining < cost: raise ... ; self.ap_remaining -= cost`. No footgun identified with the raise-before-mutate form. AP cannot go negative under normal usage — exception unwind leaves `ap_remaining` unchanged.

## B. Zero-move policy — `NotAdjacentError` is the correct response

### Finding

The roguelike and 4X community convention is that "move to current cell" (Chebyshev distance = 0) is an **invalid move command, not a silent no-op.** The dominant implementation is that the adjacency check (`distance == 1`) fails, and the action raises an error (or the UI simply does not allow the player to select their own cell as a move target).

Silent no-ops are used in some engine-level internal paths (e.g., a pathfinder that reaches its destination early), but at the **action-method boundary** — the layer the spec is defining — a zero-distance move is considered programmer error or a UI bug, not a valid player input to silently absorb.

A separate `NoOpMoveError` is not a convention in the literature surveyed; no roguelike tutorial or ECS library introduces such a type. The conventional choices are:
1. Raise `NotAdjacentError` (or its equivalent) — the spec's current choice.
2. Raise `ValueError` if distance == 0 as a distinct pre-check before the adjacency check.
3. Silent no-op, but this is the minority pattern and is associated with internal-engine paths, not action-method preconditions.

### Supporting evidence

- Rogueliketutorials.com tcod v2 tutorial: the `BumpAction` handler filters out the move-to-self case by checking that destination differs from current position before dispatching `MoveAction`. The action itself is never called with the player's own position; if it were, the position comparison would fail at the movement system level. [community] (https://rogueliketutorials.com/tutorials/tcod/v2/part-5/)
- RogueBasin Entity Component System article: "movement actions require adjacent-or-in-range targets; out-of-range is an action failure." No special casing for zero-distance. [community] (https://www.roguebasin.com/index.php/Entity_Component_System)
- PEP 544 and Python typing docs confirm that `Protocol`-based dispatch is structural; the distance check is the precondition regardless of the source. [official] (https://peps.python.org/pep-0544/)

### Verdict for the spec

`NotAdjacentError` for a zero-distance move is correct — Chebyshev distance 0 fails `is_adjacent()` (`distance == 1`), which is the defined adjacency predicate. The spec does not need a new `NoOpMoveError` type. If the UI ever needs to distinguish "moved to own cell" from "moved to a non-adjacent cell", it can catch `NotAdjacentError` and inspect `target == self.position`, but that is a v0.2 UI concern, not a v0.1 model concern.

One optional defensive addition (not required): a `NotAdjacentError` message could include whether the distance was 0 or > 1 — this aids debugging at no semantic cost.

## C. `isinstance(target, Station)` vs `_DockTarget: Protocol` asymmetry

### Finding

The asymmetry is defensible and is the pattern the Python typing community endorses for runtime dispatch. The official guidance (PEP 544, mypy docs, typing.python.org) is:

- Use `Protocol` when you control neither the caller nor the callee, or when you want open extensibility without import coupling.
- Use nominal `isinstance` when you own the class hierarchy and need type-narrowing to expose class-specific attributes after the check.

In `Starship.dock_at`, after `isinstance(target, Station)` passes, the code accesses `target.accepts_dock(self)` — a `Station`-specific method. A `Protocol` check only narrows to the protocol's declared surface; if `accepts_dock` is not on the protocol, mypy cannot prove it is safe to call after the check. The `Protocol` approach would require a `_DockTarget` that declares `accepts_dock`, `position`, and `active`, making it functionally equivalent to a nominal superclass with no extensibility benefit.

The step-6 design (using `_Dockable` in `accepts_dock`) is Protocol-appropriate because that method was written before `Starship` existed and uses only the minimum structural contract (`active: bool`). The step-7 direction (`isinstance(target, Station)`) is nominal-appropriate because `TurnManager`/`Starship` logic needs `Station`-specific behaviour, `Station` is owned in-project, and no third-party extension is anticipated for v0.1.

**`@runtime_checkable` Protocol + isinstance caveat:** Python docs and mypy docs both warn that `isinstance()` with a `@runtime_checkable` Protocol only checks attribute *existence*, not signature or type. This is a known footgun. Nominal `isinstance(target, Station)` is more reliable for runtime dispatch because it uses CPython's MRO-based check and cannot pass for a wrong-type object that happens to have an `accepts_dock` attribute. [official] (https://typing.python.org/en/latest/reference/protocols.html) [official] (https://mypy.readthedocs.io/en/stable/protocols.html)

### Verdict for the spec

No spec change needed. `isinstance(target, Station)` in `Starship.dock_at` is the right choice:
- `Station` is an owned class; nominal typing is correct.
- Protocol `isinstance` (even with `@runtime_checkable`) does not check method signatures, creating a false-safety trap.
- Introducing `_DockTarget: Protocol` would duplicate the Station interface and break when `Station` adds methods — a maintenance cost with no v0.1 benefit.

The asymmetry (Protocol for `accepts_dock(ship: _Dockable)`, nominal for `isinstance(target, Station)`) is intentional and correct. Each uses the appropriate tool for its direction.

## D. `SectorMap` import under `TYPE_CHECKING` — confirmed safe

### Finding

`from __future__ import annotations` (PEP 563) and Python 3.14 PEP 649 deferred-annotation mode both cause all annotations to be stored as strings rather than evaluated at module-load time. Under either regime:

- A name used **only in annotations** (method parameter types, return types) does not need to be importable at runtime.
- Placing the import under `if TYPE_CHECKING:` is the canonical pattern for annotation-only dependencies. [official] (https://mypy.readthedocs.io/en/stable/runtime_troubles.html)
- mypy `--strict` resolves `TYPE_CHECKING`-guarded imports during static analysis, so the import is visible to the type checker even though it never runs at runtime.

**The critical constraint is that the symbol must NOT be used in any runtime context** — specifically not in `isinstance()`, not as a default argument value, not in a decorator. If it appears only in annotations (parameter types, return types), the `TYPE_CHECKING`-only import is safe and idiomatic.

For `starship.py`, `SectorMap` and `Station` appear in method signatures only (`move_to(target, sector_map: SectorMap)`, `dock_at(station_id, sector_map: SectorMap)`). But `isinstance(target, Station)` at runtime requires `Station` to be importable at runtime — it cannot be under `TYPE_CHECKING` only.

For `turn_manager.py`, `isinstance(player, Starship)` requires `Starship` to be a runtime import. `SectorMap` appears only as a method parameter type, so it can be `TYPE_CHECKING`-only.

The umbrella spec §6 already captures this distinction correctly:
- `entities.starship` → runtime import in `combat.turn_manager` (needed for `isinstance`)
- `world.sector_map` → `TYPE_CHECKING`-only in `entities.starship` (annotation-only)
- `entities.station` → `TYPE_CHECKING`-only in `entities.starship` (annotation-only; `isinstance(target, Station)` uses the runtime import already present in `turn_manager`... wait — `dock_at` is on `Starship`, so `Station` must be a runtime import in `starship.py` for the `isinstance` check)

### Footgun identified — `entities.station` in `starship.py`

The umbrella §6 DAG lists `entities.station` as `TYPE_CHECKING`-only for `entities.starship`. But `Starship.dock_at` uses `isinstance(target, Station)` at runtime. **This is a contradiction**: `Station` must be a runtime import in `starship.py` to appear in `isinstance()`.

Two resolutions:
1. **Move to runtime import** — `from stmrr.model.entities.station import Station` as a top-level import in `starship.py`. No cycle: the import DAG shows `station` does not import `starship`, so this adds an edge `starship → station` which is acyclic.
2. **Lazy import inside the method** — `from stmrr.model.entities.station import Station` inside `dock_at`'s body. Avoids the module-level import cost; works for `isinstance`. Unconventional in mypy-strict codebases and creates a hidden runtime dependency.

Resolution 1 is the correct choice for this codebase. It is consistent with `turn_manager.py`'s runtime import of `Starship` for the same reason.

### Verdict for the spec

The umbrella §6 `TYPE_CHECKING`-only annotation for `entities.station` in `entities.starship` must be amended. `Station` needs to be a runtime import in `starship.py` because `dock_at` calls `isinstance(target, Station)`. The spec should reflect this: add `entities.station` to the runtime imports column for `entities.starship`, and update the DAG prose in §6 accordingly. No cycle is introduced.

For `SectorMap` in both files: `TYPE_CHECKING`-only is correct and safe since it appears only in annotations, never in `isinstance()` or other runtime expressions.

## E. `TurnManager.__init__` validate-before-super() — inapplicable

### Finding

The validate-before-`super().__init__()` invariant is specific to the `GameObject` subclass chain, where `GameObject.__init__` allocates a globally unique `EntityId` from a monotone counter. The invariant exists to prevent consuming an ID from a construction that will subsequently fail, leaving a gap in the ID sequence.

`TurnManager` is a plain class with no superclass with side effects. Its `__init__` would call `object.__init__()`, which has no side effects whatsoever — it simply returns. There is nothing to delay.

The correct Python idiom for "validate first" in a plain class (no meaningful superclass) is:

```python
def __init__(self, player_id: EntityId, *, current_turn: int = 1) -> None:
    # Validate top to bottom
    if not isinstance(player_id, int) or isinstance(player_id, bool):
        raise TypeError(...)
    if current_turn < 1:
        raise ValueError(f"current_turn must be >= 1, got {current_turn}")
    # Then assign
    self.player_id = player_id
    self.current_turn = current_turn
```

No `super().__init__()` call is needed (Python's implicit MRO resolution handles `object.__init__` automatically for simple single-inheritance). If `super().__init__()` is called explicitly for clarity, it comes after validation because there is nothing to delay — `object.__init__()` always succeeds and always has zero side effects.

### Supporting evidence

- Python OOP idiom: validate inputs before assigning attributes, regardless of superclass position. The only reason to delay validation relative to `super().__init__()` is when the superclass has observable side effects that must be avoided on failure. [official] (https://docs.python.org/3/reference/datamodel.html#object.__init__)
- The step-6 spec §5.2 explains the specific motivation for validate-before-`super()` in `Station.__init__`: the `EntityId` counter side effect. The commentary is explicit that this pattern is motivated by `GameObject.__init__`'s counter, not by a general rule. [official project spec] (docs/specs/v0.1-step-6-sector-map-and-station.md §5.2)
- StackOverflow consensus: "validate before assigning" is the Python idiom; "validate before `super().__init__()`" is only relevant when the super's `__init__` has side effects. [community] (https://stackoverflow.com/questions/5133262/python-abstract-base-class-init-initializion-or-validation)

### Verdict for the spec

`TurnManager.__init__` should validate top-to-bottom and assign. There is no `EntityId` counter at stake. The spec should note that validate-before-`super().__init__()` is a `GameObject`-subclass-specific invariant, not a model-layer-wide rule. For `TurnManager`, plain top-to-bottom validation is the correct and conventional approach.

## F. `ship_class` as open `str` vs Literal — stay open `str` in v0.1

### Finding

The wide-Literal pattern used for `Station.station_type` is warranted when the v0.1-to-v0.2 save-format migration cost is significant (i.e., when v0.1 files with only `"starbase"` will be read by v0.2 code that introduces `"civilian"`, `"military"`, `"neutral"`). The field must be in the save format from day one with the wide Literal so v0.1 saves are forward-compatible.

For `Starship.ship_class`, the analogous question is: will a v0.1 save file containing `"constitution"` be read by v0.2 code that adds other ship classes? Yes — but the difference from `station_type` is:

1. `ship_class` is not a discriminator for a closed-universe set of subtypes. Every ship in v0.1 is a `Starship` regardless of class. The value is metadata, not a type tag.
2. The set of valid ship classes (`constitution`, `miranda`, `oberth`, etc.) is open-ended by design (`DESIGN.md` §5.1 lists several but does not close the set). A `Literal` over a hand-enumerated list would need to be widened with every new class, creating a migration concern on every v0.2+ addition.
3. Open `str` with a runtime registry lookup is the conventional approach in ECS and RPG codebases for this kind of "entity subtype tag" that is data-driven rather than code-driven.

The Python community consensus on `Literal` vs open `str`:
- `Literal` is best for closed sets that will not grow frequently and where exhaustiveness checking is valuable (e.g., `station_type`, HTTP methods, event names). [official] (https://mypy.readthedocs.io/en/stable/literal_types.html)
- Open `str` with a runtime registry is the standard for data-driven type tags in ECS/RPG codebases (esper, tcod-ecs, Godot GDScript, pydantic discriminated unions with `str` discriminators). [community] (https://github.com/benmoran56/esper) (https://reddit.com/r/Python/comments/1g52103/)
- The r/Python discussion on "why frameworks use strings instead of enums" (225 votes, Oct 2024) shows strong community preference for `Literal` over `Enum` for small closed sets, but open `str` for extensible registries. [community] (https://reddit.com/r/Python/comments/1g52103/)

### Verdict for the spec

`ship_class: str` (open string) is correct for v0.1. Do not introduce a `ShipClassLiteral` in v0.1. Reasons:
- The set is open-ended; a Literal would fight every future addition.
- No exhaustiveness checking benefit — there is no `match ship.ship_class:` dispatch in v0.1.
- A v0.2 ship-class registry (mapping string → class stats) is the natural extension point.

The spec should document that `ship_class` accepts any non-empty string in v0.1 (only validate non-emptiness and non-bool-int) and that a registry-based validation arrives in v0.2.

## G. `advance_turn` emit-after-mutation — blinker exception propagation is a real footgun

### Finding

**blinker official documentation (1.9, current stable)** states explicitly:

> "If a receiver raises an exception, that exception will propagate up. This makes debugging straightforward, with an assumption that correctly implemented receivers will not raise."

This means: if `turn_advanced.send(self, payload=...)` is called and any receiver raises, the exception propagates **immediately** out of `send()`. Receivers connected after the raising one are **never called** (blinker iterates receivers in insertion order and stops on first exception). The `advance_turn` method exits with an exception, but the mutation (AP restored, `current_turn` incremented) has already happened.

**Django signals docs** (which use blinker-like semantics) confirm the identical behaviour and explicitly provide `send_robust()` as the alternative:

> "`send()` does not catch any exceptions raised by receivers; it simply allows errors to propagate. Thus not all receivers may be notified of a signal in the face of an error."

[official Django docs] (https://docs.djangoproject.com/en/6.0/topics/signals/)

**Implication for the spec:** If the bridge (or any future test subscriber) raises in a `turn_advanced` handler, `TurnManager.advance_turn` will exit with an exception even though `current_turn` was incremented and AP was restored. On the next call, `advance_turn` will double-increment `current_turn`.

**v0.1 mitigation analysis:**
- In v0.1, `controller/model_bridge.py` is the only subscriber to `turn_advanced`. The bridge is expected to be exception-safe (it only translates model events to Qt signals; Qt signal emission does not raise).
- The spec's umbrella §5.2 already locks "delivery is synchronous for v0.1" and "receiver invocation order is undefined" — these are the relevant invariants.
- Adding `try/except` around `send()` in `advance_turn` to catch receiver exceptions and log them would provide fault isolation at the cost of swallowing errors silently, which is worse for v0.1 debugging.
- The correct v0.1 choice is: **no try/except; document that receivers must not raise**. This matches blinker's documented design intent.

**v0.2 concern:** If v0.2 adds multiple subscribers (a replay logger, an AI system notifier), a misbehaving subscriber can corrupt `TurnManager` state. The spec should note this as an open risk and flag `send_robust()` as the v0.2 migration path.

### Verdict for the spec

The current spec (emit after mutation, no try/except) is correct for v0.1. The risk is real but acceptable because v0.1 has a single known-safe subscriber. The spec should add:
- A comment in `advance_turn` noting that receiver exceptions propagate and corrupt turn state.
- A note in §9 (Risks) that v0.2 must evaluate `send_robust()` or structured error handling when multiple subscribers are added.

## H. `hull` field semantics — accept `>= 1` at construction

### Finding

The tcod v2 official tutorial establishes the canonical pattern for HP fields in Python game entities:

```python
class Fighter(BaseComponent):
    def __init__(self, hp: int, defense: int, power: int):
        self.max_hp = hp
        self._hp = hp

    @hp.setter
    def hp(self, value: int) -> None:
        self._hp = max(0, min(value, self.max_hp))
        if self._hp == 0 and self.entity.ai:
            self.die()
```

Key observations:
1. The constructor accepts the **maximum HP** as the initial value. HP is set to max at construction — an entity starts alive with full HP.
2. The lower bound `0` is enforced by the **setter**, not the constructor. The constructor does not validate that `hp > 0` — it trusts the caller to not construct a dead entity.
3. "A hull-0 entity is dead" — triggering the `die()` method when HP reaches 0 — is the universal pattern. A hull-0 entity constructed intentionally would be an immediate death; this is never done in practice.

For `Starship.hull` in v0.1 (pre-combat, no damage math):
- The constructor should validate `hull >= 1` (reject 0 and negative). Constructing a Starship with 0 hull would represent an already-dead ship, which is semantically incoherent in v0.1 where hull is a placeholder with no damage system.
- The lower bound `> 0` at construction is the safe, defensive choice. It aligns with "hull represents the ship's structural integrity; a ship with zero hull does not exist."
- The lower bound is NOT `>= 0` because a ship with `hull=0` is inactive/dead and should not be constructible in v0.1. `GameObject.active` tracks liveness; hull = 0 would conflict with `active = True` (the construction default).

### Supporting evidence

- tcod v2 tutorial Part 6 (official reference for Python roguelike combat): max_hp set at construction, lower bound enforced by setter at 0. No constructor validation that `hp > 0`, but in practice no entity is constructed with hp=0. [community] (https://rogueliketutorials.com/tutorials/tcod/v2/part-6/)
- Common sense game design: a ship that is already destroyed at construction is a programmer error. `hull >= 1` is the correct invariant.
- The `active: bool = True` default on `GameObject` (from step-3 spec) means every freshly constructed entity starts active. A hull-0 entity would be inconsistent with `active = True`.

### Verdict for the spec

`hull >= 1` at construction time. Validate with `ValueError("hull must be >= 1, got {hull}")`. A `hull=0` entity is semantically incoherent in v0.1 (active + dead simultaneously). When v0.2 adds `CombatResolver`, it will subtract from `hull` via a method that handles the `hull <= 0` → deactivate transition, exactly as the tcod tutorial's `@hp.setter` does. The v0.1 model should make it impossible to construct an already-dead ship.

## Footguns and Gotchas

- **blinker `send()` propagates receiver exceptions immediately, skipping remaining receivers** — if the bridge raises during `turn_advanced`, `current_turn` is already incremented. State is permanently desynchronised. Corroborated by: blinker official docs 1.9, Django signals docs. Both confirm identical propagation semantics. [official] (https://blinker.readthedocs.io/en/stable/) [official] (https://docs.djangoproject.com/en/6.0/topics/signals/)

- **`isinstance(target, Station)` requires a runtime import of `Station` in `starship.py`** — the umbrella §6 DAG currently marks `entities.station` as `TYPE_CHECKING`-only for `entities.starship`, which contradicts the `isinstance()` call in `dock_at`. A runtime import is required. Corroborated by: mypy runtime_troubles docs, Python import system docs. [official] (https://mypy.readthedocs.io/en/stable/runtime_troubles.html)

- **`runtime_checkable` Protocol `isinstance()` only checks attribute existence, not types or signatures** — using a Protocol for the dock-target check would allow any object with a `accepts_dock` attribute to pass, regardless of its actual type. Nominal `isinstance(target, Station)` is safer. Corroborated by: typing.python.org reference, mypy protocols docs. [official] (https://typing.python.org/en/latest/reference/protocols.html)

- **Debit-before-raise in `_debit_ap`** — `self.ap_remaining -= cost` followed by a check is the broken order. The `if` check must precede the debit. AP going negative under this antipattern is silent and only caught if there is a downstream assertion. Corroborated by: tcod tutorial action pattern, prior research §3. [community] (https://rogueliketutorials.com/tutorials/tcod/v2/) (docs/research/2026-05-23-python314-idioms-game-model-layer.md)
