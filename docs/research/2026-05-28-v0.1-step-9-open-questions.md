# Research: v0.1 Step-9 Open Questions

Mode: research  ·  Topic: v0.1 step-9 open questions (controller/model_bridge.py — blinker-to-Qt bridge)  ·  Saved: docs/research/2026-05-28-v0.1-step-9-open-questions.md

> **Scope.** Step 9 ships `controller/model_bridge.py` — the first Qt-touching module in the
> codebase. It subscribes to the four blinker signals (`ship_moved`, `docked`, `turn_advanced`,
> `state_changed`) from `model/events.py` and re-emits them as PySide6 `Signal`s so the Qt view
> can connect to them. The model layer stays Qt-free (ADR-0003); the bridge is the single seam.

## Official Documentation

- `Signal` must be declared as a class-level attribute on a `QObject` subclass. An unbound `Signal` (class attribute) has no `.connect`/`.emit`; the bound `SignalInstance` is created per-instance by the descriptor protocol. Attempting to call `.emit()` on the class attribute produces `AttributeError`. [official] (https://doc.qt.io/qtforpython-6/PySide6/QtCore/Signal.html)

- `Signal(object)` is the established PySide6 idiom for a signal that can carry any Python value. The type argument `object` matches Python's `object` base class and is accepted for any Python object (including `None` and frozen dataclasses). The signature `finished = Signal(object)` is used verbatim in official Qt-for-Python threading examples. [official] (https://www.pythonguis.com/tutorials/multithreading-pyside6-applications-qthreadpool/) [official] (https://doc.qt.io/qtforpython-6/overviews/qtdoc-signalsandslots-syntaxes.html)

- `@Slot(object)` decoration on a slot receiving a `Signal(object)` is optional in PySide6 (unlike PyQt6 where it affects queued-connection dispatch). Without `@Slot`, PySide6 still invokes the callable. With `@Slot(object)`, PySide6 registers the C++ slot signature and provides slightly better introspection; recommended for any slot connected to `Signal(object)`. [official] (https://doc.qt.io/qtforpython-6/PySide6/QtCore/Slot.html)

- `QObject.__init__` must be called with `parent=None` or a `QObject` parent. Skipping `super().__init__()` or calling it after signal declarations causes shiboken to fail silently. [official] (https://doc.qt.io/qtforpython-6/PySide6/QtCore/QObject.html)

- blinker `Signal.connect(receiver, weak=True)` (default). The `weak` parameter controls whether the signal holds a `weakref` to the receiver. "The receiver will be automatically disconnected when it is garbage collected." The docs explicitly warn: "When connecting a receiver defined within a function, set `weak=False`, otherwise it will be disconnected when the function scope ends." This warning extends to bound methods: if the object holding the method has no other strong references outside the signal, the weakref expires and the connection is silently dropped. [official] (https://blinker.readthedocs.io/en/latest/)

- blinker `receivers_for(sender)` — "Yield each receiver to be called for `sender`. Weakly referenced receivers that are not live will be disconnected and skipped." This is the foundation of the hand-rolled `_send_robust` utility deferred in the umbrella spec. [official] (https://blinker.readthedocs.io/en/latest/)

- PySide6 6.5.2+ automatically captures Python exceptions raised inside the Qt event loop and re-raises them when control returns to Python. **This mechanism does NOT apply to blinker callbacks**, which fire synchronously inside model method calls, outside the Qt event loop. Exception propagation for blinker callbacks is governed by blinker's `send()` contract, not Qt's. [official] (https://pytest-qt.readthedocs.io/en/latest/virtual_methods.html)

## Best Practices

### Q1: Single QObject vs one QObject per signal

**Pattern A — single `ModelBridge(QObject)` with all four Signals:**

```python
class ModelBridge(QObject):
    ship_moved = Signal(object)    # payload: ShipMovedPayload
    docked = Signal(object)        # payload: DockedPayload
    turn_advanced = Signal(object) # payload: TurnAdvancedPayload
    state_changed = Signal(object) # payload: StateChangedPayload

    def __init__(self, parent: QObject | None = None) -> None:
        super().__init__(parent)
        from stmrr.model import events as ev
        ev.ship_moved.connect(self._on_ship_moved, weak=False)
        ev.docked.connect(self._on_docked, weak=False)
        ev.turn_advanced.connect(self._on_turn_advanced, weak=False)
        ev.state_changed.connect(self._on_state_changed, weak=False)
```

This is the established PySide6 idiom, directly matching the `WorkerSignals(QObject)` pattern from Qt's official multithreading docs where one `QObject` holds all signals for a `QRunnable`. It gives the view a single object to hold and connect. Thread affinity is uniform (all four signals share the bridge's thread affinity). Testing is straightforward: one mock target. [community] (https://www.pythonguis.com/tutorials/multithreading-pyside6-applications-qthreadpool/)

**Pattern B — separate `QObject` per signal (e.g. `ShipMovedBridge(QObject)`):**

Fits rare cases where different signals need different thread affinities or different lifetimes. Not needed in v0.1 (all four signals are emitted from the same thread, the single model thread). The cost is four objects to wire up, track, and parent. Rejected for v0.1 because the complexity overhead is not justified by any v0.1 requirement.

**Recommendation:** Pattern A — single `ModelBridge(QObject)`. One object, uniform thread affinity, one ownership point, one parameter to `app.py` wiring. [community] (https://www.pythonguis.com/tutorials/multithreading-pyside6-applications-qthreadpool/)

### Q2: Signal naming convention and payload shape

Two sub-questions:

**2a: Qt Signal attribute names.** The blinker signal names (`ship_moved`, `docked`, `turn_advanced`, `state_changed`) are already `snake_case` past-tense — deliberately chosen in the umbrella spec (§5.2) so "the bridge translation is mechanical." The simplest mapping is one-to-one: `ModelBridge.ship_moved = Signal(object)` mirrors `events.ship_moved`. The umbrella spec explicitly calls this out as a design goal.

**2b: `Signal(object)` vs typed multi-arg signals.** Two options:

- `Signal(object)` — emit the entire frozen dataclass payload as a single opaque object. The slot receives `def on_ship_moved(self, payload: ShipMovedPayload) -> None`. The mypy signature is `(object) -> None` on the signal side (PySide6 stubs do not express per-instance payload typing for `Signal(object)`); the slot side can be annotated `@Slot(object)`.
- `Signal(int, int, int)` — unpack fields; e.g. `Signal(int, int, int)` for `ShipMovedPayload`. Slots receive individual primitive args. Type-safe at the Qt boundary, but loses the frozen-dataclass guarantee (callers cannot use `payload.ship_id`) and makes the bridge API a second layer of unpacking after the blinker send.

`Signal(object)` is the correct choice for this use case. The payload types contain `EntityId` (NewType of `int`), `GridPosition` (dataclass), `type[GameState]` (a class object) — none of these are expressible as primitive Qt signal types. The `Signal(object)` + single frozen dataclass approach matches what Qt's own threading examples use for `result = Signal(object)`. [official] (https://www.pythonguis.com/tutorials/multithreading-pyside6-applications-qthreadpool/)

**Mypy implications of `Signal(object)`:** PySide6 stubs type `Signal(object)` as accepting and emitting `object`. Slots receiving it should be annotated with the actual payload type; mypy will warn unless the slot is explicitly typed `(object) -> None` or the received value is narrowed with `assert isinstance(payload, ShipMovedPayload)`. The `pyside6-stubs` third-party package (pypi: PySide6-stubs) improves stub accuracy but still cannot express parametric signal payload types — this is a known gap. [community] (https://pypi.org/project/PySide6-stubs/)

**Recommendation:** `Signal(object)` carrying the frozen dataclass payload intact. One-to-one naming (`ModelBridge.ship_moved` maps to `events.ship_moved`). Slot annotations use the specific payload type (`ShipMovedPayload`) with a `@Slot(object)` decorator for Qt registration.

### Q3: Receiver error policy

**Context from the umbrella spec and prior research:**
- blinker `send()` propagates the first receiver exception synchronously and skips remaining receivers.
- After `blinker.send()` fires in the model (e.g. inside `Starship.move_to`), the bridge callback fires as the blinker receiver. Inside that callback, the bridge calls `self.ship_moved.emit(payload)`. This Qt `emit()` invokes connected Qt slots **synchronously** (DirectConnection, since v0.1 is single-threaded and all objects share the main-thread affinity).
- A raising Qt slot propagates up through `emit()` → bridge callback → blinker `send()` → model method call site.

**The two-layer propagation path:**
```
Starship.move_to()
  → blinker ship_moved.send()
    → ModelBridge._on_ship_moved()  # blinker receiver
      → self.ship_moved.emit(payload)  # Qt signal
        → view_slot()  # Qt slot (synchronous)
          ← raises RuntimeError("display broke")
        ← RuntimeError propagates through emit()
      ← RuntimeError exits bridge callback
    ← blinker sees exception; propagates out of send()
  ← RuntimeError propagates out of Starship.move_to()
```

**This is the same propagation chain documented in the step-8 research (Q8) for `state_changed`.** The bridge adds a second synchronous layer but does not change the failure mode: the exception ultimately propagates to the model method call site, which in v0.1 is the controller (`action_handlers.py`). The controller is responsible for catching it (or letting it surface to the user via `app.py`'s unhandled-exception handler).

**v0.1 policy:** same as the umbrella spec's documented risk — "model lifecycle: propagate without rollback." The model mutation has already committed by the time `send()` fires. A raising Qt slot does not roll back the model action. The bridge does **not** wrap `self.*.emit(payload)` in try-except — that would swallow errors that should surface. The error policy is: propagate, let the caller (controller/`app.py`) handle.

**PySide6 6.5.2+ auto-capture in event loop:** this does NOT apply here because blinker fires outside the Qt event loop. The Qt exception capture that pytest-qt documents only catches exceptions from virtual C++ methods overridden in Python (e.g. `paintEvent`, `mousePressEvent`). Slot exceptions from a `DirectConnection` `emit()` call within a synchronous blinker callback are NOT captured by Qt's mechanism. [official] (https://pytest-qt.readthedocs.io/en/latest/virtual_methods.html)

**`_send_robust` deferral:** the umbrella spec (step-7 §9.2, step-8 Q8) defers fault-isolated dispatch to v0.2. The step-9 bridge inherits that deferral. The bridge should have a comment naming the deferral so a future maintainer adding a second view subscriber knows to also consult the `_send_robust` design.

## Footguns and Gotchas

- **blinker `weak=True` default silently disconnects bridge bound methods.** blinker's default `connect(receiver, weak=True)` holds a weakref to the receiver. For a bound method (`self._on_ship_moved`), the weakref is to the underlying object (`self`). If the `ModelBridge` instance has no strong references outside of blinker's receiver tables, the weakref expires and blinker silently drops the connection — no error, no warning, signals from the model are silently ignored. **Fix:** always pass `weak=False` when connecting a bridge bound method to a blinker signal. — corroborated by [blinker 1.10 official docs](https://blinker.readthedocs.io/en/latest/) and [Flask/Django signal guidance](https://stackoverflow.com/questions/29821515/)

- **ABCMeta + QObject metaclass conflict (`TypeError: Shiboken.ObjectType.__new__(...) is not safe`).** Directly combining `class ModelBridge(QObject, ABC)` raises a metaclass conflict because `ABCMeta` and shiboken's `ShibokenObjectType` are incompatible metaclasses. The fix is to either avoid inheriting from `ABC` (pure `QObject` subclass, no `ABC` needed since the bridge has no abstract interface), or create a merged metaclass: `class _BridgeMeta(type(QObject), ABCMeta): ...`. For the bridge, the simpler path is to not use `ABC` at all — the bridge is a concrete class, not an abstract base. — corroborated by [bugreports.qt.io PYSIDE-1767](https://bugreports.qt.io/browse/PYSIDE-1767) and [SO: metaclass conflict](https://stackoverflow.com/questions/66591752/)

- **`RuntimeError: Internal C++ object already deleted`** when a connected Qt slot calls a method on a `QObject` that has been garbage-collected. Shiboken's dual Python/C++ ownership means the Python reference can go out of scope before all Qt connections are cleaned up. In the bridge scenario: if the view widget is destroyed but not disconnected, the bridge's Qt signal `emit()` fires into a dead C++ object. Fix: use `QObject` parent-child ownership (pass the `MainWindow` as parent to `ModelBridge` in `app.py`) so the bridge is destroyed when the window is destroyed; OR explicitly disconnect in the view's `closeEvent`. — corroborated by [pyqtgraph issue #2737](https://github.com/pyqtgraph/pyqtgraph/issues/2737) and [Qt Forum forum.qt.io/topic/154590](https://forum.qt.io/topic/154590/pyside6-memory-model-and-qobject-lifetime-management)

- **Qt signal `emit()` exception propagation in DirectConnection.** For `DirectConnection` (which applies when emitter and receiver share the same thread), a slot that raises causes the exception to propagate through `emit()`. This is documented behavior (Qt C++ spec, confirmed by [SO: exception throwing in slots](https://stackoverflow.com/questions/17201318/)). The bridge's `_on_*` callbacks will see the exception propagate back through them into blinker's `send()`. This is the intended behavior for v0.1 (propagate without rollback), but the author of the bridge must be aware that `self.*.emit(payload)` is NOT exception-safe by default.

- **`Signal` must be a class attribute, not an instance attribute.** `Signal(object)` declared inside `__init__` (e.g. `self.ship_moved = Signal(object)`) will raise `AttributeError` when accessed as a bound signal — the descriptor protocol only fires on class-level attributes. This is a hard error, not a silent failure. — corroborated by [SO: Signals in PySide6](https://stackoverflow.com/questions/67231753/) and [official Signal docs](https://doc.qt.io/qtforpython-6/PySide6/QtCore/Signal.html)

- **Blinker `receivers_for()` vs `has_receivers_for()` difference.** `has_receivers_for()` does NOT check if weakly-referenced receivers are still live (documented). `receivers_for()` lazily disconnects dead weakrefs. If using `receivers_for()` in the future `_send_robust` utility, iterate from `receivers_for()`, not `has_receivers_for()`. — corroborated by [blinker 1.10 official docs](https://blinker.readthedocs.io/en/latest/)

## Security and Compatibility

- **PySide6 PYSIDE-3288 (6.9+):** "It is now possible to defer deletion of QObjects in case they get garbage-collected by a thread different from their owner thread and thus ensure the correct thread affinity." This is a safety improvement for multi-threaded teardown; v0.1 is single-threaded so it is not directly relevant, but it signals the Qt team has acknowledged the cross-thread GC issue and addressed it. The fix is transparent; no code change needed. [official] (https://doc.qt.io/qtforpython-6/release_notes/pyside6_release_notes.html)

- **Python 3.14 free-threading (PEP 703):** blinker `send()` is synchronous and not thread-safe under GIL-disabled builds. The `receivers` dict mutation during `receivers_for()` (cleaning dead weakrefs) is not protected. However, v0.1 is single-threaded — all model mutations and signal dispatch happen on the main thread. Impact is deferred to v0.2. Previously documented in step-7 research. [official] (https://py-free-threading.github.io/porting/)

- **mypy stubs for PySide6 `Signal`:** The official stubs (bundled with PySide6 since 6.4.3) type `Signal(object)` as `Signal` (unparameterized); slots receive `object`. The third-party `PySide6-stubs` package offers improved accuracy but still cannot encode per-instance payload types generically. Under `mypy --strict`, receiving a `Signal(object)` payload requires either annotating the slot as `(object) -> None` and narrowing internally, or using `# type: ignore` at the narrowing site. **The bridge itself can be typed accurately** (`emit(payload: ShipMovedPayload)`) since the bridge *knows* the payload type; it is the *view slot* that sees `object`. [community] (https://pypi.org/project/PySide6-stubs/)

- **No CVEs for blinker 1.x or PySide6 6.x surfaced in this sweep.**

## Recent Changes

- **PySide6 6.5.2 (2023):** Automatic exception capture for Python exceptions in virtual methods (e.g. `paintEvent`) — context manager-based hook no longer required. This does NOT apply to blinker callbacks or direct-connection slot exceptions. [official] (https://pytest-qt.readthedocs.io/en/latest/virtual_methods.html)

- **PySide6 6.9 / PYSIDE-3288:** Deferred QObject deletion when GC fires off-thread. Relevant as a safety improvement for v0.2 threaded teardown. [official] (https://doc.qt.io/qtforpython-6/release_notes/pyside6_release_notes.html)

- **blinker 1.10 (current):** No `send_robust`. No new API additions relevant to the bridge since 1.6 (`send_async`). The `weak=True` default has been the behavior since 1.1 (2012) — no change, but frequently misunderstood by developers coming from Qt's auto-disconnect behavior. [official] (https://blinker.readthedocs.io/en/latest/)

- **Python 3.14 / PEP 649 (deferred annotations default):** `from __future__ import annotations` is no longer needed to defer annotation evaluation in Python 3.14 — PEP 649 makes lazy evaluation the default. However, `model/events.py` uses `TYPE_CHECKING`-only imports (not just string annotations) for payload field types, and the bridge's `model_bridge.py` will NOT need `TYPE_CHECKING` guards — it imports the payload types directly at runtime (they are concrete at the controller layer). The `from __future__ import annotations` pattern used throughout the model layer should also appear in `model_bridge.py` for consistency; it is harmless. [official] (https://docs.python.org/3/whatsnew/3.14.html)

## Locked-decisions draft (for spec §9.1)

1. **Q1: Single `ModelBridge(QObject)` class with all four `Signal(object)` attributes.** One object, uniform thread affinity, one ownership point. Matches the `WorkerSignals(QObject)` idiom from Qt's official threading docs.

2. **Q2a: Signal names mirror blinker names exactly** (`ship_moved`, `docked`, `turn_advanced`, `state_changed`) per umbrella spec §5.2's "bridge translation is mechanical" principle.

3. **Q2b: `Signal(object)` carries the entire frozen dataclass payload.** The payload types contain non-primitive fields (`GridPosition`, `type[GameState]`) that cannot be expressed as primitive Qt signal types. Slots receive `object` and should narrow to the specific payload type.

4. **Q3: Error policy — propagate, do not swallow.** The bridge does NOT wrap `self.*.emit(payload)` in try-except. A raising Qt slot propagates through emit, through the blinker callback, through blinker's `send()`, and out to the model method call site. This is the documented v0.1 behavior per umbrella spec §9 (single-subscriber, fault isolation deferred to v0.2). A comment in the bridge must name this deferral.

5. **`weak=False` on all blinker connections.** The bridge is a long-lived singleton owned by `app.py`. Use `ev.ship_moved.connect(self._on_ship_moved, weak=False)` for all four signals to prevent silent disconnection.

6. **No `ABC` inheritance on `ModelBridge`.** The bridge is a concrete class; adding `ABC` triggers the shiboken metaclass conflict. If a protocol interface is needed for test injection, use `typing.Protocol` with `TYPE_CHECKING` guards only.

7. **Parent `QObject` in `app.py`.** `ModelBridge(parent=main_window)` ensures the bridge is destroyed when the main window is closed, which disconnects Qt signals automatically (Qt's `~QObject` auto-disconnects). The view should also hold no blinker subscriptions (enforced by the import-linter `view-does-not-import-model-events` contract).

8. **`@Slot(object)` on view slots.** Recommended (not mandatory) for Qt registration accuracy. The bridge's own internal handlers (`_on_ship_moved`, etc.) do not need `@Slot` since they are not registered as C++ slots — they are plain Python callables in blinker's receiver table.

## Risks

- **Silent signal loss** is the highest-severity risk. If `weak=False` is forgotten for even one of the four blinker connections, the bridge will stop receiving that signal when the Python GC next runs — typically within milliseconds in CPython. No error is raised. The test suite must verify that signals still arrive after a GC cycle: `import gc; gc.collect()` between connect and the triggering model action.

- **Dual-layer exception propagation** is novel in this codebase. Prior modules (model entities, state manager) only deal with the blinker layer. The bridge adds Qt's `emit()` call inside the blinker callback. A test that verifies the propagation chain end-to-end is mandatory.

- **QObject lifecycle vs blinker subscription.** The bridge subscribes to module-level blinker signals (which live for the process lifetime). If a second `ModelBridge` is accidentally constructed (e.g. in a test), blinker will have two subscribers and both will emit. Test isolation requires either a per-test bridge construction with explicit teardown, or a module-level singleton guard.
