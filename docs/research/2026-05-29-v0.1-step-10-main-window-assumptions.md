## Assumption 1: QStatusBar for Permanent Turn-Bar

**Verdict: Confirmed**

`QStatusBar.addPermanentWidget()` is the idiomatic Qt 6 choice for a persistent bottom bar containing interactive widgets (labels, buttons, state indicators). The official Qt 6 docs state: "Normal and Permanent messages are displayed by creating a small widget (QLabel, QProgressBar or even QToolButton) and then adding it to the status bar using `addWidget()` or `addPermanentWidget()`." The `addPermanentWidget()` variant places widgets at the right, right-aligned and non-closable by the user, while `addWidget()` places them on the left.

**Design confirmation:** QPushButton ("End Turn"), QLabel (Turn/Phase/AP), and a state-indicator QLabel all work correctly as permanent status-bar widgets in Qt 6.

**Comparison with alternatives:**
- `QDockWidget` at the bottom: valid but adds unwanted floatability/closability that must be suppressed via `setFeatures(QDockWidget.NoDockWidgetFeatures)`. Status bar is simpler for a fixed interactive bar.
- Central widget layout with fixed bottom row: possible, but gives up the QMainWindow-managed sizing and `showMessage()` transient-message capability you may want later.

**Footgun:** `addWidget()` (non-permanent) can be obscured by `showMessage()` transient messages. Use `addPermanentWidget()` for widgets that must never disappear. [official] (https://doc.qt.io/qt-6/qstatusbar.html)

## Assumption 2: Testing exec() / main() Without Blocking

**Verdict: Confirmed (both patterns are valid; monkeypatching is the correct approach for app-exit tests)**

The pytest-qt event loop **is already running** before any test function is called; `pytest-qt` calls `QApplication.processEvents()` around each test. You do **not** call `app.exec()` in tests — the event loop is managed by the framework.

**For testing a `main()` function that calls `app.exec()`:**

The correct approach is `monkeypatch.setattr(QApplication, "exec", lambda: 0)` before calling `main()`. The pytest-qt docs explicitly warn: "Calling `QApplication.exit()` from a test will cause the main event loop and auxiliary event loops to exit and all subsequent event loops to fail to start." The equivalent risk applies to calling `app.exec()` — it would block indefinitely or prevent subsequent tests from having a working event loop. [official] (https://pytest-qt.readthedocs.io/en/latest/qapplication.html)

**The QTimer.singleShot(0, app.quit) before exec() pattern** is used in non-pytest contexts (e.g., integration scripts that want a real event loop), not the recommended pytest-qt approach. In pytest, monkeypatching is cleaner because it avoids starting a nested event loop at all.

**Canonical pytest-qt recipe for a top-level QMainWindow:**
```python
def test_main_window_smoke(qtbot):
    window = MainWindow()
    qtbot.addWidget(window)  # registers for cleanup
    window.show()
    with qtbot.waitExposed(window, timeout=2000):
        pass
    assert window.windowTitle() == "Star Trek Retro Remake"
```

`qtbot.addWidget()` is the canonical registration; `qtbot.waitExposed()` is for X11/async-show environments. Under the offscreen platform, `waitExposed()` may return immediately (no real async mapping). `qtbot.addWidget()` ensures the widget is cleaned up (hidden + deleted) after the test.

**Footgun — QApplication.exit() poisoning subsequent tests:** If any code path calls `QApplication.exit()`, monkeypatch it in those tests or the entire subsequent test suite's event loop breaks. This is a documented pytest-qt bug/limitation. [official] (https://pytest-qt.readthedocs.io/en/latest/qapplication.html)

**Footgun — pytest-qt's qapp name:** pytest-qt sets `QApplication.applicationName()` to `"pytest-qt-qapp"` by default. If your `main()` sets the name via `app.setApplicationName(...)`, this will override pytest-qt's setting; that is fine. But if your code reads the name to resolve QSettings paths during import, test isolation may be affected. Use `qt_qapp_name` in `pytest.ini` to align names if needed. [official] (https://pytest-qt.readthedocs.io/en/latest/qapplication.html)

**Sources:**
- pytest-qt Testing QApplication docs [official] (https://pytest-qt.readthedocs.io/en/latest/qapplication.html)
- pytest-qt app_exit note [official] (https://pytest-qt.readthedocs.io/en/3.0.0/app_exit.html)
- pytest-qt waitExposed reference [official] (https://pytest-qt.readthedocs.io/en/latest/reference.html)

## Assumption 3: closeEvent for Teardown; super() Call Order

**Verdict: Confirmed — call teardown BEFORE super().closeEvent(event)**

`QMainWindow.closeEvent(event)` is the correct owner-side cleanup hook. It fires reliably on:
- `window.close()` (programmatic close)
- User clicking the X button
- `QApplication.quit()` → triggers close of all top-level windows
- Under offscreen/headless tests when `window.close()` is called explicitly

The canonical pattern from multiple sources (pythonguis.com, Qt Forum posts):
```python
def closeEvent(self, event: QCloseEvent) -> None:
    self._bridge.teardown()   # disconnect blinker subscriptions first
    super().closeEvent(event) # then let Qt accept the event and destroy children
```

**Why teardown before super():** `super().closeEvent(event)` calls `event.accept()` on the default implementation. In theory this is benign for teardown ordering, but the C++ side begins child destruction once the event is accepted and the window starts closing. Doing your teardown after `super()` risks accessing child widgets that are in a partially-destroyed state if destruction is synchronous. Multiple community sources and the Qt docs pattern show cleanup before `super()`.

**Comparison with `QObject.destroyed.connect(...)`:** `destroyed` fires during C++ destruction, after the Python object may have already lost its Qt side. Using `closeEvent` is safer because the widget is still fully alive when it fires. The `destroyed` signal is useful for cleanup in *other* objects watching a widget, not for self-cleanup.

**Footgun — closeEvent not firing for child QWidget subclasses in certain hosts (Maya 2025 / embedded DCC):** Known crash in some embedding contexts. Not relevant for standalone `QApplication` use. [community] (https://forums.autodesk.com/t5/maya-programming-forum/maya-2025-pyside6-window-with-closeevent-crashes-maya)

**Footgun — `event.ignore()` must call `return` after:** If your `closeEvent` conditionally calls `event.ignore()` (e.g., "Are you sure?"), do not call `super().closeEvent(event)` in the ignore branch, or Qt will accept the close anyway.

**Sources:**
- pythonguis.com clean-up on exit tutorial (updated Apr 2026) [community] (https://www.pythonguis.com/faq/clean-up-on-exit-application/)
- Qt Forum PySide6 closeEvent discussion [community] (https://forum.qt.io/topic/158020/application-not-shutting-down-properly-when-x-button-is-clicked)
- Qt for Python QCloseEvent docs [official] (https://doc.qt.io/qtforpython-6/PySide6/QtGui/QCloseEvent.html)

## Assumption 4: Window-State / Fullscreen Under offscreen Platform

**Verdict: REFUTED — do not assert fullscreen/focus states under offscreen**

This is the highest-risk assumption in the design. The offscreen QPA plugin is documented and community-confirmed to **not reliably handle:**

- `hasFocus()` — confirmed broken under offscreen. A test that passes on xcb/real display fails at `assert w.hasFocus()` under `QT_QPA_PLATFORM=offscreen`. [community] (https://forum.qt.io/topic/109767/headless-gui-testing-with-qt)
- `isFullScreen()` / `showFullScreen()` — the offscreen platform has no window manager, so window state transitions (normal → fullscreen → normal) may not generate the expected state changes. The platform does not communicate with a real WM.
- `isActiveWindow()`, `isMaximized()` — same category: WM-dependent state that the offscreen plugin does not simulate.

**What does work reliably under offscreen:**
- Widget construction, show(), hide()
- Signal emissions from user-initiated actions (via qtbot.mouseClick, keyClicks)
- `isVisible()`, `isHidden()`
- `geometry()` when set programmatically (e.g., `setGeometry`, `setMinimumSize`); the offscreen platform respects explicitly-set geometry
- `qtbot.waitExposed()` (may return immediately on offscreen since no real mapping needed)

**Recommended test approach for F11 fullscreen toggle:**

Do not assert `isFullScreen()`. Instead assert:
1. The QAction's `isChecked()` state (if the fullscreen toggle is a QAction)
2. The `windowState()` flag value — but note Qt returns the *requested* state; offscreen may not fire `windowStateChanged` reliably
3. Alternatively: use a real display in a dedicated integration test, or skip the offscreen assert and test only that the correct method was called via a mock

Best practice: keep offscreen tests focused on widget structure, signal wiring, and business logic — not on window-manager-dependent state. For the fullscreen toggle specifically, test that calling the action calls `showFullScreen()` (via mock), not that `isFullScreen()` returns True.

**`setMinimumSize(1600, 1000)` under offscreen:** Setting a minimum size is respected by the offscreen platform (it is stored and honored for layout purposes). Tests that check `minimumSize()` are reliable. Tests that try to programmatically resize below the minimum and assert rejection may or may not work — skip those under offscreen.

**Sources:**
- Qt Forum headless GUI testing thread [community] (https://forum.qt.io/topic/109767/headless-gui-testing-with-qt) — confirmed `hasFocus()` failure under offscreen
- pytest-qt debugging docs [official] (https://pytest-qt.readthedocs.io/en/4.2.0/debugging.html) — notes offscreen hides windows from debugger
- Qt Forum offscreen OpenGL error [community] (https://forum.qt.io/topic/159263/opengl-error-when-trying-to-use-qt_qpa_platform-offscreen)

## Assumption 5: QObject Lifetime — Parent + Python ref

**Verdict: Confirmed with nuance — BOTH the Qt parent AND the Python ref are required**

The two failure modes:

**Failure mode A — C++ deleted, Python wrapper alive:** When Qt deletes the C++ object (e.g., parent window closes, child is destroyed), any Python reference to the child's wrapper becomes a dead shell. Accessing it raises `RuntimeError: Internal C++ object already deleted`. This applies to child widgets/QObjects whose C++ side is deleted by Qt parent-child cleanup. [official] (https://wiki.qt.io/PySide_Shiboken_Object_Ownership)

**Failure mode B — Python GC'd, C++ alive:** If a QObject is created without a Qt parent and there is no live Python reference, the Python garbage collector deletes the wrapper, which in PySide6 also deletes the C++ object. This causes silent disappearance of slots/subscriptions. [community] (https://stackoverflow.com/questions/5339062/python-pyside-internal-c-object-already-deleted)

**For `ModelBridge(QObject)` with `parent=main_window`:**
- Qt parenting prevents failure mode B: Qt holds a strong reference to child C++ objects. However, Qt parenting **alone does not protect the Python wrapper** from GC in all PySide6 versions. The Shiboken binding tracks Python-side ownership separately from C++ parent ownership.
- `self._bridge = ModelBridge(parent=main_window)` in `MainWindow.__init__` stores a Python ref on `self` (the window), which prevents Python-side GC as long as the window is alive. This is the correct pattern.
- When the window closes and its C++ side is destroyed, Qt destroys the bridge's C++ side too (parent-child). Then the Python ref `self._bridge` becomes a dead wrapper. This is expected and correct — you should not access the bridge after the window is closed.

**The `self._manager` held by `self._bridge`:** If `_manager` is not itself a QObject with a Qt parent, it must have a Python reference somewhere. `self._bridge._manager` (i.e., the bridge holding a ref to the manager) keeps it alive as long as the bridge is alive. That is the correct pattern.

**Blinker `weak=False` implication:** With `weak=False`, blinker holds a strong reference to the subscriber callable. This means blinker itself keeps the callable alive — but if the callable is a bound method of `ModelBridge`, blinker holds a strong ref to the bound method (and transitively to `self`, the bridge). This prevents Python-side GC of the bridge even without `self._bridge`. However, this also means blinker subscriptions **must be explicitly disconnected** in `closeEvent` / `teardown()`, or the bridge will never be GC'd (memory leak after window close). The explicit `self._bridge` Python ref is still correct and required for clarity and safety. [official] (https://blinker.readthedocs.io/)

**Footgun — GC across threads:** PySide6 6.8+ added deferred deletion for QObjects GC'd from a non-owner thread (PYSIDE-3288). Under Python 3.14 with its new GIL changes, this is more relevant. The explicit Python ref and Qt parent together are sufficient for the main-thread case. [official] (https://doc.qt.io/qtforpython-6/release_notes/pyside6_release_notes.html)

**Footgun — signal connection holds callable ref but may not prevent GC of self:** Connecting a bound method to a Qt signal does NOT reliably increment the refcount of the bound method's object in all PySide6 versions. From the Qt Forum (Feb 2024): "connecting a bound method to a signal does not increment the bound method's refcount." This is why explicit Python refs (`self._bridge`) are required beyond just having Qt signals connected. [community] (https://forum.qt.io/topic/154590/pyside6-memory-model-and-qobject-lifetime-management)

**Sources:**
- PySide Shiboken Object Ownership (Qt Wiki) [official] (https://wiki.qt.io/PySide_Shiboken_Object_Ownership)
- Qt Forum PySide6 memory model thread [community] (https://forum.qt.io/topic/154590/pyside6-memory-model-and-qobject-lifetime-management)
- PySide6 release notes PYSIDE-3288 [official] (https://doc.qt.io/qtforpython-6/release_notes/pyside6_release_notes.html)
- Stack Overflow GC discussion [community] (https://stackoverflow.com/questions/17914960/pyqt-runtimeerror-wrapped-c-c-object-has-been-deleted)

## Assumption 6: QDockWidget.toggleViewAction()

**Verdict: Confirmed — self-contained, correct, idiomatic**

`QDockWidget.toggleViewAction()` returns a `QAction` that:
- Is checkable (`action.isCheckable() == True`)
- Has its text set to the dock widget's `windowTitle()`
- Automatically tracks dock visibility: checked = dock visible, unchecked = dock hidden
- Can be directly added to a View menu: `view_menu.addAction(dock.toggleViewAction())`

This is the idiomatic Qt 6 pattern for dock-toggle menu actions. The official Qt 6.11 docs confirm: "Returns a checkable action that can be added to menus and toolbars so that the user can show or close this dock widget." [official] (https://doc.qt.io/qt-6/qdockwidget.html)

**Footgun — title bar mnemonics:** The action text inherits the dock's `windowTitle()` including any `&` mnemonic markers. This can cause unexpected keyboard shortcuts in the View menu. If the dock title contains `&`, be aware it becomes a menu shortcut. [community] (https://forum.qt.io/topic/113595/dockwidget-toggleviewaction-and-mnemonic-handling)

**Footgun — toggling programmatically:** Calling `dock.hide()` or `dock.show()` updates the action's checked state. Calling `action.trigger()` hides/shows the dock. These are bidirectionally synced — no manual `visibilityChanged` connection required.

**Sources:**
- Qt 6.11 QDockWidget::toggleViewAction docs [official] (https://doc.qt.io/qt-6/qdockwidget.html)
- Qt Forum toggleViewAction and mnemonic handling [community] (https://forum.qt.io/topic/113595/dockwidget-toggleviewaction-and-mnemonic-handling)
- Stack Overflow show/hide QDockWidget [community] (https://stackoverflow.com/questions/1290194/show-hide-qdockwidget)

## Assumption 7: python -m stmrr Entry Convention

**Verdict: Confirmed — `raise SystemExit(main())` in `__main__.py` is the canonical pattern**

`src/stmrr/__main__.py` doing:
```python
from stmrr.app import main
raise SystemExit(main())
```

This is the current best-practice entry shape. Breakdown:

- `raise SystemExit(main())` is equivalent to `sys.exit(main())` (the latter calls `raise SystemExit` internally). `raise SystemExit(...)` is preferred in `__main__.py` because it avoids importing `sys` and is slightly more direct.
- `main() -> int` returning `app.exec()`'s integer return code is correct: `QApplication.exec()` returns 0 on clean exit.
- `[project.scripts]` console_scripts entry in pyproject.toml is the correct companion for installed-as-command use: `stmrr = "stmrr.app:main"`. This is **not exclusive** with `__main__.py` — both should exist, serving different invocation paths.
- `sys.exit(app.exec())` inline in `__main__.py` (without a `main()` function) is a common pattern in tutorials but is less testable. The `main() -> int` function shape allows tests to call `main()` with a monkeypatched `exec`.

**Footgun — IPython / Jupyter:** `sys.exit()` or `raise SystemExit()` in a `__main__.py` run via `%run` in IPython raises an unhandled `SystemExit` in the notebook. This is a tutorial warning but not relevant for a game binary. [community] (https://www.w3tutorials.net/blog/simple-ipython-example-raises-exception-on-sys-exit/)

**Sources:**
- Python docs: `sys.exit()` raises `SystemExit` [official] (https://docs.python.org/3/library/sys.html)
- Real Python SystemExit reference [community] (https://realpython.com/ref/builtin-exceptions/systemexit/)
- PySide6 community examples [community] (https://python.hotexamples.com/examples/PySide6.QtWidgets/QApplication/setApplicationName/)

## Assumption 8: Modal Dialogs in Tests — Monkeypatch QMessageBox.about

**Verdict: Confirmed — monkeypatching static methods is the official pytest-qt recommendation**

The pytest-qt documentation's "A note about Modal Dialogs" section explicitly recommends monkeypatching QMessageBox static methods:

```python
def test_about_dialog(qtbot, monkeypatch):
    window = MainWindow()
    qtbot.addWidget(window)
    called = []
    monkeypatch.setattr(QMessageBox, "about", lambda *args: called.append(True))
    window.trigger_help_about()
    assert called
```

This is the correct approach for `QMessageBox.about(...)`, `QMessageBox.question(...)`, `QMessageBox.information(...)` and other static convenience methods. [official] (https://pytest-qt.readthedocs.io/en/latest/note_dialogs.html)

**Alternative — QTimer.singleShot to close an actual dialog:** Used when the test needs to interact with the real dialog (check its contents, click specific buttons). The pattern is:
```python
QTimer.singleShot(100, lambda: QApplication.activeWindow().close())
dialog.exec()  # the timer fires while exec blocks, then exec returns
```
This is more complex and only needed for dialogs with meaningful content to verify. For `QMessageBox.about`, monkeypatching is simpler and sufficient. [community] (https://ert.readthedocs.io/en/18.0.3/developer_documentation/qt.html)

**Footgun — `QMessageBox.about` vs instance exec:** `QMessageBox.about(parent, title, text)` is a static method that constructs, shows, and exec()s a dialog internally. Monkeypatching the class-level method prevents blocking entirely. Do not try to use `QTimer.singleShot` with this static method since the timer has to fire before the method is called, and there is no separate `exec()` call to hook into.

**Sources:**
- pytest-qt Modal Dialogs note [official] (https://pytest-qt.readthedocs.io/en/latest/note_dialogs.html)
- ERT testing documentation (QTimer.singleShot pattern for dialogs) [community] (https://ert.readthedocs.io/en/18.0.3/developer_documentation/qt.html)
- pytest-qt issue #18 (original modal dialog question) [community] (https://github.com/pytest-dev/pytest-qt/issues/18)

## Assumption 9: setApplicationName / setOrganizationName at Startup

**Verdict: Confirmed — set immediately after QApplication() construction, before any widget or QSettings use**

The official Qt and PySide6 documentation is unambiguous: "If `setOrganizationName()` and `setApplicationName()` has not been previously called, the QSettings object will not be able to read or write any settings, and `status()` will return `AccessError`." [official] (https://doc.qt.io/qtforpython-6/PySide6/QtCore/QSettings.html)

The correct order in `main()`:
```python
app = QApplication(sys.argv)
app.setApplicationName("Star Trek Retro Remake")
app.setOrganizationName("luminous3d")
app.setOrganizationDomain("luminous3d.com")  # optional; used on macOS for plist path
# Only now construct MainWindow (which may internally use QSettings)
window = MainWindow()
```

Setting these values now (before any QSettings use) is harmless — they are instance properties on the QCoreApplication singleton and take effect immediately for all subsequent default-constructor `QSettings()` calls. There is no ordering footgun as long as they are set before the first QSettings is constructed.

**Platform path resolution (Linux):** On Linux with INI format, the path becomes `~/.config/<OrganizationName>/<ApplicationName>.ini` (or `~/.config/<OrganizationDomain>/<ApplicationName>.ini` if domain is set). Setting these now sets up the correct path for the future `saveGeometry()`/`restoreGeometry()` step.

**pytest-qt interaction:** pytest-qt sets `applicationName` to `"pytest-qt-qapp"` on the shared `QApplication`. If tests use QSettings with the default constructor and your test fixture does not reset the app name, they will write to `~/.config/pytest-qt-qapp/`. This is benign (tests should not persist real settings) but worth knowing. To avoid polluting the user's config dir in tests, prefer `QSettings(QSettings.Format.IniFormat, QSettings.Scope.UserScope, "test", "test")` with explicit paths in test fixtures, or use a temp directory.

**Sources:**
- Qt for Python QSettings docs [official] (https://doc.qt.io/qtforpython-6/PySide6/QtCore/QSettings.html)
- Python GUIs QSettings tutorial (updated Mar 2026) [community] (https://www.pythonguis.com/faq/pyside6-qsettings-how-to-use-qsettings/)
- PySide6 setOrganizationName examples [community] (https://python.hotexamples.com/examples/PySide6.QtWidgets/QApplication/setApplicationName/)

## Footguns and Gotchas (Consolidated)

- **Offscreen + window state** — `isFullScreen()`, `hasFocus()`, `isActiveWindow()`, `isMaximized()` are unreliable under `QT_QPA_PLATFORM=offscreen`. Corroborated by Qt Forum thread (hasFocus failure confirmed) and pytest-qt debugging docs. Do not write offscreen tests that assert these values.
- **QApplication.exit() in tests** — if any code path calls `QApplication.exit()` (not `app.quit()`), subsequent tests' event loops break. Monkeypatch `QApplication.exit` in those tests. [official] (https://pytest-qt.readthedocs.io/en/latest/qapplication.html)
- **Blinker weak=False + teardown required** — with `weak=False`, blinker holds a strong reference to the subscriber. The bridge will not be GC'd after window close unless `signal.disconnect(receiver)` is called explicitly in `teardown()`. [official] (https://blinker.readthedocs.io/)
- **Qt signal connection does not increment Python refcount** — do not rely on Qt signal connections to keep Python objects alive. Always hold explicit Python refs. [community] (https://forum.qt.io/topic/154590/pyside6-memory-model-and-qobject-lifetime-management)
- **addWidget() vs addPermanentWidget()** — addWidget() slots are obscured by showMessage(). Use addPermanentWidget() for turn-bar widgets. [official] (https://doc.qt.io/qt-6/qstatusbar.html)
- **pytest-qt app name override** — pytest-qt sets applicationName to "pytest-qt-qapp". QSettings in tests will use this name unless overridden.
- **PySide6 6.7.0 disconnect bug** — pytest-qt issue #552 documents "RuntimeError: Failed to disconnect" with PySide6 6.7.0. Fixed in subsequent releases. Verify with 6.11.0. [community] (https://github.com/pytest-dev/pytest-qt/issues/552)

## Recent Changes

- **PySide6 6.8+ (PYSIDE-3288):** Deferred deletion of QObjects GC'd from non-owner threads. Relevant for multi-threaded use; main-thread-only apps (like this scaffold) are unaffected. [official] (https://doc.qt.io/qtforpython-6/release_notes/pyside6_release_notes.html)
- **pytest-qt 6.7.0 PySide6 disconnect bug (#552):** `RuntimeError: Failed to disconnect` in some teardown scenarios. Monitor with PySide6 6.11.0. [community] (https://github.com/pytest-dev/pytest-qt/issues/552)
- **Python 3.14 + PySide6:** No confirmed incompatibilities found as of 2026-05-29, but Python 3.14 is recent; verify PySide6 6.11.0 wheel availability for Python 3.14 before pinning in CI.
- **QStatusBar permanent widget layout:** Behavior confirmed unchanged in Qt 6.11 vs Qt 6.5. [official]
