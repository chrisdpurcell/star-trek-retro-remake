## Prioritized findings for Codex adversarial review

**These are the highest-risk challenges a sharp reviewer will raise:**

### 1. ❗ (Medium) Spec §5.3 covariance claim is imprecise

The spec says `frozenset({SectorMapState})` "infers to `frozenset[type[GameState]]`" — it does not; it infers to `frozenset[type[SectorMapState]]`, which is acceptable to mypy's annotated assignment because `frozenset` is covariant and `type[SectorMapState]` is a subtype of `type[GameState]`. A Codex reviewer may reject this phrasing as technically wrong.

**Fix:** replace with "mypy accepts the assignment because `frozenset` is covariant in its element type; `frozenset[type[SectorMapState]]` is a subtype of `frozenset[type[GameState]]`." No Q-lock change needed; §5.3 wording change only.

### 2. ❗ (Low-medium) No CI evidence for the "no `# type: ignore` needed" claim

The assertion is correct in theory, but a Codex reviewer doing adversarial review is entitled to ask for a `reveal_type` test or a mypy-play link. The spec should either point to the CI gate that confirms this or add a `reveal_type(MainMenuState.allowed_transitions)  # Revealed type: builtins.frozenset[type[stmrr.model.state.states.GameState]]` assertion in the test file. If the project uses mypy < 1.18, the frozenset improvement (PR 18571) may not be present.

**Verify:** project mypy version pinned to ≥ 2.1.0 (for the `type[T]` narrowing fix from PR 21174) AND ≥ 1.18 (for the frozenset improvement).

### 3. ❗ (Low) The research file's python-statemachine comparison is imprecise

The research file (§Q2) says: "Matches `python-statemachine`'s default (initial state's `on_enter` fires when the state machine is bound to a model)."

This is half-right. `python-statemachine` 3.1.2 fires `on_enter` for the initial state AND fires a synthetic `__initial__` event that triggers `after_transition` callbacks. The spec deliberately fires `on_enter` only, not an equivalent event.

A Codex reviewer who reads the python-statemachine docs will note the mismatch and may use it to argue the spec should also emit an event on construction. The spec's Q2 lock is correct; the research file's supporting rationale is slightly wrong.

**Fix:** "Partial match: `python-statemachine` fires both initial `on_enter` AND a synthetic `__initial__` event. This spec fires initial `on_enter` but deliberately suppresses the event emission — see Q2 lock rationale (preserves `StateChangedPayload.from_state: type[GameState]` strict typing without widening to `type[GameState] | None`)."

## Findings by question

### Question 1 — Post-class ClassVar reassignment under mypy --strict

**Verdict: spec claim is CORRECT. No issue.**

The official mypy docs (2.1.0 stable, confirmed across 2.2.0-dev) show the canonical example:

```python
class A:
    x: ClassVar[int] = 0  # Class variable only

A.x += 1  # OK  ← explicit in docs
```

Assignment **via the class name** (`ClassName.attr = value`) on a `ClassVar`-annotated attribute is explicitly permitted by mypy. The restriction is instance-side only (`a.x = 1  # Error`). The spec's module-level lines:

```python
MainMenuState.allowed_transitions = frozenset({SectorMapState})
SectorMapState.allowed_transitions = frozenset({MainMenuState})
```

are class-name assignments, not instance assignments. Mypy strict passes these without a `# type: ignore`. [official] (https://mypy.readthedocs.io/en/stable/class_basics.html)

**One nuance the spec does not address: `frozenset` variance.** `frozenset` is **covariant** ("Most immutable container types, such as `Sequence` and `frozenset` are covariant"), so `frozenset[type[SectorMapState]]` is indeed a valid subtype of `frozenset[type[GameState]]`. [official] (https://mypy.readthedocs.io/en/stable/generics.html)

However, mypy's type **inference** at the `frozenset({SectorMapState})` literal may produce `frozenset[type[SectorMapState]]` (the narrowed concrete type) rather than `frozenset[type[GameState]]`. The assignment target is annotated as `ClassVar[frozenset[type[GameState]]]`, so mypy uses the declared type as the target context — this is standard mypy behavior for annotated assignment targets. See **Prioritized finding 1** above for the precision fix.

### Question 2 — `__init_subclass__` ordering in CPython 3.14

**Verdict: spec claim is CORRECT. No CPython 3.13/3.14 ordering change.**

PEP 487 (the authoritative source, still current) states:

> "type.__new__ calls the __set_name__ hooks on the descriptor after the new class has been initialized. Then it calls __init_subclass__ on the base class... This means that subclass initializers already see the fully initialized descriptors." [official] (https://peps.python.org/pep-0487/)

The CPython 3.13 "what's new" documents changes to internal metaclass `__new__` vs `__init__` ordering — but these affect subclasses of `type` or built-in metaclasses only, **not** user-defined classes using `__init_subclass__` on a plain `object` subclass. The step-5 `states.py` ABC uses no custom metaclass; it inherits from `ABC` (which uses `ABCMeta`). The `__init_subclass__` hook on a class using `ABCMeta` follows the same PEP 487 ordering: class body fully evaluated → `__init_subclass__` fires. No ordering change in 3.14.

**No amendment to Q-locks needed.**

### Question 3 — blinker 1.9 with frozen dataclass payload carrying `type[X]` fields

**Verdict: spec claim is CORRECT. No surprises.**

The blinker CHANGES.rst (extracted from GitHub) shows version 1.9.0 (released 2024-11-08) removed `temporarily_connected_to`, `WeakNamespace`, and `receiver_connected` as deprecated items. The `send()` method contract — synchronous emission, kwargs forwarded to all receivers, exceptions propagate from the first receiver, returns list of `(receiver, return_value)` pairs — is **unchanged**. No mention of payload handling changes, no weak-reference behavior changes for positional/keyword data passed to `send()`. [official] (https://github.com/pallets-eco/blinker/blob/main/CHANGES.rst)

Blinker 1.10.0 is listed as "unreleased" with no entries at time of research.

**Frozen dataclass + `type[GameState]` fields:** `type` objects are hashable and strongly referenced by default. Blinker's sender-keyed weak-reference mechanism applies to **senders** (the first positional argument to `send()`), not to payload keyword arguments. The payload is passed as a regular kwarg dict entry, not as a sender, so blinker never weak-refs it. No surprising behavior. [official] (https://blinker.readthedocs.io/)

### Question 4 — Initial-state `enter()` in `__init__` without `state_changed`: comparison against library precedents

**Verdict: spec claims are CORRECT, but the research file's library comparison is slightly mis-stated.** See **Prioritized finding 3**.

The current `python-statemachine` docs (3.1.2) say:

> "The `__initial__` event is a synthetic event that the engine fires during initialization to enter the initial state. It follows the same RTC model as any other event." [official] (https://python-statemachine.readthedocs.io/en/latest/processing_model.html)

`transitions` (pytransitions) takes the **opposite** approach:

> "Note that `on_enter_«state name»` callback will not fire when a Machine is first initialized." [official] (https://github.com/pytransitions/transitions)

The spec is self-consistent. The research file comparison is partial-match-not-full-match.

### Question 5 — New Python 3.14 / typing idioms that could supersede the spec

**Verdict: no superseding idioms. PEP 695 and PEP 649 are irrelevant to the spec's patterns. PEP 742 `TypeIs` does not change the `type(target) in frozenset` check.**

**PEP 695 (`type` statement, Python 3.12+):** Introduces `type Alias = ...` and generic syntax `class Foo[T]: ...`. Does not affect the `ClassVar[frozenset[type[GameState]]]` pattern.

**PEP 649 / PEP 749 (Python 3.14 lazy annotations):** "In Python 3.14, `from __future__ import annotations` will continue to work as it did before, converting annotations into strings." No behavior change. [official] (https://peps.python.org/pep-0749/)

**PEP 742 `TypeIs` (Python 3.13+):** Not a type-guard predicate; the spec's `type(target) in current_state.allowed_transitions` is a runtime membership check, not a type narrower. No change needed.

**TypeForm (PEP 747, experimental in mypy):** Irrelevant.

### Question 6 — `type(target) in frozenset[type[GameState]]` under mypy --strict

**Verdict: likely clean, one unverified edge case worth confirming.**

Mypy 2.1.0 added "Avoid narrowing `type[T]` in type calls" (PR 21174). If the project's pinned mypy is < 2.1.0, prior versions had bugs around `type()` call narrowing. **Verify project mypy version ≥ 2.1.0.**

## Footguns and Gotchas

- **mypy type-narrowing bug family (#17981, #11897): subclass ClassVar overrides without explicit annotation get narrowed.** The spec avoids this by explicitly re-annotating `allowed_transitions: ClassVar[frozenset[type[GameState]]] = frozenset()` in each subclass body. If Codex strips the annotation from the subclass body, the narrowing bug may activate. The spec's explicit annotation is load-bearing. — corroborated by mypy issues #17981, #11897 [community] (https://github.com/python/mypy/issues/17981, https://github.com/python/mypy/issues/11897)

- **blinker 1.9 removed `temporarily_connected_to` as deprecated.** The spec's test idiom uses `try/finally` connect/disconnect (not `connected_to`), so no impact. [official]

- **mypy 1.18+ "Improve support for frozenset" (PR 18571, May 2025).** If the project pins mypy below 1.18, frozenset-related type errors could surface. [official]

- **`transitions` library explicitly does NOT fire `on_enter` at initialization.** The spec diverges from one of the two comparison libraries. A reviewer may raise this; the spec correctly handles it by stating the invariant ("every observable state has been entered") rather than aligning to any library's convention.

- **blinker 1.9.0 (2024-11-08):** removed `temporarily_connected_to`, `WeakNamespace`, `receiver_connected`. `send()` contract unchanged. [official]
- **mypy 2.1.0:** "Avoid narrowing `type[T]` in type calls" fix (PR 21174).
