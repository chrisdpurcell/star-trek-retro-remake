## Official Documentation

- Python 3.13 `typing` module: `ClassVar` can now be nested in `Final` and vice versa — this is a **3.13+ only feature**. [official] (https://docs.python.org/3/library/typing.html)
- PEP 591 §"Final and ClassVar": "Variables should not be annotated with both ClassVar and Final" (original PEP rule, superseded only for dataclasses in the 3.13 typing spec update). [official] (https://python.github.io/peps/pep-0591)
- Typing spec qualifiers page: In **non-dataclasses**, combining `ClassVar` and `Final` is **redundant**, and type checkers may choose to warn or error on the redundancy. In **dataclasses**, `ClassVar[Final[int]] = 3` is the required form for a final class-level field. [official] (https://typing.python.org/en/latest/spec/qualifiers.html)
- mypy final_attrs docs (2.1.0): "Final and ClassVar should not be used together." Plain `Final` in a class body is inferred as a class variable automatically. A `Final` attribute cannot be overridden by a subclass (even with another explicit `Final` declaration). [official] (https://mypy.readthedocs.io/en/stable/final_attrs.html)
- Blinker 1.10 official docs: Public API exposes only `send()` and `send_async()`. **`send_robust` is not in the blinker API.** The `send()` docstring says explicitly: "If a receiver raises an exception, that exception will propagate up. This makes debugging straightforward, with an assumption that correctly implemented receivers will not raise." [official] (https://blinker.readthedocs.io/en/latest/)
- Python built-in exceptions: `LookupError` is the common base for `KeyError` (missing key) and `IndexError` (out-of-range index). `ValueError` = right type, wrong value; `TypeError` = wrong type entirely. [official] (https://docs.python.org/3/library/exceptions.html)
- Python 3.14 free-threading docs: In the free-threaded build (GIL disabled), there is no GIL to protect against mutation of state on Python objects shared between threads. Per-object locking is needed. [official] (https://py-free-threading.github.io/porting/)

## Best Practices

### Q1: `InactiveEntityError` vs `MissingEntityError` for entity-lookup failures

- **Collapse until callers need to distinguish.** The stdlib precedent is `LookupError` → `KeyError`/`IndexError`: the split happened because callers had meaningfully different responses (key-miss vs bounds-miss). For the v0.1 `advance_turn` case, the caller has one response: abort the turn. Two separate exception types with a single catch clause at every call site buys nothing and forces every future catch site to catch both. [community] (https://docs.python.org/3/library/exceptions.html)
- **tcod-ecs convention:** raises plain `KeyError` for a missing component (no custom ECS exception). esper uses integer IDs and queries by component type, sidestepping lookup errors entirely. Neither ECS library creates a named `InactiveEntity` / `MissingEntity` / `EntityNotFound` type — they lean on `KeyError` or return `None`. [official] (https://github.com/HexDecimal/python-tcod-ecs)
- **The split question is about caller behaviour, not semantic precision.** If v0.2 adds a `DeadEntity` state where the entity exists-but-is-inactive (different from absent), the split will become load-bearing. The correct move is to name the *current* error for the *current* semantic ("this entity ID is not a live Starship in this sector") and document the v0.2 split trigger explicitly. `LookupError` (or a custom `EntityLookupError`) works as the base; `InactiveEntityError` and a new `MissingEntityError` can be subclasses later. Catch sites catch the base. [community] (https://www.hrekov.com/blog/python-exception-hierarchy)
- **Refactor cost when conflated errors split:** if v0.1 uses a single `InactiveEntityError`, the v0.2 split only touches the *raise* sites (one per lookup), not the catch sites (if catch sites use the base class). Cost is low if the hierarchy is designed correctly from day one. [community] (https://nicepython.com/tutorial/exception-hierarchy/)

**Verdict:** Keep one exception for now but define it as a `LookupError` subclass (e.g. `EntityLookupError`), with `InactiveEntityError(EntityLookupError)` as a concrete sub-type. Document the split trigger. Catch sites catch `EntityLookupError` for future-proofness.

### Q2: `send_robust` migration trigger

- **`send_robust` does not exist in blinker.** See Footguns. The Django `send_robust` pattern (fault-isolated per-receiver dispatch) is the closest analogue. For blinker, the migration path is either (a) wrap `send()` in a try-except at the caller, (b) write a utility that iterates `signal.receivers_for(sender)` with individual try-except blocks, or (c) switch to a dispatcher queue. [official] (https://blinker.readthedocs.io/en/latest/)
- **When fault isolation becomes necessary:** Django's internal guidance (and its own internal usage) is that `send_robust`-style dispatch is warranted when subscribers come from **untrusted or independently-maintained modules** — i.e., when one subscriber's failure should not abort delivery to the rest. For a game with exactly one subscriber (the controller bridge), `send()` is correct, and the migration trigger is "second subscriber added from an independent module." [community] (https://hakibenita.com/django-reliable-signals)
- **Concrete trigger for the spec:** "Switch to fault-isolated dispatch when a second subscriber is connected to any signal OR when any subscriber is loaded from user-defined plugin code." At that point, implement a `send_robust_compat()` utility wrapping per-receiver try-except, and log caught exceptions at `ERROR` level so they are visible. [community] (https://django-advanced-training.readthedocs.io/en/latest/features/signals/)

### Q3: `Final` vs `ClassVar[Final]` for the TOML discriminator

- **For non-dataclasses (which `GameObject` subclasses are), plain `Final` in the class body is already inferred as a class variable by mypy.** The typing spec and PEP 591 agree: `BORDER_WIDTH: Final = 2.5` in a class body is treated as a class-level final. No `ClassVar` wrapper needed, and in fact adding it is redundant (may trigger a warning). [official] (https://mypy.readthedocs.io/en/stable/final_attrs.html)
- **`Final` prevents subclass override under mypy --strict:** `class ListView(Window): BORDER_WIDTH = 3` raises `error: can't override a final attribute`. This is exactly the protection needed to prevent a `PlayerStarship(Starship)` from silently shadowing `kind`. [official] (https://mypy.readthedocs.io/en/stable/final_attrs.html)
- **Dataclass exception (relevant if `Starship` is ever converted to `@dataclass`):** For dataclasses, `kind: Final[str] = "starship"` would create an instance-level final field (a dataclass field), which is wrong. The correct form for a final class variable inside a dataclass is `kind: ClassVar[Final[str]] = "starship"`. Since `Starship` is NOT a dataclass, this distinction does not apply today, but is a v0.2 gotcha if `@dataclass` is added. [official] (https://typing.python.org/en/latest/spec/qualifiers.html)
- **mypy bug #18235 (open, May 2026):** mypy 1.13+ still rejects `ClassVar[Final[str]]` even on Python 3.13 with `strict = true`, raising `error: Final can be only used as an outermost qualifier in a variable annotation [valid-type]`. Do not use `ClassVar[Final[str]]` in non-dataclass code until this bug is resolved. [official] (https://github.com/python/mypy/issues/18235)

**Verdict:** Use `kind: Final[str] = "starship"` (plain `Final`, no `ClassVar` wrapper). This gives mypy --strict override protection now, and is the correct form for non-dataclass bodies. Add a comment noting the dataclass exception for future reference.

## Footguns and Gotchas

- **`send_robust` does not exist in blinker 1.x.** A PR was proposed (#27, #3) modelled on Django's `Signal.send_robust`, but it was never merged into the pallets-eco/blinker mainline. The changelog from 0.8 through 1.10 contains no `send_robust` entry. Any spec text referencing `signal.send_robust()` will produce `AttributeError` at runtime. — corroborated by [official blinker 1.10 API docs](https://blinker.readthedocs.io/en/latest/) and [CHANGES.rst](https://github.com/pallets-eco/blinker/blob/main/CHANGES.rst)

- **`ClassVar[Final[str]]` causes a mypy error in --strict mode as of mypy 1.13.** Despite Python 3.13 permitting the nesting syntactically, mypy bug #18235 has not been resolved. Using `ClassVar[Final[str]]` in production code breaks CI. — corroborated by [mypy issue #18235](https://github.com/python/mypy/issues/18235) and [mypy final_attrs docs](https://mypy.readthedocs.io/en/stable/final_attrs.html)

- **`Final` does not block instance reassignment at runtime — only at type-check time.** `obj.kind = "other"` will silently succeed at runtime. mypy catches it, but untyped code or dynamic access bypasses the guard entirely. For TOML save discrimination, the runtime check is the `isinstance(entity, Starship)` dispatch, not the `kind` attribute. [official] (https://python.github.io/peps/pep-0591)

- **blinker `send()` exception propagation halts remaining receivers.** If the first subscriber raises, subsequent subscribers are never called. This is by design and documented, but it's a footgun when adding a second subscriber: the ordering is undefined, and a buggy subscriber silently shadows all subsequent ones. — corroborated by [blinker 1.10 docs](https://blinker.readthedocs.io/en/latest/) and [blinker PR #27](https://github.com/jek/blinker/pull/27/files)

- **Python 3.14 free-threading: GIL no longer protects `ap_remaining` when QThreadPool is used.** The GIL has historically made simple integer mutation on a Python `int` attribute effectively atomic. In Python 3.14's free-threaded build (which PySide6 AI workers may opt into), this protection is gone. Any read-modify-write sequence (`ap_remaining -= cost`) requires explicit locking. — corroborated by [py-free-threading.github.io](https://py-free-threading.github.io/porting/) and [Python 3.14 thread state docs](https://docs.python.org/3/c-api/threads.html)

- **`threading.Lock` and `QMutex` both wrap OS-native mutexes.** They are interoperable; `threading.Lock()` acquired in a QThread is fully valid. The distinction is only ergonomic. [community] (https://stackoverflow.com/questions/53905495/is-threading-lock-compatible-with-qthread-and-is-qmutex-compatible-with-python)

## Security and Compatibility

- **Python 3.14 free-threaded build (PEP 703):** The optional GIL-disabled build is production-quality in 3.14. Any code relying on the GIL as an implicit mutex around integer attribute mutation (e.g. `ap_remaining -= cost`) will have a race condition if more than one thread touches the same `Starship` object. `threading.Lock` or confining all model mutations to a single thread is required for v0.2. (https://py-free-threading.github.io/porting/)
- **mypy bug #18235:** `ClassVar[Final[str]]` annotation rejected by mypy 1.13 under `strict = true`. Filed against Python 3.13; open as of May 2026. Workaround: use plain `Final[str]`. (https://github.com/python/mypy/issues/18235)
- **No CVEs for blinker 1.x or tcod-ecs surfaced in this sweep.**

## Recent Changes

- **Python 3.13 (2024-10):** `ClassVar` can now be nested in `Final` and vice versa at the language/typing level (PEP 591 update, cpython #89547). mypy has not implemented this yet. [official] (https://docs.python.org/3/whatsnew/3.13.html)
- **blinker 1.10 (latest):** No `send_robust`. Last significant API addition was `send_async` in 1.6 (2023-04). Changelog has no `send_robust` entry in any version. [official] (https://github.com/pallets-eco/blinker/blob/main/CHANGES.rst)
- **mypy 2.1.0 (current stable):** `Final` and `ClassVar` nesting for dataclasses still blocked by open issue #18235. Non-dataclass `Final` enforcement is solid. [official] (https://mypy.readthedocs.io/en/stable/final_attrs.html)
- **Python 3.14 free-threading:** GIL is opt-in disabled; CPython JIT thread-safety issue #133171 still open (Apr 2026). For v0.1 (GIL-enabled, single model thread) there is no impact. [official] (https://github.com/python/cpython/issues/133171)
