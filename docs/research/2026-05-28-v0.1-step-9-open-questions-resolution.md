## OQ-1 — Parametric `Signal[T]` Typing

### What exists today

**Official stubs (bundled with PySide6 since 6.4.3).** `Signal(object)` is typed as accepting and
emitting `object`. There is no generic `Signal[T]` form. The stubs are generated by `pyside6-genpyi`
from C++ introspection; because Qt's C++ `Signal` is not a C++ template, there is no upstream
mechanism to generate per-instance payload types. The official stance is that `Signal(object)` is
the correct form for arbitrary Python payloads. [official] (https://doc.qt.io/qtforpython-6/PySide6/QtCore/Signal.html)

**Third-party `PySide6-stubs` (python-qt-tools, latest 6.7.3.0).** This package improves dozens of
incorrect or missing stub entries — `QByteArray`, nullable returns on `topLevelItem`, `connect()`
return type fixed to `QMetaObject.Connection`, etc. It does NOT add parametric `Signal[T]` support.
The project's own README does not mention parametric signals as a goal or in-progress item. It is
derived from the official stubs via AST-level corrections, so adding generic Signal would require
a fundamentally different approach (a mypy plugin, or a new descriptor stub class). [community]
(https://github.com/python-qt-tools/PySide6-stubs) (https://pypi.org/project/PySide6-stubs/)

**Proof-of-concept on PyQt5-stubs issue #68.** A contributor demonstrated that a generic
`pyqtBoundSignal[TA, TB]` with typed `emit(a: TA, b: TB)` is structurally possible as a stub
descriptor class. However, the contributor noted it does not handle the variadic nature of signal
parameter counts without a mypy plugin. No implementation was merged. No equivalent proposal
exists for PySide6-stubs as of May 2026. [community] (https://github.com/python-qt-tools/PyQt5-stubs/issues/68)

**Conclusion on roadmap.** There is no known active proposal, open PR, or Qt Company roadmap item
for parametric `Signal[T]` stubs. The architectural constraint is that `Signal` takes a runtime
type argument (e.g. `Signal(object)`) rather than a static type parameter, which is incompatible
with Python's generic type system without a mypy plugin or significant stub engineering. This is
unlikely to change before v0.1 ships.

### Viable workarounds today

**Option A: Typed `emit()` wrapper on `ModelBridge` (recommended).** Add a named method that
wraps `emit()` with a type-annotated signature:

```python
class ModelBridge(QObject):
    ship_moved = Signal(object)  # carries ShipMovedPayload

    def emit_ship_moved(self, payload: ShipMovedPayload) -> None:
        """Type-checked emit; keeps Signal(object) at the Qt boundary."""
        self.ship_moved.emit(payload)
```

The bridge always calls `self.emit_ship_moved(payload)` instead of `self.ship_moved.emit(payload)`
directly. `mypy --strict` sees `(ShipMovedPayload) -> None` at the call site inside the bridge.
The view slot connects to `self.ship_moved` (the raw `Signal(object)`) and must still type its
receiver as `(object) -> None` or narrow internally — that is unavoidable at the view boundary
given current stubs. This workaround has zero runtime cost and does not affect Qt's introspection.
[community] (https://github.com/python-qt-tools/PyQt5-stubs/issues/68)

**Option B: `@overload` facade under `TYPE_CHECKING`.** Declare a typed overload signature visible
only to mypy:

```python
if TYPE_CHECKING:
    def emit_ship_moved(self, payload: ShipMovedPayload) -> None: ...
else:
    emit_ship_moved = lambda self, p: self.ship_moved.emit(p)
```

This is more complex and the `else` branch looks fragile. Not recommended.

**Option C: `typing.TYPE_CHECKING`-only typed protocol.** For the view's slot, use a narrowing
`assert isinstance(payload, ShipMovedPayload)` immediately inside the slot body. This satisfies
mypy and keeps the slot annotated `(object) -> None`. No additional abstraction needed. [community]
(https://stackoverflow.com/questions/14421897/is-the-pyside-slot-decorator-necessary)

**At the `emit()` call site inside the bridge** (Option A handles this).
**At the view slot signature**: slot annotated `@Slot(object)` with `(self, payload: object)`
then `assert isinstance(payload, ShipMovedPayload)` for narrowing. This is the correct pattern
under `mypy --strict` given the current stub limitation.

**Recommendation for v0.1 spec:** Add four typed `emit_*()` wrapper methods to `ModelBridge` —
one per signal (e.g. `emit_ship_moved(self, payload: ShipMovedPayload) -> None`) — and require
the bridge's own `_on_*` handlers to call these wrappers instead of `.emit()` directly. View slots
annotate their receiver as `(self, payload: object) -> None` and narrow with `assert isinstance`.
Do NOT add `PySide6-stubs` as a dev dependency; it is last-released at 6.7.3.0 and does not help
with the Signal typing gap. Document this as a known gap pending a future mypy plugin or parametric
stub solution.

## OQ-2 — `@Slot` Decoration of the Bridge's Internal `_on_*` Handlers

### Mechanism: what `@Slot` does and does not do

When a Python method is decorated with `@Slot`, PySide6 registers it as a **native C++ slot** in
Qt's meta-object system. The method is directly callable by Qt's dispatch without an intermediate
Python proxy object.

When a Python method has NO `@Slot` decoration, PySide6 creates a **proxy wrapper object** that
holds a reference to the Python callable and provides a native slot interface to Qt. The proxy is
created at connection time. [community] (https://www.pythonguis.com/faq/what-does-slot-do/)

### When is `@Slot` load-bearing vs cosmetic?

**Single-thread / DirectConnection (v0.1 scenario).** In v0.1, all model mutations and blinker
`send()` calls happen on the main thread. The bridge's `_on_*` handlers are registered in
blinker's receiver table, not in Qt's meta-object system. Blinker calls them as plain Python
callables. `@Slot` on `_on_*` handlers does absolutely nothing: blinker does not consult Qt's
slot registration. The handlers never pass through Qt's signal dispatch. For these handlers,
`@Slot` is purely cosmetic. [community] (https://www.pythonguis.com/faq/what-does-slot-do/)
[official] (https://doc.qt.io/qtforpython-6/PySide6/QtCore/Slot.html)

**Cross-thread / QueuedConnection (v0.2 trigger).** If in v0.2 a model worker thread is introduced
(e.g. the model runs on a `QThread`, emitting blinker signals from that thread, and the bridge
subscribes to forward them into Qt's main thread via `QueuedConnection`), then `@Slot` becomes
load-bearing on any method that Qt must dispatch via `QueuedConnection`. The Qt Forum thread from
January 2025 confirms this failure mode: without `@Slot`, the proxy wrapper is created in the
emitting thread's context. When the signal is delivered to the receiver's thread via event queue,
Qt's dispatch calls the proxy from the receiver thread, but the proxy was created in the sender
thread — this inconsistency can produce a segmentation fault. The `@Slot` decorator eliminates the
proxy entirely and prevents this. [community]
(https://forum.qt.io/topic/160665/pyside6-slot-executed-in-signal-s-thread-even-with-auto-queuedconnection)
(https://www.pythonguis.com/faq/what-does-slot-do/) [official]
(https://doc.qt.io/qtforpython-6/overviews/qtdoc-signalsandslots-syntaxes.html)

**The SO thread from August 2025** notes this is an existing answer that applies to PySide6: the
`@Slot` decorator is only required on slots that may be connected across threads.

**Clarification: blinker handlers vs Qt slots.** The bridge's `_on_*` methods are blinker
receivers, NOT Qt slots. They are never connected to Qt signals via `.connect()`. Therefore the
proxy-creation risk described above does not apply to them even in a cross-thread v0.2 scenario.
The v0.2 threading risk applies to the OTHER direction: the bridge's Qt `Signal` (`self.ship_moved`
etc.) being connected to a view slot via `QueuedConnection`. In that case, `@Slot` must appear on
the **view slot**, not on the bridge's `_on_*` handlers.

### One-line rule for the spec

`@Slot` on a method is required if and only if that method is connected to a Qt signal via a
`QueuedConnection` (i.e. the connecting object and receiving object live on different threads).
For `DirectConnection` (same thread), `@Slot` is optional. For blinker receiver methods, `@Slot`
is always irrelevant.

**Recommendation for v0.1 spec:** Do NOT add `@Slot` to the bridge's internal `_on_*` blinker
handlers — they are invoked by blinker, not Qt, and `@Slot` has no effect there. Add a comment
to each `_on_*` handler: `# blinker receiver — not a Qt slot; @Slot not applicable`. The spec
should document the v0.2 trigger: "When a background model thread is introduced via `QThread` and
the bridge subscribes to a cross-thread blinker signal (or the view slot is in a different thread
than the bridge's `moveToThread()` target), add `@Slot(object)` to the **view slot** methods that
will be invoked via `QueuedConnection`."

## OQ-3 — Bridge Test Isolation Strategy

### Repo fact: pytest-qt is already available

`pyproject.toml` `[project.optional-dependencies]` `dev` includes `pytest-qt>=4.4`.
`QT_QPA_PLATFORM=offscreen` is already set in `[tool.pytest.ini_options]`. No further
configuration is needed to run QObject-constructing tests headlessly. No conftest.py exists
yet anywhere under `tests/`.

### How `qapp` and `qtbot` work (scopes)

`qapp` is **session-scoped** — this is a hard constraint, not a design choice. A `QApplication`
instance cannot be destroyed and recreated within a single process. pytest-qt's source (confirmed
via Context7 deep read) shows `qapp` calls `QApplication.instance()` and returns the existing
one if present; it uses a module-level `_qapp_instance` to keep the singleton alive. The session
scope ensures exactly one `QApplication` per test run. [official]
(https://pytest-qt.readthedocs.io/en/latest/_modules/pytestqt/plugin.html)

`qtbot` is **function-scoped** (the default for pytest fixtures). Each test gets a fresh `QtBot`
instance. `qtbot.addWidget(widget)` registers a widget for cleanup after the test. `qtbot` depends
on `qapp`, so using `qtbot` implicitly ensures the `QApplication` exists.
[official] (https://pytest-qt.readthedocs.io/en/latest/_modules/pytestqt/plugin.html)

`qtbot.waitSignal(signal, timeout=ms)` blocks test execution until the specified Qt signal emits
(or until timeout, raising `qtbot.TimeoutError`). This is the correct tool to assert that a
blinker event flowing through the bridge actually causes the bridge's Qt signal to fire. It works
because the Qt signal is emitted synchronously in the test's own thread (DirectConnection), so
`waitSignal` sees the emission immediately — no threading or event loop processing needed for v0.1.
[official] (https://pytest-qt.readthedocs.io/en/latest/_sources/signals.rst.txt)

### The blinker subscriber accumulation problem

The prior report flagged: with `weak=False`, blinker holds the bridge strongly forever. If a
`ModelBridge` is constructed in a test and not explicitly disconnected, subsequent tests that
trigger blinker `send()` will call ALL live bridge instances' `_on_*` handlers, including ones
from prior tests. This is the "two subscribers, both emit" scenario from the risk section of the
predecessor research.

The house style in `test_game_state_manager.py` is `state_changed.connect(receiver)` /
`state_changed.disconnect(receiver)` in try/finally. For a `ModelBridge` (which wraps four
connections), this means the bridge's teardown must call `ev.*.disconnect(self._on_*)` on all
four signals. [official] (https://blinker.readthedocs.io/en/stable/)

### Recommended fixture pattern

```python
# tests/conftest.py  (create this file at tests/conftest.py)
import pytest
from stmrr.model import events as ev
from stmrr.controller.model_bridge import ModelBridge

@pytest.fixture
def bridge(qtbot) -> ModelBridge:
    """Fresh ModelBridge per test, with blinker teardown on exit.

    Scope: function (default). Each test gets an isolated bridge instance.
    qtbot dependency ensures QApplication exists (session-scoped singleton).
    teardown calls bridge.teardown() which disconnects all four blinker
    subscriptions — prevents subscriber accumulation across tests.
    """
    b = ModelBridge()
    yield b
    b.teardown()  # explicit blinker disconnect — see ModelBridge.teardown()
```

**Why function scope, not session scope.**
- Session-scoped bridge accumulates blinker subscriptions from all test scenarios. A test that
  exercises `ship_moved` would see handlers from other tests still registered.
- A fresh per-test bridge costs one `QObject` construction per test — negligible.
- The `QApplication` singleton (via `qapp`, transitively via `qtbot`) is the expensive shared
  resource; it is already session-scoped.

**The `teardown()` method on `ModelBridge` (hooks into OQ-4):**

```python
def teardown(self) -> None:
    """Disconnect all four blinker subscriptions. Call in tests and on app close."""
    from stmrr.model import events as ev
    ev.ship_moved.disconnect(self._on_ship_moved)
    ev.docked.disconnect(self._on_docked)
    ev.turn_advanced.disconnect(self._on_turn_advanced)
    ev.state_changed.disconnect(self._on_state_changed)
```

**Using `waitSignal` to assert blinker-to-Qt re-emission:**

```python
def test_ship_moved_blinker_causes_qt_signal(bridge, qtbot) -> None:
    payload = ShipMovedPayload(...)
    with qtbot.waitSignal(bridge.ship_moved, timeout=1000) as blocker:
        events.ship_moved.send(object(), payload=payload)
    assert blocker.args == [payload]
```

For v0.1 (single thread, DirectConnection), `waitSignal` resolves immediately — no timeout is
expected to trigger. The `timeout=1000` is a safety net against test-framework races.

**GC-cycle test (mandatory, per predecessor risk section):**

```python
def test_ship_moved_survives_gc_cycle(bridge, qtbot) -> None:
    import gc
    gc.collect()  # force GC; weak=False means bridge survives
    with qtbot.waitSignal(bridge.ship_moved, timeout=1000):
        events.ship_moved.send(object(), payload=ShipMovedPayload(...))
```

This test is the direct guard for the `weak=False` correctness requirement.

**Recommendation for v0.1 spec:** Create `tests/conftest.py` with a function-scoped `bridge`
fixture that depends on `qtbot` (which provides the session-scoped `QApplication` singleton).
The fixture calls `bridge.teardown()` in its yield-teardown. Test files for the bridge import
`bridge` from conftest; they do not manage blinker disconnect manually. The bridge's `teardown()`
method is also the production teardown path (see OQ-4).

## OQ-4 — Explicit `teardown()` / Disconnect API on `ModelBridge`

### The two independent teardown obligations

The predecessor research established that Qt parenting (`parent=main_window`) auto-disconnects
**Qt signals** when the window is destroyed — specifically, Qt's `~QObject` destructor walks the
connection list and disconnects. This handles the view slots connected to `ModelBridge.ship_moved`
etc. **This does NOT disconnect the blinker subscriptions.** The four blinker subscriptions
(`ev.ship_moved.connect(self._on_ship_moved, weak=False)` etc.) are held in blinker's module-level
signal objects, which live for the process lifetime. These are two independent teardown obligations.

### How mature blinker consumers handle this

**Flask.** Flask signals (all blinker-based) are subscribed via `@signal.connect_via(app)` or
`signal.connect(receiver)`. Flask itself does not provide explicit `disconnect_all()` — Flask's
design is that receivers are subscribed for the lifetime of the application object, and the
application object IS the process. Flask's approach is essentially: single-instance ownership +
process lifetime. This is a legitimate pattern when there is exactly one subscriber instance.
[official] (https://flask.palletsprojects.com/en/stable/signals/)

**blinker `disconnect()` API.** `signal.disconnect(receiver, sender=ANY)` — explicit, takes the
receiver callable. Called per-signal. No `disconnect_all()` convenience exists on the signal
object itself; callers must iterate. [official] (https://blinker.readthedocs.io/en/stable/)

**`connected_to()` context manager.** blinker provides `signal.connected_to(receiver)` as a
context manager — inside the `with` block the receiver is connected, on exit it is disconnected.
This is the test-isolation tool already used in `test_events.py`. However, `connected_to()` always
uses `weak=True` (the default) — it cannot be used for the bridge's production connections where
`weak=False` is required. It IS useful in the test fixture for registering a probe receiver on
the bridge's Qt signal.

**Teardown options evaluated.**

| Option | Fit for v0.1 | Notes |
|--------|-------------|-------|
| Explicit `teardown()` method | Best fit | Called in test fixture teardown + app close event. Clear ownership. No magic. |
| Context manager (`__enter__`/`__exit__`) | Over-engineering for v0.1 | Makes the bridge a context manager; forces `with ModelBridge() as b:` call-site. Awkward for a long-lived singleton held by `app.py`. |
| Rely on single-instance ownership + process lifetime | Acceptable IF no tests share a bridge instance | Breaks test isolation since blinker subscriptions are module-level and survive tests. |
| Qt parent auto-disconnect | Handles only Qt signals, not blinker | Already used; insufficient alone. |

### Qt parenting vs blinker teardown clarification (spec-level)

The spec must state explicitly:

- `ModelBridge(parent=main_window)` — Qt parenting. When `main_window` is destroyed, Qt
  calls `~QObject` on the bridge, which disconnects all Qt signal connections from `bridge.ship_moved`,
  `bridge.docked`, etc. to view slots. This is automatic.
- `bridge.teardown()` — blinker teardown. Must be called explicitly before the bridge instance
  is discarded. The cleanest hook is the `QObject` destructor or Qt's `destroyed` signal:
  `bridge.destroyed.connect(bridge.teardown)`. Because `QObject.destroyed` is emitted before
  `~QObject` finishes, this fires at the right time and means production code gets automatic
  blinker teardown without any `app.py` ceremony.

The `destroyed` signal hook is the cleanest production path because it keeps `app.py` simple
(just `ModelBridge(parent=main_window)`, no explicit cleanup call) while ensuring blinker
teardown runs at window close. [official] (https://doc.qt.io/qtforpython-6/PySide6/QtCore/QObject.html)

### Concrete `teardown()` design

```python
def teardown(self) -> None:
    """Disconnect all blinker subscriptions.

    Called by the QObject.destroyed signal (production) and by the pytest
    bridge fixture's teardown (tests). Do NOT call emit() on any Qt signal
    after teardown() — the bridge is a zombie at that point.

    Two teardown obligations exist independently:
    1. blinker subscriptions: handled here (this method).
    2. Qt signal connections to view slots: handled automatically by Qt
       when the bridge's QObject is destroyed (parent=main_window).
    """
    from stmrr.model import events as ev
    ev.ship_moved.disconnect(self._on_ship_moved)
    ev.docked.disconnect(self._on_docked)
    ev.turn_advanced.disconnect(self._on_turn_advanced)
    ev.state_changed.disconnect(self._on_state_changed)
```

Production wiring in `app.py` (once it exists):

```python
bridge = ModelBridge(parent=main_window)
bridge.destroyed.connect(bridge.teardown)  # auto-disconnect blinker on window close
```

### v0.2 trigger

If a second `ModelBridge` is ever introduced (e.g. for a secondary view), the single-subscriber
assumption breaks. v0.2 should introduce a singleton guard or make the bridge explicitly
registerable/deregisterable with a bridge registry. The `teardown()` method's design does not
change — it just becomes more critical.

**Recommendation for v0.1 spec:** Add an explicit `teardown()` method to `ModelBridge` that
disconnects all four blinker subscriptions. In production `app.py` wiring, connect
`bridge.destroyed` to `bridge.teardown` so the blinker teardown fires automatically at window
close without any additional `app.py` ceremony. In tests, call `bridge.teardown()` explicitly
in the pytest fixture's yield-teardown. Clarify in the spec that Qt parenting handles ONLY Qt
signal connections; `teardown()` handles ONLY blinker subscriptions — these are orthogonal.
The v0.2 trigger for more elaborate teardown (bridge registry, multiple views) is: introduction
of a second `ModelBridge` instance or a secondary view subscribing independently to model events.

## Footguns and Gotchas

- **`@Slot` proxy segfault on QueuedConnection without decoration.** When a Python method
  WITHOUT `@Slot` receives a `QueuedConnection` signal, PySide6 creates a proxy in the sender's
  thread context. Qt dispatches through the proxy from the receiver's thread context at event
  loop tick. This mismatch causes a segmentation fault in some PySide6 versions (6.6.x–6.8.x
  confirmed in Qt Forum Jan 2025 report). Fix: `@Slot(object)` on any method that will be called
  via `QueuedConnection`. This does NOT apply to blinker receivers — blinker bypasses Qt's proxy
  mechanism entirely. — corroborated by
  [Qt Forum 2025-01 thread](https://forum.qt.io/topic/160665/pyside6-slot-executed-in-signal-s-thread-even-with-auto-queuedconnection)
  and [pythonguis.com FAQ March 2026](https://www.pythonguis.com/faq/what-does-slot-do/)

- **`QApplication` cannot be destroyed and recreated.** pytest-qt's `qapp` fixture is
  session-scoped for this reason. Attempting to instantiate a second `QApplication` in a test
  produces a warning and may silently reuse the existing one. Do not override `qapp_cls` in a
  function-scoped fixture. — corroborated by
  [pytest-qt readthedocs source](https://pytest-qt.readthedocs.io/en/latest/_modules/pytestqt/plugin.html)
  and [pytest-qt issue #394](https://github.com/pytest-dev/pytest-qt/issues/394)

- **blinker `connected_to()` context manager uses `weak=True`.** The test style in
  `test_events.py` uses `ship_moved.connected_to(receiver)` for one-shot test receivers. This is
  correct for temporary probe receivers — blinker holds a weakref to the lambda/function, but
  inside the `with` block the local variable keeps the strong reference alive. Do NOT use
  `connected_to()` for the bridge's production connections; `weak=False` in `__init__` is required.
  — corroborated by [blinker 1.9 official docs](https://blinker.readthedocs.io/en/stable/) and
  prior research predecessor file.

- **`Signal(object)` dict payload copy-on-emit bug.** An SO question from 2023 (corroborated by
  Qt bug reports) shows PySide6 sometimes copies `dict` payloads unexpectedly when using
  `Signal(dict)`, causing the receiver to see a stale copy. `Signal(object)` avoids this — it
  sends a pointer to the Python object, not a C++ copy. For frozen dataclasses (the payload type
  in this project), no mutation is possible anyway, but `Signal(object)` is still the correct
  type annotation. — corroborated by
  [SO 2023](https://stackoverflow.com/questions/76579504/how-dose-pyside6-signal-emit-transfer-data-for-dictionary-data-why-the-behavior-different)
  and [prior research predecessor](docs/research/2026-05-28-v0.1-step-9-open-questions.md)
